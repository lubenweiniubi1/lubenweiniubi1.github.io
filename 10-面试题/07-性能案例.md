**我**：

>
> 这个页面有十几个全屏模块，每个模块里的动画不是点一下播一次，而是**跟着滚动条实时走**——你滚到哪就播到哪，向下滚是进场动画，向上滚还得倒播退场。而且模块高度不固定，有的内容多，能到 200vh。
>
> 十几组动画同时挂在那，加上高频滚动事件，性能直接崩了。这就是我要解决的核心问题。
>
> [→ 项目详细背景](#bg)

---

**面试官**：你怎么发现性能有问题的？就是感觉卡吗？

**我**：

> 说实话第一反应确实是"感觉卡"——手指滑过去了，动画还在后面追。但感觉不能当证据，我得拿数据说话。
>
> 我用微信开发者工具的 Performance 面板录了一段滚动操作，FPS 只有二十几，流畅至少要 50+。然后逐帧分析耗时，定位到一个很具体的问题：
>
> 每次 scroll 事件触发 `setState({ scrollTop })`，然后 React diff 十几个模块的 Virtual DOM，这一步大概 5 毫秒。接着 Taro 把变更通过 `setData` 序列化传到渲染层，又是 8 毫秒。光这两步就超了 16.67 毫秒的单帧预算。
>
> 根因一句话：**scroll 里高频调用 setState 阻塞了渲染。**
>
> [→ 性能测量方法](#measure)

---

**面试官**：那你做了什么优化？

**我**：

> 核心思路就一个——**让动画脱离 React 渲染循环，放到渲染层独立执行。**
>
> 具体做法：用 `IntersectionObserver` 检测每个模块是否进入视口、判断滚动方向，然后直接调用 `element.animate(keyframes, options)` 把动画交给渲染层。Observer 是异步的，不阻塞主线程；`animate()` 虽然是个 JS API，但动画的实际执行发生在渲染层，不需要 JS 逐帧参与。
>
> 这样一来，scroll 事件里完全不需要调 setState 了。React 只负责"什么时候让渲染层开始播"，不负责"动画怎么播"。这个切换把帧预算的大头直接砍掉了。
>
> 在此基础上还做了几个辅助优化：`will-change` + `translate3d` 创建独立合成层，让 transform/opacity 动画只触发 Composite、跳过 Layout 和 Paint；全部图片换成 WebP，体积减少 70%；React.memo 防止非动画场景下的连带 re-render。
>
> [→ 优化一：animate + Observer](#opt-animate) | [→ 优化二：will-change + 合成层](#opt-will-change) | [→ 优化五：WebP 图片优化](#opt-webp)

---

**面试官**：等一下，你说 `will-change` 创建合成层——它底层到底做了什么？为什么 transform 不触发重排？

**我**：

> 浏览器的渲染流水线是：Style → Layout → Paint → Composite。`will-change: transform` 提前告诉浏览器"这个元素马上要动"，浏览器就在空闲时把它提升为独立的 GraphicsLayer——相当于在 GPU 显存里单独存了一份纹理。
>
> 有了独立合成层之后，transform 变化只影响 Composite 阶段——GPU 改一下变换矩阵就行，纹理本身不需要重新生成。这就是为什么只改 transform/opacity 不触发重排和重绘。
>
> 但有一个点要注意：每个合成层都占 GPU 显存。所以动画结束后要把 `will-change` 去掉，低端机还要控制图层数量。
>
> [→ 渲染流水线详解](#render-pipeline)

---

**面试官**：你直接在 Observer 回调里操作 DOM，不走 React 的 setState——这合理吗？

**我**：

> 这个问题问得好。我觉得这里的关键是搞清楚**谁该管什么事**。
>
> React 的职责是管理组件状态和 UI = f(state) 的映射关系，适合"数据变了 UI 跟着变"的场景。但动画是另一回事——动画的定义（keyframes）是纯 CSS 属性，执行是在渲染层，它不需要 JS 状态驱动每一帧。
>
> React 不该管"动画怎么播"，它只需要管"什么时候让渲染层开始播"。Observer 回调里调用 `element.animate()`，就是在做这个"通知"的职责。这个信息不需要进入 React 的状态管理体系——不进 setState 是刻意的，不是疏忽。
>
> [→ 职责分离的深入讨论](#separation)

---

**面试官**：那你为什么不用虚拟列表？

**我**：

> 虚拟列表在这个场景有三个条件让它不可行：
>
> 第一，也是最致命的——模块高度不固定。有的 100vh，有的 250vh。定高虚拟列表直接不能用，动态高度的需要一个测量阶段，但测量本身就包含了大量 DOM 和动画元素，代价等于先把性能问题经历一遍。
>
> 第二，虚拟列表在元素滚出可视区后会卸载它。但我的动画需要 模块持续在 DOM 里保持状态。卸载再 mount 回来，动画从零开始，会有明显的闪烁。
>
> 而且说实话，我的瓶颈不是 DOM 数量——十几模块 对浏览器来说不算什么。真正的瓶颈是 scroll 中高频 setState 阻塞了渲染。
>
> [→ 虚拟列表为什么不行](#why-not-virtual)

**面试官**：你考虑过其他方案吗？

**我**：

> 考虑过。如果现在重新做，CSS Scroll-driven Animations 是更理想的方案——动画完全在 Compositor 线程执行，连 Observer 都不需要，而且是原生支持双向滚动驱动。

**第二步：使用工具收集数据**

| **微信开发者工具 → 性能面板** | 实时查看 FPS、CPU 占用、内存 | FPS 低于 30 即为明显卡顿，CPU 持续高占用说明 JS 执行过重 |

| **真机调试** | 在真实手机上运行，观察实际性能（开发者工具有代理层开销，性能比真机差） | FPS、动画跟手度、是否闪退（GPU 内存溢出） |

**第三步：定位瓶颈**

```
录了一段滚动过程的 Performance 面板 → 分析发现：
├── scroll 事件回调高频触发 → 每次触发 React setState({ scrollTop })
│   → 页面级 state 更新导致 10+ 个 Section 组件全部 re-render
│   → React Virtual DOM diff 耗时 ~5ms
│   → Taro setData 序列化 + 跨线程传输 ~8ms
├── 动画属性通过 JS 动态设置 inline style → 每帧触发 Style 重算 + Paint
│   → Paint 耗时 ~6ms
├── 合计单帧远超 16.67ms 预算 → 实际 FPS 约 20-25
└── 瓶颈确认：scroll 中的 setState 是整个性能问题的根源
```

### 为什么 scroll 中 setState 是问题根源？

```
scroll 事件（每 ~16ms 触发一次）
  → setState({ scrollTop })
  → React Reconciler 遍历 10+ 个 Section 的 Virtual DOM 进行 diff
  → Taro 把 diff 结果通过 setData 序列化传到渲染层（跨线程 IPC）
  → 渲染层更新 WebView DOM
  → 浏览器重新 Layout → Paint → Composite

每次 scroll 事件 → 完整走一遍上述链路
JS 线程被占满 → 渲染层拿不到帧时间 → 掉帧 → 卡顿
```

> "我用微信开发者工具的 Performance 面板录制了一段滚动操作，FPS 在 20-30 之间波动——流畅体验需要 50+。帧耗时分析定位到了核心瓶颈：scroll 事件中的 setState 触发 React diff 约 5ms，setData 跨线程传输约 8ms，合计单帧突破 16.67ms 预算。根因是**scroll 事件中高频调用 setState 阻塞了渲染**。"

## <a id="opt-animate"></a>三、优化一（最核心）：animate() + Observer → 动画脱离 setState

**问题**：scroll 事件中 `setState({ scrollTop })` 是性能瓶颈的根源。每次滚动都触发 React 全量 re-render + Taro setData 跨线程传输。

**核心思路**：**动画用 `element.animate()` API 在渲染层直接执行，JS 逻辑层只做"开关"控制，scroll 中不再调用 setState。**

> `animate()` 的用法类似 Web Animations API 的 `element.animate(keyframes, options)`。虽然是 JS 调用，但**动画执行发生在渲染层**，不经过逻辑层的 setState → setData 链路。

```
优化前（问题链路）：
scroll 事件（高频）
  → setState({ scrollTop })
  → React diff（主线程阻塞）
  → setData 序列化 + 跨线程（小程序双线程开销）
  → 渲染层更新
  → 单帧超预算 → 掉帧

优化后：
Observer 检测 模块进入视口（异步，不阻塞）
  → 调用 element.animate(keyframes, options)（一次性调用）
  → 动画在渲染层独立执行
  → JS 线程空闲 → 帧预算完全留给渲染
```

**代码实现**：

```tsx
const enterKeyframes = [
  { opacity: 0, transform: 'translateY(40px)' },
  { opacity: 1, transform: 'translateY(0)' },
];
const leaveKeyframes = [
  { opacity: 1, transform: 'translateY(0)' },
  { opacity: 0, transform: 'translateY(-40px)' },
];
const animOptions = { duration: 600, easing: 'ease-out', fill: 'both' as const };

function Page() {
  const lastScrollY = useRef(0);

  useEffect(() => {
    const observer = new IntersectionObserver(
      (entries) => {
        const currentY = window.scrollY;
        const direction = currentY > lastScrollY.current ? 'down' : 'up';
        lastScrollY.current = currentY;

        entries.forEach(entry => {
          const el = entry.target as HTMLElement;
          if (entry.isIntersecting && direction === 'down') {
            el.animate(enterKeyframes, animOptions);
          } else if (!entry.isIntersecting && direction === 'up') {
            el.animate(leaveKeyframes, animOptions);
          }
        });
      },
      { threshold: [0, 0.1, 0.5, 1] }
    );

    document.querySelectorAll('.section').forEach(el => observer.observe(el));
    return () => observer.disconnect();
  }, []);

  return <>{/* 十几个模块，只渲染一次，之后不再 setState */}</>;
}
```

**为什么从根本上解决了问题？**

```
对比：

❌ 旧方案（React state 驱动，每帧 setState）：
scroll → setState → React diff → setData → 渲染
JS 线程每帧都在排队 → 渲染拿不到帧时间 → 持续掉帧

✅ 新方案（animate() 渲染层执行，Observer 做开关）：
Observer（异步触发）→ 调用 element.animate(keyframes, options)
→ 动画在渲染层独立执行，不需要 JS 逐帧参与
→ JS 只做了"告诉渲染层开始播动画"这一件事，然后就没事了
→ 动画全程不经过 React 渲染管线
```

## <a id="opt-will-change"></a>四、优化二：will-change + transform 3D → 创建独立合成层

先澄清一个容易误解的点——

**transform 本身确实只触发 Composite，但有一个前提：这个元素必须已经在自己的独立合成层（GraphicsLayer）里。** 如果元素没有独立合成层，它和其他内容共享一个图层，那 transform 变化时浏览器仍然需要重新绘制那个共享图层。

**will-change 和 translate3d 的作用就是：保证元素拥有独立合成层。**

```
没有独立合成层时（默认情况）：
┌─────────────────────────┐
│ 共享的 Paint Layer       │
│  ┌──────┐               │
│  │ 文字  │               │  ← 文字、图片、动画元素都在同一个图层
│  ├──────┤               │
│  │ 图片  │               │     transform 变化
│  ├──────┤               │  → 整个图层重新 Paint → 慢
│  │ 动画  │ ← 动了一下    │
│  └──────┘               │
└─────────────────────────┘

有独立合成层时（will-change / translate3d 之后）：
┌─────────────────────────┐
│ 共享的 Paint Layer       │
│  ┌──────┐ ┌──────┐      │
│  │ 文字  │ │ 图片  │      │  ← 静态元素不动
│  └──────┘ └──────┘      │
├─────────────────────────┤
│ 独立的 GraphicsLayer     │  ← will-change 创建的独立图层
│  ┌──────────────────┐   │
│  │  动画元素（GPU 纹理）│   │  ← 存在 GPU 显存里
│  └──────────────────┘   │     transform 变化 → 只移动这一层 → 快
└─────────────────────────┘
```

**三者分别做什么**：

| 手段 | 作用 | 类比 |
|------|------|------|
| `will-change: transform` | **提前通知**浏览器"这个元素马上要有 transform 变化"，浏览器**在空闲时**为它创建独立合成层 | 提前预约——"我下周要用会议室"，系统提前安排好 |
| `translate3d(0, y, 0)` | **强制**浏览器创建合成层（3D 变换必须走 GPU，浏览器没有选择余地） | 直接进去占位——不管系统怎么想，我现在就要 |
| 只改变 `transform` / `opacity` | 在**已有**独立合成层的前提下，变化发生在 Composite 阶段，不碰 Layout/Paint | 会议已经预约好了，你直接进去开就行 |

**所以完整的故事线是**：

```
1. will-change: transform    → 提前创建独立合成层（规范做法）
2. translate3d()             → 低端机型/老浏览器兜底（Hack 做法）
3. keyframes 中只用 transform + opacity → 在已有合成层上动画（零开销变化）
```

**代码**：

```css
.section {
  will-change: transform, opacity;  /* 规范方式请求独立合成层 */
}
.section.animation-done {
  will-change: auto;  /* 动画结束 → 释放 GPU 内存 */
}
```

```tsx
const enterKeyframes = [
  { opacity: 0, transform: 'translate3d(0, 40px, 0)' },
  { opacity: 1, transform: 'translate3d(0, 0, 0)' },
];
```

> **⚠️ 不要滥用**：每个独立合成层在 GPU 显存里存一份纹理。10 个模块 各有 5 个动画元素 = 50 个图层 = 可能几百 MB 显存。**用完移除**（`will-change: auto`），低端机控制图层数量。

### 浏览器怎么决定一个元素要不要给独立合成层？

```
浏览器的启发式规则（Chromium 为例）：

自动提升的情况：
├── 3D transform（translate3d、rotate3d 等）
├── <video>、<canvas>、<iframe>
├── opacity 动画 + 同时有 transform 动画
├── will-change 显式声明
├── 元素在主动画元素的上方（z-index 重叠）
└── filter（blur、drop-shadow 等）

不自动提升的情况：
├── 2D transform 单独使用（translateY、rotate 等）← 浏览器可能偷懒
├── 元素太小（不值得单独建图层）
└── 低端 GPU（避免显存压力）
```

### 性能对比总结

| 属性变更方式 | 触发 Layout | 触发 Paint | 触发 Composite | 前提条件 |
|------------|------------|------------|---------------|---------|
| `left: 0 → 100px` | ✅ | ✅ | ✅ | — |
| `background: red → blue` | ❌ | ✅ | ✅ | — |
| `transform: translateX(100px)` | ❌ | ❌ | ✅ | **元素已有独立合成层** |
| `opacity: 0 → 1` | ❌ | ❌ | ✅ | **元素已有独立合成层** |

> **一句话总结**："transform 让变化跳过 Layout/Paint，will-change 保证 transform 有跳过的前提条件。"

## <a id="opt-observer"></a>五、优化三：Observer 多阈值 + 方向判断 → 精准控制动画时机

**Observer 的优势**：
- **异步回调**，不阻塞主线程，浏览器在空闲时执行
- **自动管理**：元素离开视口或被移除时自动停止观察
- **多阈值**：一次注册，在多种交叉状态下都能触发，不需要轮询
- 对比 scroll 事件：scroll 是同步的、每帧都触发的高频事件；Observer 是异步的、只在交叉状态变化时才触发

```tsx
const observer = new IntersectionObserver(
  (entries) => {
    const direction = getScrollDirection();
    entries.forEach(entry => {
      const el = entry.target as HTMLElement;
      if (entry.isIntersecting && direction === 'down') {
        el.animate(enterKeyframes, animOptions);
      } else if (!entry.isIntersecting && direction === 'up') {
        el.animate(leaveKeyframes, animOptions);
      }
    });
  },
  { threshold: [0, 0.1, 0.25, 0.5, 0.75, 1] }
);
```

## <a id="opt-react"></a>六、优化四：React 层面优化 → 减少非动画场景的 re-render

**定位**：虽然核心的滚动动画已经通过 animate() 脱离了 React 渲染循环，但页面还有其他会触发 setState 的场景——数据加载、用户点击、表单交互等。这些 setState 仍然会触发所有子组件的默认 re-render。

> 不是替代 animate() 方案，而是补充：animate() 解决了"滚动驱动动画"这个最高频的瓶颈，React.memo 解决剩余的"零散 setState 引发的连带 re-render"。

```tsx
const Section = React.memo(function Section({ data, onAction }: SectionProps) {
  const processedData = useMemo(() => heavyTransform(data), [data]);
  const handleClick = useCallback(() => onAction(data.id), [onAction, data.id]);
  return (
    <div className="section">
      <StaticContent data={processedData} />
      <button onClick={handleClick}>操作</button>
    </div>
  );
});
```

### React.memo 的代价——为什么不是所有组件都包一层？

```
React.memo 的开销：
1. 每次父组件渲染时，需要遍历 props 做浅比较 → 有少量 CPU 开销
2. 如果 props 几乎每次都变（比如直接传对象字面量），浅比较永远返回 false
   → 白花比较开销 + 仍然 re-render → 负优化

适用场景（收益 > 开销）：
├── 组件 props 少且多为基本类型
├── 父组件频繁渲染，但子组件 props 很少变化
└── 组件渲染开销大（DOM 深、有复杂子组件）

不适用场景：
├── props 每次都变 → memo 没用
├── 组件本身渲染极轻（一个 <span>）→ 比较开销 ≈ 渲染开销，没意义
└── props 很多且深层嵌套 → 浅比较不够用
```

## <a id="opt-webp"></a>七、优化五：图片资源优化 → WebP 替代 PNG/JPG

**问题**：页面有大量高清图片作为 模块背景和内容展示，PNG/JPG 格式文件体积大，影响首屏加载和滚动时的解码性能。

**效果**：

```
优化前：合计约 8 MB
优化后：合计约 2.5 MB（减少 ~69%）
```

**WebP 为什么更小？**

| 对比维度 | JPEG | PNG | WebP |
|---------|------|-----|------|
| **有损压缩** | DCT 算法，块状压缩 | 不支持 | VP8 帧内压缩，预测编码，更高效 |
| **无损压缩** | 不支持 | DEFLATE | 比 PNG 小 26% 左右 |
| **透明通道** | 不支持 | 支持（RGBA） | 支持（有损+无损均可） |
| **解码速度** | 快 | 中等 | 与 JPEG 接近 |

**降级方案**：

```html
<picture>
  <source srcset="hero-bg.webp" type="image/webp">
  <img src="hero-bg.jpg" alt="背景图">
</picture>
```

**WebP 的局限**：

1. **解码仍走 CPU**：大量大图同时解码仍然吃 CPU。Chrome 99+ 开始支持硬件解码，但老设备不支持
2. **不支持渐进式显示**：JPEG 可以逐行显示（progressive JPEG），WebP 只能完整下载后显示
3. **AVIF 是下一代选择**：比 WebP 再小 20-30%，且支持 HDR。但解码速度比 WebP 慢

## <a id="separation"></a>八、职责分离：为什么 Observer 回调里直接操作 DOM 是合理的？

```
这个优化的本质是搞清楚"谁该管什么事"：

React 的职责：管理组件状态和 UI = f(state) 的映射关系
             适合"数据变了 → UI 需要跟着变"的场景

渲染层的职责：执行动画（animate() 传入 keyframes 后由渲染层独立完成）
             动画是渲染层的原生能力，不依赖 JS 状态

我们的模块 动画：
- 动画定义（keyframes）→ 纯 CSS，属渲染层
- 触发时机判断（Observer）→ JS 做一次判断
- 执行 → 渲染层独立完成

换句话说：React 不应该管"动画怎么播"，它只需要管"什么时候让渲染层开始播"。
Observer 回调里调用 element.animate()，就是做了这个"通知"的职责。
不经过 setState 是刻意的——这个信息不需要进入 React 的状态管理体系。
```

## <a id="why-not-virtual"></a>九、为什么不用虚拟列表？

### 核心原因（按重要性排序）

**原因一（最致命）：Section 高度不固定，远超 100vh**

虚拟列表的核心前提是**能提前知道每个 item 的高度**（定高）或**能测量到真实高度**（动态高度）。但我的场景中：

```
Section 1: 100vh  — 标准
Section 2: 250vh  — 内容丰富，有大量图文和子动画
Section 3: 100vh  — 标准
Section 4: 180vh  — 包含可展开的详情区域
```

- 定高虚拟列表直接不可用（高度差异巨大）
- 动态高度虚拟列表需要一个"测量阶段"——先渲染每个 item 拿到真实高度
- 但渲染本身就包含了大量 DOM 节点和动画元素——测量的代价等于把性能问题先经历一遍

**原因二：动画进度与滚动位置的强绑定**

```
虚拟列表的运作方式：
┌──────────────┐
│  Item N      │ ← 可视区，渲染真实 DOM
│  Item N+1    │ ← 可视区，渲染真实 DOM
├──────────────┤ 可视区边界
│  (空白占位)   │ ← Item 0~N-1   用 padding-top 撑高
│  (空白占位)   │ ← Item N+2~End 用 padding-bottom 撑高
└──────────────┘

但动画需要：
- 模块的 offsetTop 反映真实位置
- 模块在 DOM 中持续存在以保持动画状态

冲突：
- 虚拟列表的空白区域（padding/translateY）让 scrollTop 与 模块的视觉效果脱钩
- 模块被卸载后 getBoundingClientRect() 直接返回 0
- 无法计算动画进度
```

**原因三：动画的连续性要求**

- Section 滚到可视区边缘时，动画处于中间状态
- 虚拟列表在 模块滚出可视区后会**卸载**该模块 → 所有动画状态丢失
- 重新滚回来时 模块重新 mount → 动画从 0 开始而非从上次位置继续
- 这个闪烁对用户体验是灾难性的

### 总结话术

> "虚拟列表解决的是'列表项太多导致 DOM 数量爆炸'的问题。我的场景中真正的瓶颈不是 DOM 数量（十几模块 对比成千上万列表项），而是 **scroll 事件中高频 setState 阻塞了渲染**。虚拟列表的 mount/unmount 机制会打破动画的连续性。再加上 模块高度远超 100vh 且不固定，动态高度虚拟列表的测量成本本身就不低。"

## <a id="render-pipeline"></a>十二、浏览器渲染流水线（完整版）

### 总览：七个阶段一张表

| 阶段 | 做什么 | 产出物 | 在哪条线程 |
|------|--------|--------|-----------|
| ① Parse | HTML → DOM Tree，CSS → CSSOM Tree | DOM + CSSOM | 主线程 |
| ② Style | 计算每个节点的最终样式（合并默认值、继承、选择器优先级） | ComputedStyle | 主线程 |
| ③ Layout | 计算每个节点的**几何位置和尺寸**（x, y, w, h） | Layout Tree | 主线程 |
| ④ Layer | 构建图层树，决定哪些元素提升为独立 GraphicsLayer | Layer Tree | 主线程 |
| ⑤ Paint | 为每层生成**绘制指令**（Display List：画矩形、文字、阴影…） | Display List | 主线程 |
| ⑥ Raster | 将绘制指令转为**像素位图**（GPU Texture），按 Tile 分块执行 | GPU 纹理 | Raster Worker 线程 |
| ⑦ Composite | GPU 将各层纹理按 z-order + transform 矩阵**合成**输出到屏幕 | 屏幕画面 | 合成线程 |

### will-change 底层到底做了什么？

```
will-change 触发浏览器的三个行为：

1. 创建 GraphicsLayer（合成层）
   - 将该元素的渲染结果作为独立纹理存入 GPU 显存
   - 该层的 transform/opacity 变化只影响纹理的合成位置
   - 不触发重排（Layout）和重绘（Paint）

2. 提前准备 GPU 资源
   - 浏览器通常会延迟创建合成层（按需）
   - will-change 让浏览器提前创建，避免动画首帧的创建开销

3. 建立独立的渲染上下文
   - 该元素的内容（包括子元素）渲染为单独的位图
   - 位图变化时只需重新合成，不影响其他元素
```

### 为什么 transform 不触发重排？

```
浏览器的渲染流水线（Chromium 渲染引擎）：

DOM Tree ──→ Style ──→ Layout Tree ──→ Paint Layer ──→ GraphicsLayer ──→ Display

  │              │            │               │                │
  │              │            │               │                └── transform 只影响这里
  │              │            │               └── color 等属性影响这里
  │              │            └── width/left 等属性影响这里
  │              └── 计算每个元素的具体样式
  └── 解析 HTML

关键：transform 作用于 GraphicsLayer 的合成阶段，
元素在文档流中的位置（Layout 阶段计算）没有改变。
所以不需要重新 Layout，也不需要重新 Paint。
```

### Tiles（瓦片化）—— 为什么长页面不会占满 GPU 显存

浏览器把每层切成 256×256px 的 **Tile（瓦片）**，只栅格化视口附近的：

```
页面切分示意（每个 □ = 256×256 Tile）：

┌──┬──┬──┐
│ □│ □│ □│  ← 视口上方，未栅格化
├──┼──┼──┤
│ □│ ■│ □│  ← ■ 视口内的 Tile，已栅格化，纹理在 GPU 缓存
├──┼──┼──┤
│ □│ □│ □│  ← 视口下方，未栅格化
└──┴──┴──┘
```

> 面试一句话："浏览器把页面切成 256px 的小块，只把屏幕附近的变成像素存 GPU 里，远的一律不动。"

### 两条线程的分工——为什么 animate() 不会被 JS 卡顿影响

| | 主线程（Main Thread） | 合成线程（Impl Thread） |
|---|---|---|
| **负责** | JS 执行、Style、Layout、Paint | Raster 调度、Composite、VSync 响应 |
| **触发条件** | JS 操作 DOM/样式、resize、scroll 等 | 每次 VSync 自动运行 |
| **能不能被 JS 阻塞？** | ✅ 就是被它阻塞的 | ❌ 独立运行，主线程卡死也不影响 |

```
这意味着什么？

场景：页面上有一个 JS 长任务跑了 200ms

主线程：被阻塞 200ms，React setState 全卡住，Layout/Paint 全停
合成线程：照常跑！每 16.67ms 照常 Composite，往屏幕输出画面

→ 如果动画用的是 CSS Animation / animate()，且只改了 transform/opacity
→ 这 200ms 里动画仍然 60fps 流畅运行
→ 但如果动画走的是 JS setState → 这 200ms 里完全卡死

这就是我们优化方案选 animate() 而不继续用 setState 的底层原因。
```

### 完整示例：滚动动画的两种渲染路径

```
用户滚动 → Section 动画元素 transform: translateY(20px → 0px)

━━━ 路径 A：无独立合成层（没加 will-change）━━━

JavaScript   Style     Layout    Paint      Raster    Composite
(处理滚动)   (重算)    (重排)    (指令)     (像素)     (合成)
  ~2ms       ~1ms      ~3ms      ~5ms       ~4ms       ~1ms
                                            ↑
                                    纹理要重新生成！
总耗时 ≈ 16ms → 撞到帧预算边界 → 随时掉帧

━━━ 路径 B：有独立合成层（will-change 已生效）━━━

JavaScript   Style     Layout    Paint      Raster    Composite
(处理滚动)   (跳过)    (跳过)    (跳过)     (跳过)     (合成)
  ~2ms                                              ~1ms
                                                     ↑
                                          纹理已在 GPU 缓存中
                                          只改变换矩阵！
总耗时 ≈ 3ms → 帧预算绰绰有余 → 稳定 60fps

will-change 的价值：
不是让 transform 本身更快，
而是让 transform 变化时"需要做的事"从 6 步缩减为 2 步。
```

### 会触发 Layout 的操作（能避免就避免）

```
读取（强制同步布局——最隐蔽的性能杀手）：
offsetTop/Left/Width/Height、scrollTop/Left/Width/Height、
clientTop/Left/Width/Height、getComputedStyle()、
getBoundingClientRect()、getClientRects()

写入：
width/height/padding/margin/border/top/left/right/bottom
display、position、float、overflow
添加/删除 DOM 节点、修改文本内容

→ 原则：批量读 → 批量写。读写在同一个帧内交替 = 强制同步布局
→ 更佳：用 transform 替代 left/top，用 opacity 替代 visibility
```

## <a id="url-to-page"></a>十三、从输入 URL 到页面显示（必考）

### 总览：七大阶段

```
① URL 解析 → ② DNS 解析 → ③ TCP 连接 → ④ TLS 握手
→ ⑤ HTTP 请求/响应 → ⑥ 浏览器解析 → ⑦ 渲染流水线
```

### 阶段一：URL 解析

浏览器判断你输入的是"搜索关键词"还是"合法 URL"，补全协议和端口。

### 阶段二：DNS 解析 —— 域名 → IP

**查询链路（逐级缓存）**：

```
浏览器 DNS 缓存
  → OS hosts 文件
  → 路由器缓存
  → ISP DNS 服务器（运营商）
  → 根域名服务器（.）
  → 顶级域名服务器（.com）
  → 权威 DNS 服务器（baidu.com）
```

DNS 查询传统用 UDP（端口 53）。现在也有 DoH 和 DoT，用加密信道防止 DNS 劫持。

### 阶段三：TCP 连接 —— 三次握手

```
客户端                          服务器
  │                               │
  │── SYN (seq=x) ──────────────→│  ① 客户端：我要连接，初始序号 x
  │                               │
  │←─ SYN+ACK (seq=y, ack=x+1) ──│  ② 服务器：收到，我的序号 y，确认 x+1
  │                               │
  │── ACK (ack=y+1) ────────────→│  ③ 客户端：确认 y+1，连接建立
  │                               │
  │◄═══════ 连接建立 ═══════════►│
```

**为什么是三次？** 两次不够——如果服务器回的 SYN+ACK 丢了，客户端认为连接失败，服务器却以为连接已建立，浪费资源。三次刚好让双方都确认了"自己能发也能收"。

### 阶段四：TLS 握手（HTTPS 专属）

HTTPS 用到了**两种加密**，各司其职：

| | 对称加密 | 非对称加密 |
|---|---|---|
| **密钥** | 加解密用**同一把**密钥 | 一把**公钥**加密，一把**私钥**解密 |
| **速度** | **快**（毫秒级） | **慢**（是前者的 100-1000 倍） |
| **作用** | 加密**实际传输的数据** | 加密**对称密钥的交换过程** |

**核心思路：用非对称加密来安全地传输对称密钥，然后用对称加密来高效传输实际数据。**

**会话密钥是怎么生成的？**

```
三方各自贡献一部分随机数，用 PRF（伪随机函数）混合：

Client Random   —— 客户端在 ClientHello 中提供（明文）
Server Random   —— 服务器在 ServerHello 中提供（明文）
Premaster Secret —— 客户端生成，用服务器公钥加密传输

    ↓  三方混合，通过 PRF 函数

Master Secret（主密钥）
    ↓  再通过 PRF 分成

Session Keys（会话密钥）：
├── 客户端 → 服务器 的对称密钥
├── 服务器 → 客户端 的对称密钥（通常是不同的！）
└── HMAC 密钥（用于消息完整性校验）
```

> **关键理解**：即使攻击者监听到了 Client Random 和 Server Random（它们是明文的），但没有 Premaster Secret（被公钥加密了，只有服务器私钥能解开），就推不出会话密钥。

**TLS 1.3 的改进**：握手从 2-RTT 降到 1-RTT，删掉了 RSA 密钥交换，强制前向安全性。

### 阶段五：HTTP 请求与响应

#### HTTP 缓存（面试高频！）

**决策流程**：

```
浏览器请求资源
      │
      ▼
┌─ 强缓存命中？ ────────────────────────────┐
│ (Cache-Control / Expires 未过期)            │
│                                            │
│ YES → 直接从本地缓存取（200 from cache）      │
│                                            │
│ NO（过期了 / 没配）                          │
│   │                                        │
│   ▼                                        │
│ 浏览器发起请求，带上验证头                   │
│   │                                        │
│   ▼                                        │
│ 服务器判断资源是否变了                       │
│   │                                        │
│   ├─ 没变 → 304 Not Modified（协商缓存命中）  │
│   └─ 变了 → 200 OK + 新资源                  │
└────────────────────────────────────────────┘
```

**实际工程中怎么配？**

```
HTML 文件（index.html）：
  Cache-Control: no-cache            ← 每次都验证

带 hash 的静态资源（app.a1b2c3.js）：
  Cache-Control: max-age=31536000, immutable  ← 缓存 1 年

图片（logo.png，不带 hash）：
  Cache-Control: max-age=86400      ← 缓存 1 天
  ETag: "xyz789"                    ← 1 天后走协商缓存
```

### 阶段六：浏览器解析 —— 字节 → DOM / CSSOM

遇到 `<script>` 怎么办？

```
<脚本解析阶段>
├── <script src="...">      → 暂停 HTML 解析，下载 + 执行 JS
├── <script async>          → 并行下载，下载完立即执行
├── <script defer>          → 并行下载，等 HTML 解析完再执行
└── <link rel="stylesheet"> → 并行下载 CSS，不阻塞 HTML 解析；但阻塞 JS 执行
```

### 阶段七：渲染流水线 —— DOM/CSSOM → 屏幕像素

```
DOM Tree + CSSOM Tree
        │
        ▼
   Render Tree（只包含可见节点）
        │
        ▼
   Layout（重排：几何位置和大小）
        │
        ▼
   Paint（重绘：生成位图）
        │
        ▼
   Composite（合成：按 z-index 合并为最终画面）
        │
        ▼
   屏幕显示
```

## <a id="taro-principle"></a>十四、TaroJS 跨平台原理

### 编译时做了什么？

**线 1：React 代码本身的编译（Babel）**——跟普通 React 项目一样。

**线 2：Taro 做的额外工作——名字翻译 + 模板生成**

```
① 标签名翻译：  <View> → <view>、<Text> → <text>
② 属性名翻译：  className → class
③ 生成小程序模板文件（WXML）
④ 生成小程序配置文件（app.json / page.json）
```

### 运行时做了什么？

React 需要 `document.createElement`、`appendChild` 等浏览器 API，小程序不提供这些。Taro 运行时模拟了它们：

```
React 调用                              Taro 提供
─────────                              ────────
document.createElement('view')    →    new TaroElement('view')
element.appendChild(child)       →    TaroElement.appendChild()
element.addEventListener(...)    →    注册到 Taro 事件系统
```

模拟完之后通过 `setData()` 把变更传到渲染层。

### 适配层

适配层 = 统一接口 + 平台分支 + 编译时 Tree Shaking：

```ts
function request(options) {
  if (process.env.TARO_ENV === 'weapp') return wx.request(options);
  if (process.env.TARO_ENV === 'alipay') return my.request(options);
  if (process.env.TARO_ENV === 'h5') return fetch(options.url, options);
}
```

编译微信小程序时，`process.env.TARO_ENV` 被替换为 `'weapp'`，其他分支被 Tree Shaking 删除。

### Taro 方案的局限性

1. **setData 通信瓶颈不可消除**：逻辑层和渲染层分离是物理架构
2. **运行时体积**：模拟 DOM/BOM 的代码有几十 KB
3. **复杂交互不如原生**：高频手势、实时动画多了一层中转。我们动画选 `animate()` 而不是 React state 驱动，就是有意识地绕过了这层中转
4. **调试链路长**：源码 → 编译 → 小程序运行，需要 source map
5. **平台差异无法 100% 抹平**：总有需要写平台判断的时候

# InterSection Observer
 
 从浏览器渲染进程架构的角度，IntersectionObserver 之所以快，核心在于它**巧妙地利用了浏览器的多线程架构，将计算工作从主线程卸载到了合成器线程（Compositor Thread）**。

## 浏览器渲染进程架构基础

现代浏览器（如 Chrome）采用**多进程架构**，每个标签页对应一个独立的**渲染进程（Renderer Process）**。渲染进程内部包含多个关键线程：

```
┌─────────────────────────────────────────┐
│        渲染进程 (Renderer Process)        │
├─────────────────────────────────────────┤
│  ┌─────────────┐  ┌──────────────┐      │
│  │ 主线程      │  │ 合成器线程    │      │
│  │ (Main       │  │ (Compositor  │      │
│  │  Thread)    │  │  Thread)     │      │
│  ├─────────────┤  ├──────────────┤      │
│  │ • JS执行    │  │ • 图层合成    │      │
│  │ • 布局计算  │  │ • 视口检测    │      │
│  │ • 样式计算  │  │ • 滚动处理    │      │
│  │ • 重排重绘  │  │ • 交叉检测    │      │
│  └─────────────┘  └──────────────┘      │
│  ┌─────────────┐  ┌──────────────┐      │
│  │ 光栅化线程  │  │ 网络线程      │      │
│  │ (Raster     │  │ (Network     │      │
│  │  Thread)    │  │  Thread)     │      │
│  └─────────────┘  └──────────────┘      │
└─────────────────────────────────────────┘
```

## IntersectionObserver 的底层实现原理

### 1. **主线程的"甩锅"机制**

当创建 `IntersectionObserver` 实例时：

```javascript
const observer = new IntersectionObserver(callback, options);
observer.observe(targetElement);
```

浏览器执行以下操作：

- **主线程**：只负责**注册观察器**和**接收回调通知**
- **合成器线程**：接管**实际的交叉检测计算**

```javascript
// 伪代码示意
// 主线程
function createObserver(callback, options) {
  // 1. 将观察器信息注册到合成器线程
  compositorThread.registerObserver({
    target: targetElement,
    root: options.root,
    threshold: options.threshold,
    rootMargin: options.rootMargin
  });
  
  // 2. 主线程立即返回，不阻塞
  return observer;
}

// 合成器线程（独立运行）
function compositorThreadLoop() {
  while (true) {
    // 在每一帧的合成阶段
    for (let observer of registeredObservers) {
      // 计算目标元素与视口的交叉状态
      const intersection = calculateIntersection(observer.target);
      
      // 如果状态变化，通过 IPC 通知主线程
      if (intersection.changed) {
        ipc.postMessageToMainThread({
          type: 'INTERSECTION_CHANGE',
          data: intersection
        });
      }
    }
    // 等待下一帧
    await nextFrame();
  }
}
```

### 2. **合成器线程的优势**

合成器线程专门负责**图层合成和滚动处理**，具有以下特性：

- **独立于主线程**：即使主线程被 JavaScript 阻塞，合成器线程仍能正常工作
- **硬件加速**：利用 GPU 进行图层变换和合成
- **无需重排重绘**：直接操作图层位置信息，不触发布局计算

### 3. **与传统 scroll 监听的对比**

#### 传统方案（主线程阻塞）：

```javascript
// ❌ 性能差：所有计算都在主线程
window.addEventListener('scroll', () => {
  // 1. 触发回流（Reflow）
  const rect = element.getBoundingClientRect();
  //    ↑ 这个调用会强制浏览器重新计算布局
  
  // 2. 主线程同步执行
  if (rect.top < window.innerHeight) {
    loadImage();
  }
  
  // 3. 需要手动防抖节流
});
```

**性能瓶颈**：
- 每次滚动都触发，高频调用
- `getBoundingClientRect()` 触发**强制同步布局（Forced Synchronous Layout）**
- 主线程被阻塞，影响其他任务（如用户交互、动画）

#### IntersectionObserver（合成器线程处理）：

```javascript
// ✅ 性能优：计算在合成器线程
const observer = new IntersectionObserver((entries) => {
  // 回调在主线程执行，但触发频率极低
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      loadImage();
    }
  });
});

observer.observe(element);
```

**性能优势**：
- **计算卸载**：交叉检测在合成器线程完成
- **异步通知**：通过 IPC 机制通知主线程，非阻塞
- **智能批处理**：浏览器在空闲时批量处理回调

## 关键技术细节

### 4. **图层（Layer）级别的检测**

合成器线程维护着页面的**图层树（Layer Tree）**：

```
Layer Tree (合成器线程维护)
├── Root Layer (视口)
│   ├── Layer 1 (静态内容)
│   ├── Layer 2 (滚动容器)
│   │   ├── Layer 2.1 (目标元素) ← IntersectionObserver 监听这个
│   │   └── Layer 2.2 (其他元素)
│   └── Layer 3 (固定定位元素)
```

当页面滚动时：
- 合成器线程**只需更新图层的变换矩阵（Transform Matrix）**
- 直接比较图层边界框（Layer Bounds）与视口的交叉
- **无需重新计算布局（Layout）**

### 5. **IPC 通信机制**

当交叉状态变化时，合成器线程通过**进程间通信（IPC）** 通知主线程：

```
合成器线程                     主线程
    │                           │
    │ 交叉状态变化               │
    ├───────────────────────────┤
    │  IPC 消息:                │
    │  {                        │
    │    type: 'INTERSECTION',  │
    │    entry: {...}           │
    │  }                        │
    │                           │
    │                           ├─→ 触发回调函数
    │                           │    (异步执行)
    └───────────────────────────┘
```

### 6. **与浏览器渲染管线的集成**

IntersectionObserver 与浏览器的**渲染管线（Rendering Pipeline）** 深度集成：

```
每一帧的渲染流程：
┌─────────────────────────────────────────┐
│  1. 主线程:                             │
│     • 执行 JavaScript                   │
│     • 更新 DOM/CSSOM                    │
│     • 计算样式                          │
├─────────────────────────────────────────┤
│  2. 合成器线程:                         │
│     • 接收图层更新                      │
│     • ✅ 执行 IntersectionObserver 检测  │ ← 关键！
│     • 合成图层                          │
│     • 发送绘制命令到 GPU                │
├─────────────────────────────────────────┤
│  3. GPU:                                │
│     • 光栅化                            │
│     • 显示到屏幕                        │
└─────────────────────────────────────────┘
```
 

## 总结

IntersectionObserver 快的根本原因是：

1. **架构优化**：利用浏览器多线程架构，将计算从主线程卸载到合成器线程
