**面试官**：讲讲你做过最有挑战的项目吧。

**我**：

> 我在保时捷经销商系统里从零搭建了一套跨端组件库。背景是这样的——项目要同时跑在微信小程序、抖音小程序和 H5 三个端上。一开始每个端各写一套组件，代码互相抄，改一个按钮样式要改三个地方。组件质量越改越不一致，改 bug 更是灾难。
>
> 我的角色是前端负责人，从技术选型到架构设计到落地推广，全程主导。
>
> 先说技术选型。跨端框架当时能选的其实就 Taro 和 uni-app，我们团队全是写 React 的，uni-app 从一开始就是给 Vue 设计的，React 支持是后来加的，用起来没那么顺手。Taro 3 的思路是让 React 完整跑在逻辑层，我们不用改写法习惯。代价也有——运行时会多几十 KB，复杂动画多一层中转性能不如原生，但我们的页面主要是表单和列表，不是重动画场景，这个代价完全能接受。
>
> 框架定了之后，接下来是组件库本身的架构。核心思路是分层——每个组件目录下建一个 `adapter/` 文件夹，里面按平台拆成 `weapp.ts`、`tt.ts`、`h5.ts` 三个文件。组件本体不写 `if/else` 判断平台，只调用适配器。比如 Button 组件，`adapter/weapp.ts` 里处理微信的 `openType` 和原生 `loading`，`adapter/tt.ts` 处理抖音的差异，`adapter/h5.ts` 用 CSS 模拟小程序才有的能力。业务方只引入 `<Button>` 就行，不用关心底下是哪个端。
>
> 工程上用了 yarn workspace + Lerna 的 Monorepo 结构，五个包：`components` 放 UI 组件，`styles` 放设计 Token 和 CSS Variables 主题，`utils` 放平台检测和单位转换，`demo` 是一个 Taro 应用用来做三端真机验证，`storybook` 用来隔离开发组件。yarn workspace 管包之间的依赖引用——demo 通过 alias 直接指到 components 源码，改一行代码热更新生效，不需要先 build。Lerna 用的是 independent 模式，每个包独立版本号，改一个 Button 的样式只升 `components` 的版本，`styles` 和 `utils` 不受影响。发布走 `lerna version --no-private` 自动检测改动范围、生成 changelog，然后 `lerna publish from-git` 推到 npm。
>
> 最终一套代码覆盖三个端，代码复用率 70%，新页面接入从两周缩短到三天。
>

**面试官**：Taro 和 uni-app 的运行时方案具体有什么区别？

**我**：

> 核心区别在思路。uni-app 走的是编译时路线——你写的 Vue 模板，它直接编译成小程序的 WXML 模板，相当于把你的代码翻译了一遍。好处是产物干净，没有额外的运行时开销。但问题是它从一开始就是为 Vue 设计的，React 支持是后来加的，hooks 的一些用法会有兼容问题。
>
> Taro 3 反过来——它不翻译你的代码，而是在小程序逻辑层里模拟了一套浏览器环境。`document.createElement`、`appendChild` 这些 API，Taro 自己实现了一遍。React 以为自己跑在浏览器里，其实跑在 Taro 模拟的容器里。好处是 React 特性完整支持，我们不用改写法。代价就是那层模拟环境多了几十 KB，复杂交互比如高频动画多了一层中转。
>
> 后来 Taro 4 也在往编译时方向靠，算是两种思路在互相取长补短。
>
> [→ 选型框架](#selection) | [→ Taro 原理](#taro-principle)

---

**面试官**：组件库的架构是怎么设计的？

**我**：

> 核心思路是**分层**——把"不变的"和"会变的"拆开。
>
> 不变的放在 `packages/styles` 和 `packages/utils`。`styles` 里是设计 Token 和 CSS Variables 主题变量，主题切换只需要换 `data-theme` 属性，不用重新编译。`utils` 放平台检测（`isWeapp`、`isTT`、`isH5`）和单位转换（`pxToRpx` 之类的）。
>
> 会变的放在每个组件自己的 `adapter/` 目录下。比如 Button 组件，`adapter/weapp.ts` 导出 `getWeappButtonProps`，`adapter/tt.ts` 导出 `getTtButtonProps`，`adapter/h5.ts` 导出 `getH5ButtonProps`。组件本体用 `useMemo` 根据当前平台选对应的适配器，合并 props。这样平台差异完全隔离在 adapter 文件里，组件本体不写一行平台判断。
>
> 还有一个设计决策值得一提——Storybook 不依赖 Taro 运行时。Button 组件里做了降级：如果 `TaroButton` 不可用就回退到原生 `<button>`。这样在 Storybook 的纯 React 环境里也能渲染和调试组件，不需要启动小程序模拟器。
>
> 工程上用了 yarn workspace + Lerna 的 Monorepo 结构。`packages/components` 放 UI 组件，`packages/styles` 放设计 Token 和 CSS Variables 主题，`packages/utils` 放平台检测和工具函数，`packages/demo` 是 Taro 应用做三端真机验证，`packages/storybook` 用来隔离开发组件。yarn workspace 管包之间的依赖引用——demo 通过 alias 直接指到 components 源码，改一行代码热更新生效，不需要先 build。Lerna 用的是 independent 模式，每个包独立版本号，改一个 Button 的样式只升 `components` 的版本，`styles` 和 `utils` 不受影响。
>
> [→ 架构设计](#architecture) | [→ 适配层详解](#adapter)

**面试官**：适配层具体怎么抹平平台差异？

**我**：

> 三种策略，按场景选择。
>
> 第一种是**编译时抹平**——利用 Taro 的条件编译。比如 `htmlType` 属性只有 H5 有（H5 的 `<button type="submit">` 可以触发表单提交），小程序没有这个概念。写成 `if (process.env.TARO_ENV === 'h5')`，编译微信和抖音时这段代码被 Tree Shaking 删掉，产物里不留痕迹。
>
> 第二种是**运行时抹平**——封装统一 API，内部按平台调用不同的原生 API。比如 `Storage.get(key)`，微信调 `wx.getStorageSync`，抖音调 `tt.getStorageSync`，H5 直接用 `localStorage.getItem`。平台检测用 `Taro.getEnv()` 封装了一层，带 H5 兜底——Storybook 环境里 Taro 没初始化，默认按 H5 处理，这样在纯 React 环境里也能跑。
>
> 第三种是**降级策略**——某个平台不支持的能力，用替代方案。举真实的例子，Button 的 `loading` 属性在微信和抖音上是原生支持的，H5 没有，我们在 H5 适配器里用 CSS class 模拟了一个 loading 态。再比如 `openType` 在微信和抖音上都支持，但 H5 完全没这个概念，H5 端就降级为普通的 `onClick` 处理。每个组件的文档里会标兼容情况，业务方清楚哪些端有什么限制。
>
> 最终效果：业务方引入 `<Button>` 这个统一组件就行，不用关心底层是微信还是抖音。
>
> [→ 适配策略](#adapter) | [→ 降级策略](#degradation)

**面试官**：这个过程里遇到了什么坑？

**我**：

> 说实话，坑比预想的多。说三个典型的。
>
> 第一个，Storybook 环境的 Taro 依赖问题。组件里用了 `@tarojs/components` 的 `Button`，但 Storybook 是纯 React 环境，Taro 根本没初始化，`TaroButton` 是 `undefined`，组件直接报错。解决方式是在组件里做了一层降级——`const ButtonElement = TaroButton || 'button'`，Storybook 环境自动回退到原生 `<button>`。后来这个降级逻辑还顺便解决了单元测试环境的问题——Jest 里也没有 Taro 运行时。
>
> 第二个，微信小程序主包 2MB 的限制。Taro 运行时本身就占了几十 KB，加上组件库代码，主包很快就逼近上限了。解决方案是把组件库拆分——基础组件放主包，业务组件全部放到分包按需加载。同时构建上做了优化——`utils` 的源码直接打进 `components` 的产物里，消费者只装一个包，不用额外引入 `utils`。
>
> 第三个，也是最头疼的——组件 API 设计。一开始让每个组件尽量暴露平台原生能力，比如把微信 `button` 的所有属性都透传出去。结果 API 又臭又长，换个平台很多属性就无效了。后来改成"最小公共接口"——优先暴露三个端共有的核心属性，平台特有的通过 `platformProps` 按需传入。这看起来限制了灵活性，实际上让组件用起来简单很多。
>
> [→ 踩坑记录](#pitfalls)

**面试官**：你说最终代码复用率 70%，这个数字怎么来的？剩下的 30% 为什么复用了？

**我**：

> 70% 是按代码行数算的——组件库总代码里，70% 是各端共享的通用逻辑和组件模板。
>
> 复用不了的那 30%，主要是三类。第一是平台特有的适配代码——比如 Button 的 `openType` 在微信和抖音上用法有细微差异，`size="mini"` 只有微信支持，这些对应的 adapter 文件就是不可复用的。第二是样式校准——同一套 CSS Variables 在不同平台渲染效果有细微差异，需要平台特定样式微调。第三是各端的入口配置——微信的 `app.json` 和抖音的 `app.json` 结构不同，没法统一。
>
> 说实话 70% 不是终点。如果现在重新做，我会把更多的逻辑抽到通用层——当初有些为了赶进度直接在组件里写了平台判断，其实可以用适配器模式统一处理的。
>
> [→ 复用率分析](#reuse)

**面试官**：如果重新做，你会怎么选型和设计？

**我**：

> 选型上，如果只做微信和 H5，我会认真考虑直接用原生开发——微信小程序原生 + React H5，通过一套设计规范保证一致性，不走跨端框架。跨端框架在两端差异大的时候反而是累赘。
>
> 架构上，最大的遗憾是没有一开始就做好组件文档和单测。组件库建好之后推广到业务团队，很多人不敢用，因为不知道组件有哪些参数、边界情况处理得怎么样。后来花了大力气补文档和 Storybook，如果能重来，第一个组件写完就先搭好文档体系。
>
> 另外，现在 CSS Scroll-driven Animations、View Transitions API 这些新标准在跨端框架里完全不支持，如果有重动画需求，可能需要在组件库里预留原生动画的注入点，让业务方在某些端可以用原生实现替代跨端方案。
>
> [→ 反思与改进](#reflection)

---

**面试官**：你从中学到了什么？

**我**：

> 最大的收获是**技术选型得有依据**。选 Taro 不是因为"它火"或者"大家都用"，而是因为我们团队全是 React、uni-app 对 React 支持不够好、而且我们的场景不是重动画不需要极致性能。把每个选项的适用边界想清楚，而不是拍脑袋。这个习惯可以复用任何技术选型。
>
> 第二是**抽象要在有足够样本之后做**。组件库初期我急于设计"完美的通用接口"，结果有些抽象过度——一个简单的按钮组件过度设计了十几个参数，用起来反而比原生还麻烦。后来改成先写具体实现、发现三个以上重复模式再抽取，抽象质量明显更高。
>
> 第三是**组件库不是写完就完事了**，文档、测试、版本管理、升级迁移指南——这些"配套工程"才是决定一个组件库能不能被团队用起来的关键。

## <a id="bg"></a>一、项目背景

### 为什么需要跨端组件库？

保时捷经销商系统的业务需要在**三个端**上运行：

| 端 | 技术载体 | 特点 |
|---|---------|------|
| 微信小程序 | 微信原生 + Taro | 用户量大，API 丰富，但审核严格 |
| 抖音小程序 | 抖音原生 + Taro | 用户增长快，但 API 不如微信完善 |
| H5 | React SPA | 浏览器环境，能力最强，但无原生能力 |

**组件库建设前的问题**：

```
改一个 Button 圆角：
├── weapp/Button/index.jsx   ← 改
├── douyin/Button/index.jsx  ← 改
└── h5/Button/index.tsx      ← 改

三个仓库，改一个地方要同步两次，一次漏改 = 线上 UI 不一致
```

**组件库建设后的效果**：

```
改一个 Button 圆角：
└── packages/components/Button/index.tsx  ← 改一次，三端生效
```

3. **平台差异抹平**：不同端的能力差异怎么处理？

## <a id="selection"></a>二、技术选型：结构化决策框架

> 🚨 **面试要点**：选型题考的不是你选了谁，而是**你的决策逻辑**。一定要展示"需求 → 候选 → 评估 → 决策"的完整链路，而不是"大家都用这个"。

### 决策四步法

```
第一步：明确核心需求
├── 技术栈约束：团队是 React 背景（7 人，全部 React）
├── 端覆盖要求：微信小程序 + 抖音小程序（必须）；H5 + H5（计划）
├── 复用目标：业务组件一套代码，多端运行
└── 组织约束：多个业务项目需要共享组件

第二步：列出候选方案
├── Taro 3.x（React + 运行时适配）
├── uni-app（Vue + 编译时转换）
├── Remax（React + 运行时，类似 Taro）
├── 原生开发 × 4（每个端各写一套）
└── 自研框架（排除：团队规模不支持）

第三步：建立评估维度
├── React 兼容度：对 React 写法的支持程度
├── 多端支持：微信/抖音/H5 的覆盖度
├── 社区活跃度：GitHub stars、issue 响应速度、更新频率
├── 技术栈契合度：与团队现有技术栈的匹配度
└── 迁移成本：从零接入的工作量

第四步：打分决策
```

### 评估矩阵

| 评估维度（权重） | Taro 3.x | uni-app | Remax | 原生 × 4 |
|----------------|----------|---------|-------|----------|
| React 兼容度 (30%) | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 多端支持 (25%) | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 社区活跃度 (20%) | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ | — |
| 技术栈契合 (15%) | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 迁移成本 (10%) | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ |
| **加权总分** | **4.5** | 3.2 | 3.3 | 3.6 |

### Taro 胜出的关键原因

**Taro 3.x 的运行时方案 vs uni-app 的编译时方案**：

```
Taro 3.x（运行时适配）：
React 代码 ──→ Babel 编译 ──→ React Reconciler（正常运行）
                                    │
                              Taro DOM 模拟层（TaroElement）
                                    │
                              setData ──→ 小程序渲染层

关键优势：React 可以完整运行，hooks、context、Suspense 等
React 特性都能用，写法习惯完全保留。

uni-app（编译时适配——主要指 Vue 模板编译路线）：
Vue 模板 ──→ uni-app 编译器 ──→ 小程序 WXML 模板
                                    │
                              JS 逻辑 ──→ setData ──→ 渲染层

Vue 模板直接编译成小程序模板，对 Vue 生态更友好。
但 uni-app 的 React 支持是后来加的，不少 React 特性有限制。
```

### Tradeoff 坦诚说

> "选了 Taro 也有一些代价，我可以说说。第一，运行时体积多了几十 KB——Taro 模拟 DOM/BOM 的代码在小程序主包里占了约 60KB。第二，复杂交互不如原生——比如高频手势、实时动画，Taro 多了一层运行时中转，性能不如原生。我们的动画方案就用 `element.animate()` API 绕过了 Taro 这层中转。第三，多端调试链路长——源码到最终产物经过了 Babel + Taro 编译器两层转换，source map 有时候定位不准。"

### 面试话术（选型题通用模板）

> "这个选型我走了四步。第一步明确需求——列出必须满足的硬约束。第二步拉候选方案——把市面上能做这件事的方案都列出来。第三步建立评估维度——根据需求给每个维度设权重。第四步打分决策。最终选了 Taro，但我也会把每个选项的 tradeoff 说清楚，让团队知道我们放弃了什么。"

## <a id="architecture"></a>三、组件库架构设计

### 整体架构

```
┌──────────────────────────────────────────────────┐
│                业务层（各业务项目）                  │
│    项目 A（微信）   项目 B（抖音）   项目 C（H5）    │
└─────────────────────┬────────────────────────────┘
                      │ npm install / alias
┌─────────────────────▼────────────────────────────┐
│              组件库 Monorepo                       │
│                                                  │
│  packages/components/     UI 组件                  │
│  └── Button/adapter/                              │
```

| 包 | 职责 | 是否发布 npm |

### 设计原则

```
1. 最小公共接口原则
   组件只暴露三端共有的能力作为主要 API。
   平台特有能力通过 platformProps 透传，不污染主接口。

   反例：
   <Button
     openType="share"            ← 微信和抖音支持，H5 不支持
     size="mini"                 ← 只有微信支持
     htmlType="submit"           ← 只有 H5 支持
     ...
     ...
   />

   正例：
   <Button
     type="primary"
     size="large"
     onTap={handleClick}
     platformProps={{        ← 平台特有属性收敛在这里
       weapp: { openType: 'share', size: 'mini' },
       douyin: { openType: 'share' },
       h5: { htmlType: 'submit' },
     }}
   />

2. 渐进式抽象原则
   先写具体实现 → 发现三个以上相似模式 → 再抽取通用逻辑。
   不要一开始就追求"完美的通用接口"。

3. 适配器模式
   每个平台差异封装为独立的适配器，组件本体不写 if/else。
```

### 组件设计模式：以 Button 为例

```tsx
// packages/components/src/Button/Button.tsx

import { getWeappButtonProps, isWeapp } from './adapter/weapp';
import { getTtButtonProps, isTt } from './adapter/tt';
import { getH5ButtonProps, isH5 } from './adapter/h5';

// Storybook / 单测环境降级：Taro 未初始化时回退到原生 button
const ButtonElement: any =
  typeof TaroButton === 'function' ? TaroButton : 'button';

export const Button = forwardRef<ButtonInstance, ButtonProps>((props, ref) => {
  const { className, ...restProps } = props;

  // 根据当前平台选择对应的适配器
  const platformProps = useMemo(() => {
    if (isWeapp()) return getWeappButtonProps(props);
    if (isTt())    return getTtButtonProps(props);
    if (isH5())    return getH5ButtonProps(props);
    return {};
  }, [props]);

  return (
    <ButtonElement
      ref={ref}
      className={classNames('ui-button', className)}
      {...platformProps}
      {...restProps}
    />
  );
});
```

## <a id="adapter"></a>四、平台差异抹平：三种策略

### 策略对比

```
策略              │  适用场景          │  开销    │  例子
─────────────────┼───────────────────┼─────────┼──────────
编译时抹平         │  能力有无（A 有 B 无）│  零运行时  │  htmlType（仅 H5 支持）
运行时抹平（适配器）│  API 不同名同功能    │  少量封装  │  Storage、Request、Navigate
降级策略           │  UI 能力差异        │  有损体验  │  H5 loading CSS 模拟、openType H5 降级 onClick
```

### 策略一：编译时抹平（条件编译）

```tsx
// 编译时 platform 是常量，不满足条件的分支被 Tree Shaking 删除

if (process.env.TARO_ENV === 'h5') {
  // htmlType 只有 H5 支持（<button type="submit">）
  // 编译微信和抖音时整段被 Tree Shaking 删除
  return <button htmlType="submit">提交</button>;
}

if (process.env.TARO_ENV === 'weapp') {
  // size="mini" 只有微信支持，抖音编译时这段被删除
  return <button size="mini">小按钮</button>;
}
```

**适用场景**：某个能力只有特定平台有，其他平台没有等价物。

### 策略二：运行时抹平（适配器模式）

```tsx
// packages/utils/src/adapter/storage.ts

// 对外暴露统一接口
export const Storage = {
  get(key: string): string | null {
    return getPlatformStorage().getSync(key);
  },
  set(key: string, value: string): void {
    getPlatformStorage().setSync(key, value);
  },
};

// 内部按平台分发（编译时 Tree Shaking 选择对应实现）
function getPlatformStorage() {
  if (process.env.TARO_ENV === 'weapp') return wx;
  if (process.env.TARO_ENV === 'douyin') return tt;
  if (process.env.TARO_ENV === 'h5') return webStorage;
}
```

**适用场景**：不同平台 API 名称不同，但功能相同。

### 策略三：降级策略

以 Button 组件为例，真实的能力差异：

```
能力        │  微信  │  抖音  │  H5
───────────┼───────┼───────┼──────
openType    │  ✅   │  ✅   │  ❌
loading     │  ✅   │  ✅   │  ❌（CSS class 模拟）
size (mini) │  ✅   │  ❌   │  ❌（CSS class 控制）
htmlType    │  ❌   │  ❌   │  ✅
```

降级规则：
├── `openType` H5 不可用 → 降级为 `onClick` 手动处理
├── `loading` H5 不可用 → CSS class 模拟 loading 态
├── `size="mini"` 仅微信支持 → 抖音和 H5 用 CSS 模拟小尺寸
└── `htmlType` 仅 H5 支持 → 小程序端忽略该属性

**降级标注**：每个组件文档顶部标注兼容性矩阵，业务方清楚知道哪些端有什么限制。

## <a id="pitfalls"></a>五、踩坑记录

### 坑 1：抖音 CSS 差异 —— `overflow: scroll` 不生效

```
现象：H5 和微信正常滚动的列表，抖音端完全不能滚动。

排查：抖音小程序的渲染引擎对 -webkit-overflow-scrolling 的处理不同，
      且部分 CSS 属性需要加抖音私有前缀。

解决：用 scroll-view 组件替代原生 overflow: scroll。
      在组件库里统一封装 <ScrollContainer>，内部自动按平台选择实现。

教训：不要假设各端 CSS 行为一致。组件库里能封装的就封装，
      不要让业务方去处理这些差异。
```

### 坑 2：微信主包体积限制

```
现象：微信小程序主包限制 2MB，
      组件库 + Taro 运行时直接超标。

排查工具：IDE 的包体积分析面板 + webpack-bundle-analyzer。

解决：
1. 组件库拆分 → 基础组件（Button、Input、Layout）放主包
                  业务组件（Form、Table、Chart）放分包
2. Taro 运行时优化 → 按需引入 Taro 模块，不用 import * from '@tarojs/taro'
3. 图片外置 → 图标用 iconfont，大图上 CDN

教训：体积优化要一开始就做，事后拆包比事前规划痛苦得多。
```

### 坑 3：过度抽象 —— 按钮组件设计了 18 个参数

```
现象：一开始追求"覆盖所有平台的所有能力"，
      把微信 button 的 18 个属性全部透传成组件参数。
      结果：props 接口又臭又长，新人看到直接放弃。

解决：改成"最小公共接口"原则。
      三端共有的 5 个核心属性作为主接口（type、size、disabled、loading、onTap），
      其余平台特有属性合并到 platformProps 里。

教训：抽象要在有足够样本之后做。
      法则是"出现第三次重复时再抽取"，而不是"先设计完美的通用接口"。
```

## <a id="reuse"></a>六、70% 代码复用率的组成

### 复用了什么（70%）

```
可复用代码：
├── 通用层（core + shared）        → ~30%（hooks、utils、types，完全平台无关）
├── 组件业务逻辑                     → ~25%（状态管理、校验逻辑、数据处理）
└── 组件模板/结构                    → ~15%（JSX 结构、条件渲染、列表渲染）
```

### 复用不了什么（30%）

```
不可复用代码：
├── 平台适配器代码                   → ~12%（adapter/ 目录下 weapp.ts、tt.ts、h5.ts）
├── 平台样式校准                     → ~10%（CSS 差异、单位转换、安全区适配）
└── 平台入口配置                     → ~8%（app.json、project.config.json 结构不同）
```

### 如何衡量复用率

```
度量方式：
├── 代码行数法：共享代码行数 / 总代码行数 = 70%
├── 文件数量法：共享文件数量 / 总文件数量  = 65%
└── 功能覆盖法：共享功能数 / 总功能数      = 85%（功能层面复用率更高）

面试时说代码行数法最直观，功能覆盖法说明业务价值更高。
```

## <a id="reflection"></a>七、反思与改进

### 如果重新做

| 方面 | 当初的做法 | 现在的改进 |
|------|-----------|-----------|
| **文档** | 组件写完才补文档 | 第一个组件写完就搭好 Storybook，边写边文档 |
| **测试** | 几乎没有单测 | 核心组件单测覆盖率 ≥ 80%，视觉回归测试 |
| **抽象时机** | 一开始就设计通用接口 | 出现三次重复再抽取，不过度抽象 |
| **体积优化** | 事后补救 | 一开始就规划主包/分包策略 |
| **选型评估** | 团队内部讨论 | 加上 POC 验证，两周内用候选方案写同一个 demo 对比 |

### 还能用什么方案？

**方案一：原生开发 + 设计系统**

放弃跨端框架，每个端用原生开发，通过统一的设计规范和 Design Token 保证视觉一致性。优势是性能最好，劣势是代码无法复用、维护成本翻四倍。适合"两端以内、团队够大"的场景。

**方案二：Taro 4.x / Harmony 版本**

Taro 后续版本对鸿蒙的支持、编译性能都有提升。持续关注跨端框架的演进。

**方案三：Web Components 标准**

理论上 Web Components 是真正的"跨框架"方案。但在小程序环境里不支持，暂时不可行。长期来看，如果小程序开放更多 Web 标准支持，这是一个值得关注的方向。

## <a id="taro-principle"></a>八、TaroJS 跨平台原理（速查）

### 编译时做了什么？

**线 1：React 代码本身的编译（Babel）**——跟普通 React 项目一样。

**线 2：Taro 做的额外工作——名字翻译 + 模板生成**

```
① 标签名翻译：  <View> → <view>、<Text> → <text>
② 属性名翻译：  className → class
③ 生成小程序模板文件（WXML）
④ 生成小程序配置文件（app.json / page.json）
```

### 运行时做了什么？

React 需要 `document.createElement`、`appendChild` 等浏览器 API，小程序不提供这些。Taro 运行时模拟了它们：

```
React 调用                              Taro 提供
─────────                              ────────
document.createElement('view')    →    new TaroElement('view')
element.appendChild(child)       →    TaroElement.appendChild()
element.addEventListener(...)    →    注册到 Taro 事件系统
```

模拟完之后通过 `setData()` 把变更传到渲染层。

### 适配层

适配层 = 统一接口 + 平台分支 + 编译时 Tree Shaking：

```ts
function request(options) {
  if (process.env.TARO_ENV === 'weapp') return wx.request(options);
  if (process.env.TARO_ENV === 'douyin') return tt.request(options);
  if (process.env.TARO_ENV === 'h5') return fetch(options.url, options);
}
```

编译微信小程序时，`process.env.TARO_ENV` 被替换为 `'weapp'`，其他分支被 Tree Shaking 删除。

### Taro 方案的局限性

1. **setData 通信瓶颈不可消除**：逻辑层和渲染层分离是物理架构
2. **运行时体积**：模拟 DOM/BOM 的代码有几十 KB
3. **复杂交互不如原生**：高频手势、实时动画多了一层中转
4. **调试链路长**：源码 → 编译 → 小程序运行，需要 source map
5. **平台差异无法 100% 抹平**：总有需要写平台判断的时候

## 核心原则（一句话记住）

```
组件库的价值不只是"代码复用"，
而是让业务方不用关心平台差异。
技术选型要结构化评估，抽象要在有足够样本之后做。
```
