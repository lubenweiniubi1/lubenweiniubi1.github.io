# AI 应用工作流 —— 面试高频问题与答案

> 基于 REA Group 项目真实经验整理，聚焦 AI 工具、MCP、Skill 封装等方向。

---

## Q1: 你在项目中是怎么用 AI 工具的？只是代码补全吗？

**回答：**

不只是代码补全。在 REA Group 项目中，我参考 **superpowers 工作流**的设计理念，把 AI 融入到了**完整交付周期**，每个阶段之间有明确的闸门（Hard Gate），不过闸门不进入下一阶段：

```
Spec（需求澄清） → Plan（实现计划） → Implement（编码） → Verify（验证） → Issue（结构化记录） → Review → Finish（PR + 合并）
```

**核心理念**：每个阶段有明确的**进入条件**和**输出物**，AI 产出 + 人审核通过，才能流转到下一阶段。和 superpowers 的 brainstorming → writing-plans → executing-plans → verification → review → finish 一脉相承。

具体来说：

1. **Spec 阶段（对应 brainstorming）**
   - AI 把产品需求拆成结构化 Spec 文档（背景/目标/验收标准/边界条件/技术方案）
   - **闸门**：Spec 必须我 review 通过，才进入 Plan 阶段。绝不跳过设计直接写代码——再简单的需求也走这个流程

2. **Plan 阶段（对应 writing-plans）**
   - AI 基于 Spec 拆解出可执行的任务清单，每个任务有明确的文件范围、实现步骤、验证方式
   - 计划写得像一个"零上下文的新人也能执行"的程度
   - **闸门**：Plan 审核通过才进入编码

3. **Implement 阶段（对应 subagent-driven-development）**
   - 每个任务用一个**独立的 AI Agent**执行——干净的上下文，不继承之前的会话历史。这保证了每个任务专注、不受之前对话的干扰
   - AI 按 Plan 中的任务逐条实现，每完成一个任务立即进入验证

4. **Verify 阶段（对应 verification-before-completion）**
   - 核心铁律：**没有验证证据，不说"完成了"**。AI 生成单测 → 立即执行 → 拿到绿色结果才算通过
   - 边界场景和 happy path 都必须覆盖，不是"看起来没问题"

5. **Issue 阶段**
   - AI 自动生成结构化 Git Issue，固定格式记录：**来源卡**（这个 Feature 对应哪个需求卡片）、**实现方法**（用了什么技术方案）、**测试方式**（怎么验证的、覆盖了哪些场景）、**变更清单**（改了哪些文件）
   - 目的：每个 Feature 都有可追溯的完整档案，后续排查问题或复盘时不需要翻代码和聊天记录
   - **闸门**：Issue 内容我确认无误后发布

6. **Review 阶段（对应 requesting-code-review）**
   - AI 先做一轮预检（代码规范、潜在 bug、性能问题），再提交给同事
   - 核心理念：**高频 Review，不等最后**。每完成一个任务就 Review，而不是积压到 PR

7. **Finish 阶段**
   - AI 自动生成 PR 描述（常规信息 + Issue Link），确保 PR ↔ Issue 双向可追溯
   - AI 自动生成符合 Conventional Commits 规范的 commit message，通过 CI/CD 发布

**关键转变**：AI 不只是"写得更快"，而是改变了我的工作方式——我把精力集中在**闸门决策**上（这个 Spec 对不对？这个 Issue 的信息完整吗？这个验证充不充分？），把重复执行和格式化文档交给 AI。

**可以追问的点**：
- 为什么每个阶段要有闸门？→ 防止 AI 一路狂奔产出大量需要返工的代码。越早发现问题，修复成本越低
- 为什么每个任务要独立 Agent？→ 避免上下文污染。AI 在长对话中容易"跑偏"，独立 Agent 每次从干净状态开始，产出质量更稳定
- Spec 怎么保证质量？→ 有固定模板，AI 按模板填充，我 review 重点看验收标准和边界条件是否完整
- Issue 为什么要结构化？→ 固定格式保证每个 Feature 都有完整档案——来源、方案、测试、变更四个维度覆盖。半年后回头看也能快速定位上下文
- PR 和 Issue 怎么关联？→ PR 描述自动带上 Issue Link，合并时自动关闭 Issue，保证双向可追溯

## Q2: 你有没有自己封装过 Skills？怎么设计的？

**回答：**

有的。在 REA 项目中，我基于 Claude Code 封装了 4 个自定义 Agent Skill，每个 Skill 对应工作流中的一个闸门阶段，确保 AI 产出质量和流程纪律：

| Skill | 对应阶段 | 解决的问题 | 核心设计 |
|-------|---------|-----------|---------|
| **Spec Skill** | brainstorming | 需求理解不一致、遗漏边界条件 | 内置 Spec 模板（背景/目标/验收标准/边界条件/技术方案），AI 按模板输出结构化文档。**闸门：Spec 不经人 review 通过，不进 Plan** |
| **Unit Test Skill** | verification | 单测覆盖率不足、遗漏边界场景 | 内置 Jest 测试规范（AAA 模式、Mock 策略、边界枚举），AI 自动扫描源码生成测试。**闸门：测试不绿，不进入 Issue** |
| **Issue Skill** | issue | Feature 信息散落、排查时无法回溯上下文 | 结构化 Issue 模板（来源卡/实现方法/测试方式/变更清单），每个 Feature 一份完整档案。线上问题时额外包含根因分析及修复文档。**闸门：Issue 内容不完整，不进 Review** |
| **Code Review Skill** | review | PR Review 耗时长、人工遗漏 | 内置团队的 ESLint 规则 + 常见反模式清单，AI 逐条检查并标注风险等级。**闸门：预检不过，不提交人工 Review** |

Code Review Skill 内置的**常见反模式清单**（举例）：

| 类别 | 反模式 | 为什么是问题 |
|------|--------|-------------|
| **React** | useEffect 缺少依赖数组 | 可能导致闭包陷阱或无限循环 |
| **React** | JSX 中写内联对象/箭头函数 | 每次 render 创建新引用，导致子组件不必要的重渲染 |
| **React** | 列表渲染缺少 key / key 用 index | Diff 算法失效，列表重排时出现渲染错误 |
| **TypeScript** | 滥用 `any` 类型 | 放弃类型检查，埋下运行时隐患 |
| **TypeScript** | `as` 强制类型断言 | 绕过编译器，类型声明与实际值不一致 |
| **逻辑** | 简单 if-else 写成策略模式 | 过度设计，增加理解成本 |
| **逻辑** | 直接修改 state 而非 setState | React 不会触发重渲染，UI 与数据不同步 |
| **工程** | 一个组件超过 300 行 / 函数超过 50 行 | 职责不清，难以测试和维护 |
| **工程** | 硬编码魔法数字和字符串 | 后续修改时容易遗漏，应该提取为常量 |
| **安全** | 用户输入未做 XSS 处理 | dangerouslySetInnerHTML 直接渲染用户输入 |

**设计原则**：
1. **每个 Skill 是一个闸门（Hard Gate）**：有明确的输入/输出/通过条件，不过闸门不进下一阶段。这防止 AI 一路狂奔产出大量需要返工的代码
2. **内置模板 = 强制标准化**：每个 Skill 内置固定输出模板，保证 AI 产出结构一致、可预期，团队任何人用同一个 Skill 产出质量一致
3. **人始终在闸门处把关**：AI 负责"执行"，人负责"判断"——在闸门处决定是否放行。这和 superpowers 的核心理念一致：AI 辅助，人做决策
4. **每个 Skill 可独立测试和迭代**：拿历史真实案例回放，对比人工产出和 AI 产出，持续优化 prompt

**可以追问的点**：
- 你怎么测试这些 Skill？→ 拿真实历史需求回放，对比人工产出和 AI 产出，迭代 prompt；新 Skill 上线前先在个人项目里跑 5-10 个真实 case
- Skill 和普通 prompt 有什么区别？→ Skill 是工程化的 prompt——有固定输入输出格式、有闸门条件、可版本管理、可团队共享、可组合成流水线。Prompt 是"一句话"，Skill 是"一个流程"

## Q3: 你了解 MCP 吗？在你的工作流中怎么用的？

**回答：**

了解。MCP（Model Context Protocol）本质上是给 AI 配了一套**标准化的"手和眼睛"**——让 AI 不只是聊天，而是能实际操作开发环境。打个比方：没有 MCP 时，AI 是个只能说话的顾问；有了 MCP，AI 是个能直接干活的搭档。

具体到开发场景，举几个真实例子：

**场景一：排查一个组件报错**

- **没有 MCP**：我把报错信息贴给 AI → AI 猜可能是某行的问题 → 我打开文件找到那行贴回去 → AI 再给修复建议 → 我手动改
- **有了 MCP**：我说"首页列表报错了"→ AI 自己读报错文件 → 追踪引用链找到根因 → 直接在文件里改 → 我审查确认。**我全程没有复制粘贴一行代码**

**场景二：重构一个 API 的调用方式**

- **没有 MCP**：我用 grep 手动搜所有调用点 → 一个个改 → 容易漏
- **有了 MCP**：AI 直接搜全局引用（MCP 的 Grep 工具）→ 列出所有调用点 → 逐文件批量修改 → 跑相关测试确认没断。**不会漏，因为 AI 搜得比人全**

**场景三：本地 debug 一个容器里跑的服务**

- **没有 MCP**：我开终端 `docker logs xxx` → 复制日志给 AI → AI 分析 → 我回去改代码 → 再 `docker restart` → 再看日志...来回切 5-6 次
- **有了 MCP**：AI 通过 MCP 直接查容器日志 → 定位问题 → 改代码 → 重启容器 → 等启动完成再查日志验证修复。**我在旁边看着就行**

**场景四：完整的端到端链路（最体现 MCP 价值）**

比如我要新增一个 API 端点：

```
AI 读项目结构(MCP) → 找到类似 API 参考实现(MCP) → 生成代码
→ 读路由配置追加注册(MCP) → 跑集成测试(MCP) → 测试绿了
→ 生成 commit message → 我确认提交
```

整个过程 AI 操作了：文件读取、代码搜索、文件写入、命令执行、Docker 日志。如果没有 MCP，每一步都需要我手动搭桥。

**怎么和 Skill 配合？**

我封装的 Skill 本质上是一套 prompt 模板 + MCP 工具编排。比如 Unit Test Skill 的执行逻辑：

1. 读源码文件（MCP: Read）
2. 分析依赖关系，找到要 mock 的模块（MCP: Grep）
3. 按 AAA 模式生成测试代码
4. 写入测试文件（MCP: Write）
5. 跑 `npx jest`（MCP: Bash）
6. 读输出确认全绿（MCP: Bash 输出）

6 步里 5 步靠 MCP 完成，我只在最后确认"全绿了，过"。

**核心认知**：MCP 让 AI 从"给我建议"变成"替我干活，我来审查"。这个转变才是提效的本质——人的角色从执行者变成决策者。

**可以追问的点**：
- 安全怎么考虑？→ MCP 是客户端侧协议，权限在本地控制；比如我的配置里 git push 和数据库写入都需要显式确认。MCP 只是给 AI 开了一扇门，钥匙在我手上
- 和 Function Calling 的区别？→ Function Calling 是各家模型自己的调用机制，换个模型就得重写。MCP 是标准协议，一次定义、跨模型/跨工具通用。类似 USB-C 和各家私有充电口的区别
- 你自己写过 MCP Server 吗？→ 目前主要是用社区和平台提供的 MCP Server（文件系统、Docker、Git），然后通过 Skill 编排它们。但理解 MCP 协议规范，如果需要对接自定义内部系统（如内部 CI/CD 平台），可以基于 SDK 快速搭建

## Q4: 你这条 AI 流水线实际提效了多少？有数据吗？

**回答：**

坦白说，REA 项目中没有做严格的 A/B 对照量化。但我可以从几个维度描述实际感受：

**定性效果**：

| 环节 | 之前 | 之后 |
|------|------|------|
| 需求拆解 | 半天到一天手写 Spec | AI 初稿 10 分钟 + 我审查 20 分钟 |
| 单测编写 | 一个组件 1-2 小时 | AI 生成 2 分钟 + 我调整覆盖 15 分钟 |
| Code Review | 等同事 Review，半天到一天 | AI 预检 1 分钟 + 同事聚焦看逻辑 |
| Issue 复盘 | 忙起来就忘了写 | AI 自动生成，5 分钟 Review 归档 |

**更重要的变化**：
- **上下文切换减少**：不用在"写代码"和"搜文档/搜 StackOverflow"之间频繁切换，AI 在 IDE 里一站式解决
- **Review 质量提升**：以前 Review 花大量精力在格式、命名、漏测上，现在 AI 把这些问题提前拦住了，人的精力可以集中在架构和业务逻辑上
- **经验沉淀**：以前线上问题处理完就过去了，现在 Issue Skill 强制生成复盘文档，团队的知识库在持续积累

**如果面试官追问具体数字**：可以说"估算单测环节节省约 70% 时间，Review 环节节省约 50% 时间，但整体交付周期受限于需求沟通、联调等非编码环节，端到端大概提效 30-40%。"

## Q5: AI 写的代码质量怎么样？你敢直接上生产吗？

**回答：**

**不敢，也不会。** AI 生成的代码永远需要人审查。

我的做法是三道防线：

**第一道：AI 自检（Skill 内置约束）**
- Skill 的 prompt 里内置了团队规范（命名、结构、ESLint 规则），AI 输出时就已经在遵循规范
- Code Review Skill 会在提交前做一轮自动检查

**第二道：自动化测试**
- Unit Test Skill 生成的测试会立即执行，AI 写的代码必须通过自己生成的测试
- CI 流水线中还有集成测试和 E2E 测试

**第三道：人工审查**
- 我个人会逐段 Review AI 改动的代码，重点关注：业务逻辑是否正确、边界条件是否覆盖、是否有过度设计
- PR 阶段同事会做第二轮 Review

**实际遇到过的 AI 问题**：
- AI 有时候会过度设计，一个简单的 if-else 写成策略模式
- AI 生成的 Mock 数据不够真实，需要我补充
- 对老旧代码的改动，AI 不了解历史上下文，容易引入兼容性问题

**结论**：AI 能写出 80 分的代码，但剩下 20 分需要人来补。把 AI 当高级实习生用——干活快，但需要有经验的人把关。

## Q6: 你觉得 AI 辅助开发的核心能力是什么？不是会用工具就行吗？

**回答：**

会用工具只是入门。我认为 AI 辅助开发的核心能力分三层：

**第一层：工具使用（必要但不充分）**
- 会用 Claude Code / Cursor / Copilot
- 会写 prompt
- 大概 1-2 周就能上手

**第二层：工作流设计（拉开差距的地方）**
- 能把 AI 嵌入到完整的开发流程中，而不是只在写代码时用
- 懂得什么时候用 AI、什么时候靠自己——不是所有问题都适合丢给 AI
- 能把重复性的流程封装成 Skill，让 AI 可复用、可标准化

**第三层：判断力与审查能力（核心壁垒）**
- 能快速判断 AI 输出的质量：这段代码能直接用吗？需要怎么改？
- 能发现 AI 的盲区：逻辑漏洞、过度设计、性能隐患
- 能理解业务上下文，这个 AI 做不到

**一句话总结**：会用 AI 工具的门槛很低，但能把 AI 融入工程化流程、保障输出质量、沉淀团队经验的能力，才是真正值钱的地方。初级开发用 AI 写得更快，高级开发用 AI 让整个团队的交付质量更高。

## Q7: 如果让你从零搭建一套 AI 辅助开发体系，你会怎么做？

**回答：**

我会参考 superpowers 的设计理念，分四步走：

**第一步：建立闸门意识（第 1-2 周）**
- 选一个 AI 编码工具（Claude Code / Cursor），在个人开发中深度使用
- 最关键的不是学工具，而是建立"闸门思维"：每次让 AI 写代码前，先问自己——Spec 清楚了吗？Plan 合理吗？不做完设计不让 AI 写代码
- 目标是养成"先想清楚再动手"的习惯，而不是"让 AI 先写一版看看"

**第二步：流程嵌入 + 闸门搭建（第 3-4 周）**
- 把交付流程拆成明确阶段：Spec → Plan → Implement → Verify → Review → Finish
- 每个阶段定义三个东西：**输入物**（进入条件）、**输出物**（AI 产出）、**通过条件**（闸门标准）
- 示例：Implement 阶段的通过条件是"所有新增代码的测试全绿 + 无 ESLint 报错"
- 每个节点明确：AI 做什么（执行），人做什么（闸门判断）

**第三步：Skill 封装 + 独立 Agent（第 2-3 个月）**
- 识别团队高频重复的工作模式，封装成 Skill
- 每个 Skill 对应一个闸门阶段，内置模板 + 通过条件
- 关键设计：每个任务用独立 Agent 执行，干净上下文，避免 AI 在长对话中"跑偏"
- 在团队内推广，收集反馈迭代

**第四步：度量与优化（持续）**
- 建立简单度量：每个闸门的通过率、返工率、交付周期
- 定期回顾：哪个闸门总被跳过？哪个 Skill 产出质量不稳定？
- 持续优化 ROI，砍掉没用的，打磨有用的

**关键原则**：
- **闸门是核心纪律**：再简单的需求也走 Spec → Plan → Implement 流程，不跳过
- **独立 Agent 提质量**：每个任务用新鲜上下文，不继承之前对话的"噪音"
- **人始终在闸门处把关**：AI 负责执行，人负责判断——而这个判断力才是不可替代的

## Q8: Skill 封装和直接写 prompt 有什么区别？为什么要封装？

**回答：**

**区别很大。** 直接写 prompt 是"每次重来"，Skill 封装是"一次打磨，持续复用"。

| 维度 | 直接写 Prompt | Skill 封装 |
|------|-------------|-----------|
| **一致性** | 每次输出格式不统一 | 内置模板，输出结构可预期 |
| **复用性** | 只有你自己知道怎么用 | 团队成员可以直接用，"零上下文新人"也能执行 |
| **可迭代** | 改了也不知道效果 | 版本管理，可 A/B 对比 |
| **知识沉淀** | 存在你脑子里 | 存在代码仓库里 |
| **组合性** | 每次手动串联 | 多个 Skill 串联成流水线，每个 Skill 是流水线的一个闸门节点 |

**举个实际例子**：

直接写 prompt：`"帮我给这个组件写测试"`——AI 每次产出的格式、覆盖深度都不一样。

封装成 Unit Test Skill：Skill 内置了 AAA 模式说明、Mock 策略、边界枚举清单、覆盖率要求，AI 每次输出都是同一个标准。团队成员用这个 Skill 产出的测试质量是一致的。

**本质上**：Prompt 是个人技巧，Skill 是工程能力。前者靠个人经验，后者可以团队传承和持续改进。

## 面试应对策略总结

| 面试官可能的关注点 | 你的核心信息 |
|-------------------|-------------|
| "你只是会用工具吧？" | 强调**闸门工作流设计**和 Skill 封装，而不只是"我用 AI 写代码"。核心是：设计了 Spec → Plan → Implement → Verify → Review → Finish 六阶段流水线，每个阶段有进入/通过条件 |
| "AI 代码能上生产吗？" | 三道防线 + 闸门机制：Spec 闸门（需求审对了没）→ Verify 闸门（测试全绿了没）→ Review 闸门（代码审过了没）。不过闸门不进下一阶段 |
| "提效数据呢？" | 坦诚说明没有严格 A/B，给定性描述 + 保守估算。重点不是"快了多少"，而是**返工率下降**和**上下文切换减少** |
| "你们团队别人也用吗？" | 强调 Skill 的零上下文设计——任何人拿过来都能用，产出质量一致，展示你的影响力 |
| "AI 会不会取代你？" | 闸门决策不可替代：AI 负责流水线里的执行，人负责每个闸门处的判断——这个 Spec 对吗？这个验证够吗？这个架构决策合理吗？ |
| "你这条流水线怎么保证质量？" | 两个机制：① Hard Gate——每阶段有明确通过条件，不过不往下走；② 独立 Agent——每任务干净上下文，避免长对话污染 |

**一个金句收尾**：AI 让我从"写代码的人"变成了"做决策的人"——花更多时间在闸门处判断"对不对、够不够、好不好"，把"怎么做"交给 AI。

**另一个角度**：这条流水线的本质是**把软件工程的纪律用 AI 强制执行**——Spec 先行、Plan 驱动、独立 Agent 实现、验证才说完成、高频 Review。这些纪律人类团队知道该做但经常因为赶进度而跳过，AI 反而帮你把它们固化了。
