## Q1: 如何实现动态 Remote？（不写死在 webpack.config.js 中）

**回答：**

生产环境中，Remote 的 URL 往往是动态的（不同环境不同地址、AB 测试等）。

```js
// ===== 方案1：webpack 5 的 Promise-Based Remotes =====
// webpack.config.js
plugins: [
  new ModuleFederationPlugin({
    name: 'host',
    remotes: {
      // ⭐ 写成函数形式，运行时才决定 URL
      app_b: `promise new Promise(resolve => {
        // 从配置接口获取 Remote 地址
        const url = window.__REMOTE_CONFIG__?.app_b
          || 'http://localhost:3002/remoteEntry.js';

        // 动态加载 script
        const script = document.createElement('script');
        script.src = url;
        script.onload = () => {
          resolve({
            get: (request) => window.app_b.get(request),
            init: (arg) => {
              try { return window.app_b.init(arg); }
              catch(e) { console.log('remote already initialized'); }
            },
          });
        };
        document.head.appendChild(script);
      })`,
    },
  }),
],

// 在 HTML 中注入环境配置
// <script>
//   window.__REMOTE_CONFIG__ = {
//     app_b: 'http://cdn.example.com/app_b/v1.2.3/remoteEntry.js',
//   };
// </script>
```

```js
// ===== 方案2：使用 @module-federation/runtime（推荐） =====
// npm install @module-federation/runtime

import { init, loadRemote } from '@module-federation/runtime';

// 运行时初始化
init({
  name: 'host',
  remotes: [
    {
      name: 'app_b',
      entry: 'http://localhost:3002/remoteEntry.js',
    },
  ],
});

// 动态加载
const Button = await loadRemote('app_b/Button');
```

## Q2: 多版本共存场景怎么处理？

**回答：**

#### 场景：你需要同时使用 Remote A（依赖 React 18）和 Remote B（依赖 React 17）

```js
// ===== 方案：使用不同的 shareScope =====

// Host — webpack.config.js
plugins: [
  new ModuleFederationPlugin({
    name: 'host',
    shared: {
      react: {
        singleton: true,
        requiredVersion: '^18.0.0',
        shareScope: 'default',       // 默认 scope
      },
    },
    remotes: {
      // Remote A（React 18）→ 用默认 scope
      app_modern: 'app_modern@http://localhost:3001/remoteEntry.js',

      // Remote B（React 17）→ 用独立 scope
      app_legacy: `promise new Promise(resolve => {
        // ... 加载 script
        // 注意：Remote B 的 webpack 配置中 shareScope 需要匹配
        // shared: { react: { shareScope: 'legacy-scope' } }
      })`,
    },
  }),
],
```

**最佳实践：** 尽量避免多版本共存，增加维护负担。如果必须，优先考虑把遗留系统慢慢升级。

## Q3: Module Federation + Nginx 部署是怎么做的？

**回答：**

```nginx
# nginx.conf — Host 应用
server {
    listen 80;
    server_name app.example.com;

    # Host 的静态资源
    location / {
        root /var/www/host/dist;
        try_files $uri $uri/ /index.html;
    }

    # 代理 Remote A 的静态资源
    location /remote-a/ {
        # ⭐ 关键：remoteEntry.js 和 chunk 文件都需要可访问
        proxy_pass http://remote-a-service:3001/;
        add_header Access-Control-Allow-Origin *;
    }

    # 代理 Remote B 的静态资源
    location /remote-b/ {
        proxy_pass http://remote-b-service:3002/;
        add_header Access-Control-Allow-Origin *;
    }
}
```

**对应的 webpack 配置：**

```js
// Host 的 remotes 指向 Nginx 代理后的地址
remotes: {
  app_a: 'app_a@/remote-a/remoteEntry.js',
  app_b: 'app_b@/remote-b/remoteEntry.js',
}
```

**部署要点：**
1. 所有 Remote 的资源必须和 Host 在同一域名下，或用 CORS 允许跨域
2. `remoteEntry.js` 和所有 chunk 文件都要通过 Nginx 可访问
3. Remote 的 `publicPath` 要配成 Nginx 代理后的路径
4. Remote 更新后，Host 无需重新构建 — 这是 MF 最核心的优势

## Q4: 如何实现微前端的渐进式升级？（老项目一步步迁移）

**回答：**

Module Federation 非常适合渐进式迁移：

```
阶段1：现有单体应用（巨石应用）
  ┌────────────────────────┐
  │  /dashboard  /users    │
  │  /settings   /reports  │
  │  /billing    /admin    │
  └────────────────────────┘

阶段2：抽取一个模块 → 独立部署
  ┌──────────┐  ┌──────────┐
  │ Monolith │  │ Remote   │
  │ /dashboard│  │ /users  │ ← 新项目，独立部署
  │ /settings │  └──────────┘
  │ /reports  │
  │ /billing  │
  │ /admin   │
  └──────────┘

阶段3：逐步抽取，最终完全拆分
  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ Host     │  │ Remote A │  │ Remote B │  │ Remote C │
  │ /        │  │ /users  │  │ /reports │  │ /admin  │
  └──────────┘  └──────────┘  └──────────┘  └──────────┘
```

**实施步骤：**

```js
// 步骤1：在原单体应用中加上 ModuleFederationPlugin（暴露模块）
new ModuleFederationPlugin({
  name: 'legacy',
  exposes: {
    './UserModule': './src/pages/users',
  },
  shared: { react: { singleton: true } },
}),

// 步骤2：创建新项目，消费原项目的模块
new ModuleFederationPlugin({
  name: 'new_users',
  remotes: {
    legacy: 'legacy@http://legacy.example.com/remoteEntry.js', // 过渡期
  },
}),

// 步骤3：新项目重写模块后，反过来暴露给原项目
new ModuleFederationPlugin({
  name: 'new_users',
  exposes: {
    './UserModule': './src/pages/users', // 新版
  },
}),

// 步骤4：原项目切换到消费新版模块
// 步骤5：原项目删除旧模块代码
// 步骤6：最终下线 legacy 服务
```

## Q5: Webpack Native Federation 是什么？和 Module Federation 什么关系？

**回答：**

Native Federation 是 **Angular 团队提出的一种更轻量的实现**，不依赖 webpack。

| | Module Federation（Webpack） | Native Federation |
|---|---------------------------|-------------------|
| 依赖 | Webpack 5+ | **无构建工具依赖** |
| 实现 | Webpack runtime + shareScope | 纯浏览器 ESM import maps |
| 共享依赖 | webpack 的 shared 协商 | import maps 管理 |
| 浏览器支持 | 所有现代浏览器 | 需要支持 import maps |
| 心智模型 | webpack 魔法 | 更接近浏览器原生行为 |

```html
<!-- Native Federation 使用 import maps -->
<script type="importmap">
{
  "imports": {
    "app_b/Button": "http://localhost:3002/Button.js",
    "react": "https://cdn.com/react@18.umd.js"
  }
}
</script>

<script type="module">
  // 直接用 import，浏览器原生解析
  import Button from 'app_b/Button';
</script>
```

**现状：** 目前 Native Federation 还不够成熟，Module Federation（Webpack 版）仍然是生产环境的首选。

## Q6: Module Federation 性能怎么优化？

**回答：**

```js
// 1. 合理配置 shared，减少重复加载
shared: {
  react: { singleton: true, requiredVersion: '^18.0.0' },  // 全局一份
  'react-dom': { singleton: true },
  // 大型工具库也可以共享
  'antd': { singleton: true, requiredVersion: '^5.0.0' },
}

// 2. Remote 做 code splitting
//    exposes 只暴露入口组件，内部逻辑按路由/功能拆分 chunk

// 3. 设置合理的 publicPath
//    避免 MF 加载资源时路径错误导致多次请求
output: {
  publicPath: 'https://cdn.example.com/app/',
}

// 4. 预加载策略
//    在关键时机提前加载 remoteEntry
const preloadRemote = (url) => {
  const link = document.createElement('link');
  link.rel = 'preload';
  link.as = 'script';
  link.href = url;
  document.head.appendChild(link);
};

// 监听可能的用户操作（如 hover 菜单）
menuItem.addEventListener('mouseenter', () => {
  preloadRemote('http://remote.example.com/remoteEntry.js');
});

// 5. 使用 HTTP/2 server push 推送 remoteEntry
// 6. CDN 边缘缓存 remoteEntry.js 和 chunk
```

## ⚡ 与 qiankun 高级场景对比

| 场景 | qiankun | Module Federation |
|------|---------|-------------------|
| **动态注册** | `registerMicroApps` 支持动态增删 | Promise-based remotes / runtime SDK |
| **渐进式迁移** | 新功能做子应用，老代码不动 | 逐步 expose 模块，切换引用方向 |
| **部署** | 必须 Nginx 反向代理统一域名 | 同域名最佳，CORS 也可跨域 |
| **版本共存** | ❌ 不支持（同一框架单例） | shareScope 分组可实现多版本共存 |
| **构建工具** | 不限（fetch HTML 方式） | 主流 Webpack 5+，Rspack 也支持 |
| **运行时更新** | 子应用更新后主应用刷新即生效 | Remote 部署后 Host 自动拉最新（无需刷新） |
