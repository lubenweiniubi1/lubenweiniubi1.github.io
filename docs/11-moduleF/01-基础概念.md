# Module Federation 基础概念

> 📘 **难度：⭐ 入门** | 适合第一次接触微前端的同学

---

## Q1: 什么是 Module Federation？一句话说清楚。

**回答：**

Module Federation（模块联邦）是 **Webpack 5 内置的微前端方案**，它允许多个独立构建的应用在**运行时共享模块**，一个应用可以像 import 自己的代码一样，import 另一个应用的组件。

```js
// 应用 A 直接使用应用 B 的 Button 组件
// 这行代码在编译时不会报错，运行时才去加载
const Button = React.lazy(() => import('app_b/Button'));

function AppA() {
  return (
    <React.Suspense fallback={<div>Loading...</div>}>
      <Button>来自远程应用的按钮</Button>
    </React.Suspense>
  );
}
```

> 🔑 **一句话**：Module Federation = 让 webpack 能 import 其他应用暴露的模块，就像 import 本地文件一样。

---

## Q2: 为什么需要 Module Federation？它解决什么问题？

**回答：**

| 传统开发的问题 | Module Federation 的解法 |
|---------------|------------------------|
| 多个项目共享组件只能 `npm publish`，每次改都要发包 | 直接运行时共享，改完即生效 |
| A 项目改了组件，B 项目要重新 `npm install` + 构建 | 消费方自动获取最新版本 |
| 大项目改一处，整个应用重新 CI/CD | 各模块独立构建，独立部署 |
| 不同团队想共用代码但不想共享 repo | 运行时联邦，代码无需放在同一个仓库 |

**一句话：让你在微前端架构中，既享受"独立构建部署"的灵活，又不牺牲"import 本地代码"的开发体验。**

## Q3: Module Federation 的核心概念有哪些？（必懂）

**回答：**

只有三个核心概念，记住这个比喻：

> 一个模块就是**一栋房子**🏠，导出是**门**（exposes），远程引用是**窗**（remotes），共享依赖是**水管**（shared）。

| 概念 | 关键词 | 比喻 | 代码位置 |
|------|--------|------|---------|
| **Host**（宿主） | 消费方 | 客人，参观其他人的房子 | `remotes` 配置 |
| **Remote**（远程） | 提供方 | 主人，开放自己的房间 | `exposes` 配置 |
| **Shared**（共享） | 公共依赖 | 公共水管，大家用同一套 | `shared` 配置 |

```js
// Remote（提供方）— webpack.config.js
module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'app_b',         // ⭐ 模块名字，全局唯一
      filename: 'remoteEntry.js',  // 对外暴露的入口文件
      exposes: {             // ⭐ 暴露哪些模块给外部使用
        './Button': './src/components/Button',
        './utils': './src/utils',
      },
    }),
  ],
};

// Host（消费方）— webpack.config.js
module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'app_a',
      remotes: {             // ⭐ 声明要消费哪些远程模块
        app_b: 'app_b@http://localhost:8002/remoteEntry.js',
      },
    }),
  ],
};
```

## Q4: 消费方（Host）和提供方（Remote）的关系是怎样的？

**回答：**

```
┌─────────────────────────┐     ┌─────────────────────────┐
│       App A (Host)      │     │       App B (Remote)     │
│                         │     │                         │
│   webpack.config.js:    │     │   webpack.config.js:    │
│   remotes: {            │     │   exposes: {            │
│     app_b: "app_b@..."  │ ──► │     './Button': '...',  │
│   }                     │     │   }                     │
│                         │     │                         │
│   // 使用方                                暴露方       │
│   import('app_b/Button')│     │                         │
└─────────────────────────┘     └─────────────────────────┘
```

**关键规则：**
- 一个应用可以**同时是 Host 和 Remote**（既可以消费别人的模块，也可以暴露自己的）
- Host 不绑定特定 Remote，只需要知道 Remote 的 URL
- Host 和 Remote 之间没有"主从"关系，大家都是平等的

## Q5: 能不能用最简单的例子，看下完整流程？

**回答：**

#### 最小例子 — 两个应用，一个暴露 `Button`，一个使用它

```
my-app/
├── app_provider/          ← Remote（提供方）
│   ├── src/
│   │   └── Button.jsx
│   ├── public/index.html
│   └── webpack.config.js
│
└── app_consumer/          ← Host（消费方）
    ├── src/
    │   └── App.jsx
    ├── public/index.html
    └── webpack.config.js
```

**Step 1: Provider 配置**
```js
// app_provider/webpack.config.js
const { ModuleFederationPlugin } = require('webpack').container;

module.exports = {
  // ... 其他配置
  devServer: { port: 3001 },
  plugins: [
    new ModuleFederationPlugin({
      name: 'app_provider',
      filename: 'remoteEntry.js',
      exposes: {
        './Button': './src/Button',
      },
      shared: { react: { singleton: true }, 'react-dom': { singleton: true } },
    }),
  ],
};
```

```jsx
// app_provider/src/Button.jsx — 暴露出去的组件
export default function Button({ children, onClick }) {
  return <button onClick={onClick} style={{ padding: '8px 16px' }}>{children}</button>;
}
```

**Step 2: Consumer 配置**
```js
// app_consumer/webpack.config.js
plugins: [
  new ModuleFederationPlugin({
    name: 'app_consumer',
    remotes: {
      app_provider: 'app_provider@http://localhost:3001/remoteEntry.js',
    },
    shared: { react: { singleton: true }, 'react-dom': { singleton: true } },
  }),
],
```

**Step 3: Consumer 使用**
```jsx
// app_consumer/src/App.jsx
import React, { Suspense } from 'react';

// ⭐ 懒加载远程组件
const RemoteButton = React.lazy(() => import('app_provider/Button'));

export default function App() {
  return (
    <div>
      <h1>消费方应用</h1>
      <Suspense fallback={<div>加载中...</div>}>
        <RemoteButton onClick={() => alert('从远程来的按钮！')}>
          点击我
        </RemoteButton>
      </Suspense>
    </div>
  );
}
```

**Step 4: 运行**
```bash
# 终端1 — 先启动 Provider
cd app_provider && npm start   # localhost:3001

# 终端2 — 再启动 Consumer
cd app_consumer && npm start   # localhost:3002
```

打开 `localhost:3002`，点击按钮就能看到效果了 🎉

## Q6: Module Federation 和 "npm 发包共享" 的根本区别是什么？

**回答：**

| 对比 | npm 发包 | Module Federation |
|------|---------|-------------------|
| 共享时机 | 编译时（install 时下载） | **运行时**（浏览器才加载） |
| 版本更新 | A 发新版 → B 安装新版 → B 重新构建 | A 部署新版 → B 自动获取（无需重新构建） |
| 适用粒度 | 工具库、组件库 | 页面级、应用级模块 |
| 依赖管理 | npm install 解决 | shared 配置协商 |
| 独立部署 | ❌ 需要发新包协调 | ✅ 各自独立部署 |

**核心差异**：npm 发包 = 提前下载到本地，MF = 运行时去别人家门口拿。

## ⚡ 与 qiankun 快速对比（新手理解版）

| 维度 | qiankun | Module Federation |
|------|---------|-------------------|
| **类比** | 公寓管理员（统一调度） | 邻居互访（去中心化） |
| **关系** | 主-从（主应用管子应用） | 平等（任何应用可以互相引用） |
| **加载方式** | fetch HTML → 解析 JS/CSS → 沙箱执行 | webpack 运行时加载 JS chunk |
| **隔离性** | 有 JS 沙箱、CSS 隔离 | **无隔离**，都在同一作用域 |
| **路由** | 主应用统一管理 | 各自管理 |
| **技术门槛** | 中等 | 较低（上手快，但深度复杂） |

## 📌 本章总结

| 概念 | 一句话 |
|------|--------|
| Module Federation | Webpack 5 的微前端方案，运行时共享模块 |
| Host vs Remote | Host = 消费者，Remote = 提供者（可以互换） |
| remotes 配置 | 声明"我要用谁的模块" |
| exposes 配置 | 声明"我暴露什么给外部" |
| shared 配置 | 声明"公共依赖不要重复加载" |
