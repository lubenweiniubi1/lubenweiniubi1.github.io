# 前端性能优化面试题

> 对应考察维度：**专业能力 → 前端性能优化**
> 只讲性能：指标 → 网络 → 渲染 → 运行时 → 深度优化方案

## 一、常见面试题（先看题，再看知识点）

1. **性能优化你会从哪几个方向去做？怎么衡量效果？**

16. **前端性能监控怎么做？Performance API、PerformanceObserver、web-vitals 库怎么配合？**

## 二、性能优化的整体框架

回答"你做过哪些性能优化"这类开放题，先给一个**框架**，让面试官知道你有体系；再挑 2-3 个深挖。

```
┌── 指标层（Metrics）─── LCP / INP / CLS / TTFB / FCP，可量化
├── 网络层（Network）─── DNS / TCP / TLS / HTTP / 缓存 / CDN / 压缩
├── 构建层（Build）───── 代码分割 / Tree Shaking / 压缩 / 图片格式
├── 渲染层（Render）─── 关键 CSS / SSR / 骨架屏 / 字体
└── 运行时层（Runtime）─ 主线程调度 / 虚拟列表 / 防抖节流 / 内存
```

**回答话术三段式**：
1. **指标**（15 秒）："我们首屏从 4s 降到 1.5s，LCP 从 3.2s 降到 1.1s"
2. **方案**（2 分钟）：讲 2-3 个最有深度的优化（如虚拟列表 + 分步加载 + 接口缓存）
3. **效果**（15 秒）：用数据收尾，展示优化前后的对比

## 三、性能指标（Metrics）—— 优化的前提

优化必须有量化目标，否则就是"感觉快了"。

### 3.1 Core Web Vitals（谷歌三大核心指标）

| 指标 | 全称 | 衡量什么 | 好 / 差阈值 |
|------|------|---------|------------|
| **LCP** | Largest Contentful Paint | 最大内容元素渲染时间（加载性能） | ≤ 2.5s / > 4s |
| **INP** | Interaction to Next Paint | 交互到下一次绘制的延迟（响应性能，2024 年替代 FID） | ≤ 200ms / > 500ms |
| **CLS** | Cumulative Layout Shift | 累计布局偏移（视觉稳定性） | ≤ 0.1 / > 0.25 |

### 3.2 其他常用指标

| 指标 | 全称 | 含义 |
|------|------|------|
| **TTFB** | Time to First Byte | 首字节时间，衡量服务器 + 网络 |
| **FCP** | First Contentful Paint | 首次内容绘制，第一个像素出现 |
| **FID** | First Input Delay | 首次输入延迟（已被 INP 取代，但仍常见） |
| **TTI** | Time to Interactive | 可交互时间，页面能稳定响应用户输入 |
| **TBT** | Total Blocking Time | 总阻塞时间，FCP 和 TTI 之间所有 Long Task 阻塞时长的加权和 |

### 3.3 怎么衡量？

- **实验室数据（Lab Data）**：Lighthouse、WebPageTest。稳定但脱离真实用户。
- **真实用户数据（RUM，Real User Monitoring）**：用 [web-vitals](https://github.com/GoogleChrome/web-vitals) 库上报。
- **Performance API**：`performance.getEntriesByType('navigation' | 'resource' | 'paint')`。

```js
import { onLCP, onINP, onCLS } from 'web-vitals'
onLCP(m => send('lcp', m.value))
onINP(m => send('inp', m.value))
onCLS(m => send('cls', m.value))
```

## 四、网络层优化

### 4.1 关键渲染路径（Critical Rendering Path）

浏览器拿到 HTML 之后：

```
HTML → 解析 DOM ────────┐
                        ├─→ Render Tree → Layout → Paint → Composite
CSS  → 解析 CSSOM ──────┘

⚠️ CSS 阻塞 CSSOM 构建 → 阻塞渲染
⚠️ 同步 <script> 阻塞 HTML 解析 → 阻塞 DOM
```

关键洞察：
- **CSS 是渲染阻塞资源**。放 `<head>` 里越早请求越好，但要避免加载"非关键 CSS"。
- **JS 默认是解析阻塞资源**。用 `async` / `defer` 或放 `<body>` 底部。
- **首屏关键 CSS 内联到 HTML** 里（Critical CSS Inline），能省一个 CSS 网络往返。

### 4.2 资源提示（Resource Hints）

| 指令 | 语义 | 场景 |
|------|------|------|
| `dns-prefetch` | 提前做 DNS 解析 | 首页会跳到某个域名（如 CDN、埋点） |
| `preconnect` | 提前建立连接（DNS + TCP + TLS） | 立刻要用的第三方域 |
| `preload` | 高优先级预加载**当前页**要用的资源 | 首屏关键字体、大图、关键 CSS |
| `prefetch` | 低优先级预加载**未来页面**可能要的资源 | 用户"很可能点"的下一个页面的 JS |
| `modulepreload` | 预加载 ES Module | Vite 生产构建里的入口 chunk |

```html
<link rel="dns-prefetch" href="//img.cdn.com">
<link rel="preconnect" href="https://img.cdn.com" crossorigin>
<link rel="preload" href="/hero.webp" as="image">
<link rel="prefetch" href="/next-page.js" as="script">
```

**⚠️ 常见坑**：`preload` 不写 `as` 会重复加载；字体的 `preload` 必须加 `crossorigin`（即使同源）。

### 4.3 script 的 `async` vs `defer`

```
<script>                普通    ── 下载 → 阻塞解析 → 执行 → 继续解析
<script async>          异步    ── 并行下载，下完立即执行（顺序不保证）
<script defer>          延迟    ── 并行下载，DOM 解析完后按顺序执行
<script type="module">  等价于 defer（但不能加 async）
```

- **`async` 适合独立的第三方脚本**（埋点、广告），谁先下完谁先跑。
- **`defer` 适合业务代码**：保证执行顺序 + 不阻塞 HTML 解析。

### 4.4 HTTP / 传输层

| 手段 | 收益 |
|------|------|
| **HTTP/2 多路复用** | 一条 TCP 连接跑多个流，无队头阻塞（TCP 层）|
| **HTTP/3 (QUIC)** | 基于 UDP，彻底消除队头阻塞、0-RTT 建连 |
| **CDN** | 边缘节点分发静态资源，用户就近下载 |
| **Gzip / Brotli** | 文本类资源压缩率 60%~80%，Brotli 比 Gzip 再省 15% |
| **HTTPS + HSTS** | 强制 HTTPS，省一次 http→https 重定向 |

### 4.5 接口缓存策略（四层结构）

这是高频题，单独展开。分四层：**HTTP 缓存 → 请求去重 → 内存 SWR → 持久化缓存**。

#### 4.5.1 HTTP 缓存（浏览器原生层）

```
┌────────────────────────────────────────┐
│  强缓存（Fresh）：直接用本地副本         │
│  ├─ Cache-Control: max-age=3600        │
│  ├─ Cache-Control: s-maxage=...        │
│  └─ Expires: <date>（已过时）           │
├────────────────────────────────────────┤
│  协商缓存（Stale, 需验证）：发 304      │
│  ├─ ETag ↔ If-None-Match               │
│  └─ Last-Modified ↔ If-Modified-Since  │
└────────────────────────────────────────┘
```

| 指令 | 含义 |
|------|------|
| `no-store` | 完全不缓存 |
| `no-cache` | 可缓存但每次要协商 |
| `max-age=N` | N 秒内是强缓存 |
| `public` / `private` | 是否允许 CDN 缓存 |
| `stale-while-revalidate=N` | 过期后 N 秒内先用旧的，同时后台拉新（浏览器原生 SWR） |

**GET 才被缓存**，POST 默认不缓存。

#### 4.5.2 请求去重（In-Flight Deduplication）

同一页面里 3 个组件同时 `getUserInfo()` → 别发 3 次。

```js
const pending = new Map();
function fetchWithDedup(url, options) {
  const key = url + JSON.stringify(options ?? {});
  if (pending.has(key)) return pending.get(key);
  const p = fetch(url, options).then(r => r.json())
    .finally(() => pending.delete(key));   // 关键：无论成败都清
  pending.set(key, p);
  return p;
}
```

关键点：
- **finally 里删**，一次报错这个 key 就永远拿不到新数据。
- **key 要含请求体**，否则"同 URL 不同参数"会被错误合并。
- 只管**并发窗口**，落地后交给下一层。

#### 4.5.3 内存缓存 + SWR（Stale-While-Revalidate）

朴素 TTL 版：

```js
const cache = new Map();
async function fetchWithCache(url, ttl = 60_000) {
  const hit = cache.get(url);
  if (hit && hit.expireAt > Date.now()) return hit.data;
  const data = await fetch(url).then(r => r.json());
  cache.set(url, { data, expireAt: Date.now() + ttl });
  return data;
}
```

SWR 版思想（[SWR](https://swr.vercel.app/) / [React Query](https://tanstack.com/query) 的核心）：

```
第 1 次：无缓存 → 发请求 → 返回 + 存
第 2 次：① 立即返回旧数据（UI 秒开）
        ② 后台请求新数据 → ③ 更新 UI
```

React Query / SWR 库额外能力：`staleTime`、`gcTime`、`revalidateOnFocus`、`revalidateOnReconnect`、乐观更新、依赖查询。

#### 4.5.4 持久化缓存

| 存储 | 容量 | 同步/异步 | 适合什么 |
|------|------|-----------|---------|
| `localStorage` | ~5 MB | 同步（阻塞主线程） | 极小配置、字典 |
| `sessionStorage` | ~5 MB | 同步 | 本标签页需要 |
| `IndexedDB` | 数百 MB+ | 异步 | 图片、大 JSON、离线数据 |
| Cache Storage (SW) | 磁盘配额 | 异步 | PWA、离线接口 |

⚠️ 别放 Token（XSS 一读就没），别塞大数据（同步 API 卡主线程）。

#### 4.5.5 缓存失效（最难的部分）

> "There are only two hard things in Computer Science: cache invalidation and naming things." —— Phil Karlton

| 策略 | 适用 |
|------|------|
| TTL 到期 | 数据变化频率可预估 |
| 主动失效（Invalidate on Mutation） | `queryClient.invalidateQueries(['users'])` |
| 乐观更新 + 回滚 | 点赞、关注 |
| 版本号 / ETag | 配置、静态资源 |
| WebSocket / SSE 推送失效 | 协作应用（在线文档、看板） |

**中心化管理**：让 React Query / SWR / RTK Query 统一管，别每个组件自己维护一个 Map。

## 五、构建层优化

> 只讲构建产物层面对性能的影响。Webpack / Vite 本身的配置细节属于工程化范畴。

### 5.1 代码分割（Code Splitting）

**目标**：首屏只加载首屏用得到的代码。

三个粒度：

1. **路由级**：`React.lazy(() => import('./pages/Dashboard'))`
2. **组件级**：重组件懒加载（图表、编辑器、地图）
3. **依赖级**：第三方大库单独 chunk（`splitChunks` / `manualChunks`）

### 5.2 Tree Shaking

删除未引用的 ESM 导出。**失效情况**：CommonJS、有副作用的模块、Babel 把 ESM 转成 CJS。

```json
// package.json 明确告诉打包器"我没有副作用"
{ "sideEffects": false }

// 或指定哪些文件有副作用
{ "sideEffects": ["*.css", "./src/polyfill.js"] }
```

### 5.3 压缩与格式

| 类型 | 手段 |
|------|------|
| JS/CSS | Terser / esbuild 压缩，Gzip / Brotli 传输 |
| 图片 | WebP（比 JPEG 小 25-35%）/ AVIF（再小 20-50%）|
| SVG | `svgo` 优化路径、去除元数据 |
| 字体 | 子集化（只保留用到的字符）、WOFF2 |

### 5.4 分步加载策略

```
第一步（Critical）：首屏可见内容 → SSR / 内联关键 CSS
第二步（High）：首屏交互 JS → 代码分割 + preload
第三步（Medium）：非首屏路由 → 路由懒加载 + prefetch
第四步（Low）：图片 / 字体 / 分析脚本 → 懒加载 + async
```

React 中实践：

```jsx
const Dashboard = React.lazy(() => import('./pages/Dashboard'));
const HeavyChart = React.lazy(() => import('./components/HeavyChart'));

function App() {
  return (
    <Suspense fallback={<Skeleton />}>
      <Routes>
        <Route path="/dashboard" element={<Dashboard />} />
      </Routes>
    </Suspense>
  );
}
```

## 六、图片优化（单独一节，收益最大）

图片通常占页面体积的 60%+，是首屏优化的头号靶子。

### 6.1 格式选择

| 格式 | 特点 | 场景 |
|------|------|------|
| JPEG | 有损，通用 | 照片 |
| PNG | 无损，支持透明 | 截图、图标、需要透明 |
| WebP | 有损/无损都支持，比 JPEG 小 25-35% | 现代浏览器通用替代 |
| AVIF | 比 WebP 再小 20-50% | 现代浏览器（2020+），压缩最强 |
| SVG | 矢量，任意缩放不失真 | 图标、Logo |

### 6.2 响应式图片：`srcset` + `sizes`

让浏览器根据设备像素比 / 视口宽度选择最合适的图。

```html
<img
  src="hero-800.jpg"
  srcset="hero-400.jpg 400w, hero-800.jpg 800w, hero-1600.jpg 1600w"
  sizes="(max-width: 600px) 100vw, 50vw"
  alt="hero"
>
```

### 6.3 `<picture>`：格式回退

按浏览器支持能力挑选格式，老浏览器自动降级。

```html
<picture>
  <source srcset="hero.avif" type="image/avif">
  <source srcset="hero.webp" type="image/webp">
  <img src="hero.jpg" alt="hero">
</picture>
```

### 6.4 懒加载

```html
<!-- 原生懒加载：视口内才开始加载 -->
<img src="a.jpg" loading="lazy" decoding="async" width="800" height="600">
```

- `loading="lazy"`：视口外的图片延迟加载。
- `decoding="async"`：解码不阻塞主线程。
- **一定写 `width` / `height`**：避免图片加载完后布局跳动（CLS ↑）。

### 6.5 首屏图片的特殊待遇

首屏 LCP 图片千万**不要**懒加载，反而要 `<link rel="preload" as="image">` 甚至 `fetchpriority="high"`。

```html
<link rel="preload" as="image" href="hero.webp" fetchpriority="high">
<img src="hero.webp" fetchpriority="high" alt="">
```

## 七、渲染层优化

### 7.1 SSR / SSG / ISR / RSC

| 模式 | 全称 | 说明 | 适合 |
|------|------|------|------|
| **CSR** | Client-Side Rendering | 浏览器跑 JS 渲染 | 后台管理系统 |
| **SSR** | Server-Side Rendering | 服务端渲染 HTML | 内容型页面、SEO |
| **SSG** | Static Site Generation | 构建时生成 HTML | 博客、文档 |
| **ISR** | Incremental Static Regeneration | 后台定期重生成 SSG | 半静态的电商详情 |
| **RSC** | React Server Components | 组件级服务端渲染 | Next.js App Router |

SSR 的收益不在 TTI，而在 **LCP 和 SEO**——用户不必等 JS 下载完就能看到内容。

### 7.2 骨架屏 vs Spinner

- **骨架屏（Skeleton）**：模拟真实布局的灰色占位，用户感知"内容就要来了"。
- **Spinner**：只表达"正在加载"，用户感知不到进度。

现代做法：SSR 直接出骨架屏 → 数据到位后水合（Hydrate）。

### 7.3 字体优化

```css
@font-face {
  font-family: 'X';
  src: url('x.woff2') format('woff2');
  font-display: swap;     /* 先用系统字体渲染，字体到位后替换 */
}
```

- `font-display: swap` 避免"看不见文字（FOIT，Flash of Invisible Text）"。
- **字体子集化**（Subsetting）：只打包用到的字符，中文字体一般能从 10 MB 压到几百 KB。
- 首屏字体用 `<link rel="preload" as="font" crossorigin>`。

### 7.4 避免 CLS 的核心手段

- 图片、`iframe`、视频**必须给尺寸**（`width` / `height` 或 `aspect-ratio`）。
- 不要在已渲染内容上方**插入**新元素（广告、公告条 → 用 `position: fixed` 或预留空间）。
- 字体加载用 `font-display: swap` + `size-adjust`。

## 八、运行时优化

### 8.1 主线程与 Long Task

浏览器只有一个主线程，任何超过 **50ms** 的任务都算 Long Task，会阻塞用户输入。

**检测：**

```js
const observer = new PerformanceObserver(list => {
  for (const entry of list.getEntries()) {
    if (entry.duration > 50) console.warn('Long Task:', entry);
  }
});
observer.observe({ entryTypes: ['longtask'] });
```

**拆分：**

```js
// 用 requestIdleCallback / scheduler.postTask 把长任务切片
function splitLongTask(tasks, chunkSize = 5) {
  let i = 0;
  function next() {
    const end = Math.min(i + chunkSize, tasks.length);
    for (; i < end; i++) tasks[i]();
    if (i < tasks.length) {
      // Chrome 已有的现代 API,优先级可控
      if ('scheduler' in window) scheduler.postTask(next, { priority: 'background' });
      else requestIdleCallback(next);
    }
  }
  next();
}
```

现代 API 推荐顺序：**`scheduler.yield()` > `scheduler.postTask()` > `requestIdleCallback` > `setTimeout(fn, 0)`**。

### 8.2 防抖（Debounce）与节流（Throttle）

| 维度 | 防抖 | 节流 |
|------|------|------|
| 行为 | 连续触发只执行最后一次 | 固定时间间隔执行一次 |
| 适用 | 搜索输入、resize、表单校验 | 滚动加载、拖拽、鼠标移动 |

**防抖（带立即执行 + cancel）：**

```js
function debounce(fn, delay = 300, immediate = false) {
  let timer = null;
  const debounced = function (...args) {
    if (timer) clearTimeout(timer);
    if (immediate) {
      const callNow = !timer;
      timer = setTimeout(() => { timer = null; }, delay);
      if (callNow) fn.apply(this, args);
    } else {
      timer = setTimeout(() => {
        fn.apply(this, args);
        timer = null;
      }, delay);
    }
  };
  debounced.cancel = () => { clearTimeout(timer); timer = null; };
  return debounced;
}
```

**节流（第一次立即执行 + 停止后执行最后一次）：**

```js
function throttle(fn, delay = 300) {
  let lastTime = 0, timer = null;
  return function (...args) {
    const now = Date.now();
    const remaining = delay - (now - lastTime);
    if (remaining <= 0) {
      if (timer) { clearTimeout(timer); timer = null; }
      fn.apply(this, args);
      lastTime = now;
    } else if (!timer) {
      timer = setTimeout(() => {
        fn.apply(this, args);
        lastTime = Date.now();
        timer = null;
      }, remaining);
    }
  };
}
```

### 8.3 虚拟列表（Virtual List / Windowing）

只渲染可视区域内的 DOM 节点，用空白撑开滚动条。

```
startIndex = Math.floor(scrollTop / itemHeight)
endIndex   = Math.ceil((scrollTop + containerHeight) / itemHeight)
visible    = data.slice(startIndex - overscan, endIndex + overscan)
paddingTop = startIndex * itemHeight
paddingBottom = (total - endIndex) * itemHeight
```

**不定高列表**两种方案：
1. **预估高度 + 实测校准**（react-window）：预设 80px，`getBoundingClientRect()` 拿真实高度，更新缓存。
2. **ResizeObserver 动态监听**（react-virtuoso）：每一项挂个 observer，自动更新。

**空白闪烁**三种解法：
- **overscan**：上下各多渲染 2-5 个。
- **骨架屏占位**：未渲染区域预置灰底。
- **`min-height` 兜底**：避免高度突变。

### 8.4 大列表搜索的性能

- **前端筛选**：数据量大（>10k）时用 **Web Worker** 在独立线程搜索。
- **输入防抖**：300ms 后再触发。
- **结果也走虚拟列表**：避免搜索后 DOM 数量爆炸。
- **高亮**：用 `split` + 拼片段，别用正则全局 `replace innerHTML`（安全 + 性能双差）。

Web Worker 极简用法：

```js
// worker.js
self.onmessage = ({ data: { keyword, list } }) => {
  const result = list.filter(x => x.name.includes(keyword));
  postMessage(result);
};

// main.js
const worker = new Worker(new URL('./worker.js', import.meta.url), { type: 'module' });
worker.postMessage({ keyword, list: bigList });
worker.onmessage = ({ data }) => setResult(data);
```

### 8.5 React 运行时优化

| 手段 | 什么时候用 | 什么时候别用 |
|------|-----------|-------------|
| `React.memo` | 组件常收到相同 props 且渲染开销大 | props 每次都变、组件很轻 |
| `useMemo` | 计算开销大（比大 O 至少 O(n)）| 只是包个字面量、开销比比较还小 |
| `useCallback` | 传给 memo 化的子组件 / 依赖项 | 只给普通元素的 onClick |
| `key` 稳定 | 列表渲染 | 用数组 index 当 key（会导致复用错乱）|
| `useTransition` | 非紧急更新（搜索结果、tab 切换） | 用户直接输入的字段 |

**核心原则**：React 19 有编译器（React Compiler）自动优化，手动 `useMemo` / `useCallback` 反而增加维护成本。**先测量再优化**，别提前 memo 化。

### 8.6 强制同步布局（Layout Thrashing）

在同一帧里**读 → 写 → 读 → 写**几何属性 → 浏览器被迫多次布局。

```js
// ❌ 坏例子
elements.forEach(el => {
  el.style.width = el.offsetWidth + 10 + 'px'; // 读一次 → 写一次 → 下次读又强制重新布局
});

// ✅ 好例子：先全部读，再全部写
const widths = elements.map(el => el.offsetWidth);
elements.forEach((el, i) => { el.style.width = widths[i] + 10 + 'px'; });
```

触发布局的属性：`offsetWidth/Height/Top/Left`、`clientWidth/...`、`scrollTop/...`、`getBoundingClientRect()`、`getComputedStyle()`。

### 8.7 CSS 动画：只动 `transform` 和 `opacity`

浏览器渲染管线：**Style → Layout → Paint → Composite**。

| 属性 | 触发阶段 | 性能 |
|------|---------|------|
| `width` / `height` / `top` / `left` | Layout | 最差 |
| `background` / `color` / `box-shadow` | Paint | 差 |
| `transform` / `opacity` | 仅 Composite | 最佳（GPU 加速）|

```css
/* ❌ 会触发布局 */
.box { transition: left 0.3s; }
.box:hover { left: 100px; }

/* ✅ 只在合成层动 */
.box { transition: transform 0.3s; }
.box:hover { transform: translateX(100px); }
```

**`will-change`** 提前告诉浏览器"这里马上要动"，但别滥用（会占显存）。

## 九、性能监控（RUM）

真正的用户环境比你的开发机糟糕得多。要监控**真实用户**的数据。

### 9.1 采集三大类

| 类别 | 用什么 API |
|------|-----------|
| 核心指标 | `web-vitals` 库（LCP / INP / CLS / TTFB / FCP）|
| 资源加载 | `PerformanceObserver({ entryTypes: ['resource'] })` |
| 长任务 | `PerformanceObserver({ entryTypes: ['longtask'] })` |
| 错误 | `window.onerror` + `unhandledrejection` |
| 用户行为 | 自定义埋点 |

### 9.2 上报策略

- **`navigator.sendBeacon`**：页面 unload 时也能可靠上报（不阻塞跳转）。
- **合并批量**：短时间内的多条指标合并成一个请求。
- **采样**：不用 100% 采集，1%~10% 通常就够统计意义。

```js
import { onLCP, onINP, onCLS } from 'web-vitals'

function report(name, value) {
  const data = JSON.stringify({ name, value, url: location.href, t: Date.now() });
  navigator.sendBeacon('/rum', data);
}

onLCP(m => report('lcp', m.value));
onINP(m => report('inp', m.value));
onCLS(m => report('cls', m.value));
```

### 9.3 关注分位数，不是平均值

平均值会被少数极端值拉偏。**关键看 P75 / P95**：
- P75 LCP：75% 的用户体验的 LCP。Google 用这个作为"通过 Core Web Vitals"的门槛。
- P95：那些体验最差的 5% 用户——他们的问题往往才是真问题。

## 十、面试话术模板

被问 "你做过哪些性能优化？"，套这个模板：

> "我们的 XX 页面首屏 LCP 从 3.2s 降到 1.1s，主要做了三件事：
>
> **第一，网络层**：把首屏关键 CSS 内联进 HTML，LCP 图片 `<link rel="preload">` 加 `fetchpriority="high"`，第三方脚本改成 `defer`，节省了 300ms 的关键路径。
>
> **第二，构建层**：路由用 `React.lazy` 拆分，重组件（图表库、编辑器）按需引入，vendor 拆出单独 chunk，首屏 JS 从 800KB 降到 220KB。
>
> **第三，运行时**：长列表接了虚拟列表（react-virtuoso），INP 从 400ms 降到 80ms。搜索框加了 300ms 防抖 + Web Worker 做筛选，避免主线程被大数据阻塞。
>
> 上线后我们用 web-vitals 采集了 P75 数据，LCP、INP、CLS 都进了 Google 的绿色区间。"

**要点**：**指标要具体（有数字）、方案要分层、结尾用数据收尾**。
