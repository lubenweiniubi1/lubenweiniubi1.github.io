#  OD 外包全栈

## 一句话总结

**简历定位与真实能力的错配，是这次面试失败的根本原因。**"全栈"两个字引来了 Node 深问，而你没有接住，导致信任崩塌；其余题目的混乱表达，则是准备不足的直接表现。

## 致命问题

### 简历写"全栈"，Node 事件循环答不上来

这不是一道题的得失，是**信任崩塌**。

面试官的内心 OS：
> "他说是全栈，但连 Node 事件循环和浏览器有什么不同都说不清，那他简历里写的那些 Node 项目到底参与了多少？其他技术栈的真实水平是不是也得打个问号？"

**教训：简历上每一个词，都要做好被深挖的准备。接不住的，去掉或换成更保守的表达。**

## 各题诊断

### 1. CDN 为什么过期

- **问题**：回答太表层，只说"缓存时间到了"
- **应该**：讲缓存策略设计（Cache-Control、hash/版本号、CDN 刷新/预热、缓存一致性）

### 2. Node 事件循环 vs 浏览器事件循环

- **问题**：完全答不上来
- **必须补**：Node 6 阶段模型、`process.nextTick` 优先级、`setImmediate` vs `setTimeout(fn, 0)`

### 3. 最难的项目

- **问题**：说得混乱、磕巴
- **应该**：提前准备 STAR 故事，写成逐字稿，练 5 遍以上直到能自然说出口
- **STAR 框架**：
  1. 一句话背景（项目、角色）
  2. 核心难点（1-2 个，不是一堆）
  3. 我的方案（决策 + 为什么）
  4. 结果（数据化）
  5. 复盘（如果再做的更好）

### 4. Taro 原理

- **应该**：编译时（AST 转换）+ 运行时（统一 API/组件抹平差异），可对比 uni-app

### 5. 多端组件库选型

- **应该**：展示决策框架（需求 → 候选 → 评估维度 → 最终决策 + tradeoff），而不是"大家都用这个"

### 6. 减少 FCP

- **应该**：分层回答（网络层 → 资源层 → 渲染层 → 度量），展现系统性思维

### 7. RSC / CSR / SSR 区别

- **应该**：从渲染位置、JS 体积、SEO、首屏、交互五个维度对比，加分项是提到 Streaming SSR

### 手撕题 1：事件循环打印顺序

```
script start
async1 start
async2
promise1
async2 end
promise2
async1 end
setTimeout
```

**关键**：`await Promise.resolve()` 产生微任务，`async2 end` 比 `promise2` 先入微任务队列。

### 手撕题 2：Scheduler 并发控制

- **问题**：如果没做出来，说明 Promise 链和异步控制理解不够
- **核心**：队列 + 运行计数器 + 任务完成后自动调度下一个

## 三层问题诊断

### 第一层：准备不足（根本原因）
- 大部分题有标准回答框架，没提前准备逐字稿
- "最难的项目"这种必考题，没提前写好 STAR 故事

### 第二层：表达问题（直接表现）
- 结巴、混乱 → 没有结构化表达训练
- 讨好型语气 → 面试中过度迎合显得不自信

### 第三层：心态问题（深层原因）
- 把面试当"被审判"而不是"技术交流"
- 外包岗位让你预设了"低人一等"的心理
