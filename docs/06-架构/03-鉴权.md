## 二、基础概念：为什么需要鉴权？

HTTP 是**无状态**协议——服务器不会记得你上一次请求过什么。但业务上我们需要：

- 知道**你是谁**（认证 Authentication）
- 知道**你能做什么**（授权 Authorization）

于是就有了各种"给请求打上身份标签"的方案。

## 三、Cookie 鉴权（最经典的方案）

### 3.1 核心流程

```
① 用户 POST /login   →  账号密码
② 服务器校验通过     →  生成 sessionId，存入服务端 Session 库（内存/Redis）
③ 响应 Set-Cookie: sessionId=xxx; HttpOnly
④ 浏览器自动保存 Cookie
⑤ 之后每次请求      →  浏览器自动带上 Cookie
⑥ 服务器用 sessionId 去 Session 库查用户 → 认出你
```

### 3.2 关键点

- **浏览器自动携带**：同源请求会自动带上 Cookie，前端不用手动加。
- **服务端有状态**：Session 数据存在服务器上，用户越多内存/Redis 压力越大。
- **HttpOnly**：JS 读不到，能防 XSS 窃取。
- **Secure**：只在 HTTPS 下发送。
- **SameSite**：Lax/Strict/None，防 CSRF。

### 3.3 缺点

| 问题 | 说明 |
|------|------|
| 服务端有状态 | 分布式部署要共享 Session（黏性会话 / Redis 集中存储） |
| 跨域麻烦 | Cookie 默认不跨域，`withCredentials` + CORS 才行 |
| 移动端/小程序不友好 | 原生 App、跨端环境没有浏览器的 Cookie 机制 |
| CSRF（Cross-Site Request Forgery，跨站请求伪造）风险 | 因为会"自动携带"，恶意站点也能诱导发起请求 |

## 四、请求头鉴权（Token / JWT，当前主流）

### 4.1 核心流程

```
① 用户 POST /login   →  账号密码
② 服务器校验通过     →  签发一个 Token（通常是 JWT）
③ 前端把 Token 存起来（localStorage / sessionStorage / Cookie）
④ 之后每次请求      →  手动带上   Authorization: Bearer <token>
⑤ 服务器校验签名     →  取出用户信息，无需查库
```

### 4.2 JWT（JSON Web Token，JSON 网络令牌）是什么？

JWT = `Header.Payload.Signature`，三段用 `.` 拼接，**Base64 编码**（不是加密！）。

```
eyJhbGciOiJIUzI1NiJ9  .  eyJ1aWQiOjEsImV4cCI6MTcwMH0  .  签名
     Header                     Payload                Signature
```

- **Header**：算法类型，例如 HS256
- **Payload**：业务信息，如 `userId`、`exp` 过期时间（⚠️ 明文可解，别放密码）
- **Signature**：`HMAC(secret, header + payload)`，防篡改

**常见误区**：JWT 不是加密的，任何人都能 Base64 解开看内容，只是不能篡改。

### 4.3 为什么现在主流用 Token？

- ✅ **服务端无状态**：拿签名验一下就行，天然适合分布式、微服务
- ✅ **跨端友好**：Web、App、小程序都能用同一套
- ✅ **跨域简单**：放在请求头里，不受 Cookie 同源策略约束
- ❌ **无法主动踢人**：Token 一旦签发，过期前都有效（需配合黑名单/短期 Token）
- ❌ **存储有风险**：localStorage 会被 XSS 读到

### 4.4 Token 存哪里？

| 存储位置 | XSS（Cross-Site Scripting，跨站脚本攻击）风险 | CSRF（Cross-Site Request Forgery，跨站请求伪造）风险 | 说明 |
|----------|---------|-----------|------|
| localStorage（本地存储） | ⚠️ 高 | ✅ 低 | 最常见，但要做好 XSS 防护 |
| sessionStorage（会话存储） | ⚠️ 高 | ✅ 低 | 关闭标签页就没了 |
| Cookie (HttpOnly，仅服务器可读) | ✅ 低 | ⚠️ 高 | 需要 SameSite + CSRF Token |

**折中方案**：`accessToken` 放内存，`refreshToken` 放 HttpOnly Cookie。

### 4.5 无感刷新（双 Token 方案）

```
accessToken   短期（15min），每次请求都带
refreshToken  长期（7天），只在 accessToken 过期时用

流程：
   请求 A → 401 过期
   → 用 refreshToken 换新的 accessToken
   → 重发请求 A（用户完全无感知）
```

前端一般在 axios 响应拦截器里实现，注意**并发请求要排队**，避免多次刷新。

## 五、前端路由守卫（不同用户不同页面）

**⚠️ 前端鉴权只是 UX，不是安全边界**。真正的权限校验必须在后端。前端做的事只是"不给你看菜单"，接口该拦还得后端拦。

### 5.1 三种粒度

1. **页面级**：某些路由只有登录用户/管理员能访问
2. **菜单级**：根据角色动态渲染侧边栏
3. **按钮级**：根据权限点显示/隐藏按钮（v-permission 指令 / `<Permission>` 组件）

### 5.2 Vue Router 示例

```js
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token')
  const userRole = store.state.user.role

  if (to.meta.requiresAuth && !token) {
    return next('/login')          // 没登录 → 去登录
  }
  if (to.meta.roles && !to.meta.roles.includes(userRole)) {
    return next('/403')             // 权限不够
  }
  next()
})
```

### 5.3 React 示例

```jsx
function ProtectedRoute({ children, roles }) {
  const { user } = useAuth()
  if (!user) return <Navigate to="/login" />
  if (roles && !roles.includes(user.role)) return <Navigate to="/403" />
  return children
}
```

### 5.4 动态路由（后端返回菜单）

```
① 登录成功 → 后端返回该用户的路由/菜单列表
② 前端调用 router.addRoute() 动态注册
③ 未注册的路由 → 404
```

这样"权限即路由"，无法通过 URL 猜到隐藏页面。

## 六、进阶话题

### 6.1 CSRF（Cross-Site Request Forgery，跨站请求伪造）攻击与防御

- **原理**：恶意站点让你的浏览器发起带 Cookie 的请求。
- **防御**：
  - `SameSite=Lax/Strict` Cookie
  - CSRF Token（表单里带一个后端签发的随机值）
  - 关键操作二次确认（如短信验证码）

### 6.2 XSS（Cross-Site Scripting，跨站脚本攻击）攻击与防御

- **原理**：注入脚本，读取 localStorage 里的 Token。
- **防御**：
  - 转义所有用户输入（框架默认帮你做了）
  - CSP（Content-Security-Policy，内容安全策略）响应头
  - Token 尽量不放 localStorage

### 6.3 单点登录 SSO（Single Sign-On）

多个子系统共享登录状态。常见方案：

- **CAS**（Central Authentication Service，中央认证服务）：中心认证服务器，票据（ticket）跳转
- **OAuth 2.0**（Open Authorization 2.0，开放授权 2.0）：授权码模式，第三方登录（微信、GitHub）
- **OIDC**（OpenID Connect，开放身份连接）：OAuth 之上的身份层，返回 id_token

### 6.4 OAuth 2.0 四种授权模式

| 模式 | 场景 |
|------|------|
| 授权码 Authorization Code | 最常用，Web 应用 |
| 隐式 Implicit（隐式授权，已废弃） | 已过时，纯前端 SPA |
| 密码 Password（资源所有者密码凭证） | 高信任场景（自家 App） |
| 客户端 Client Credentials（客户端凭证） | 服务器之间调用 |

## 七、方案对比速查表

| 维度 | Cookie + Session | Token / JWT |
|------|------------------|-------------|
| 状态 | 服务端有状态 | 服务端无状态 |
| 存储位置 | 服务端 Session 库 | 客户端 |
| 携带方式 | 浏览器自动 | 前端手动 `Authorization` |
| 跨域 | 麻烦 | 简单 |
| 多端支持 | 弱 | 强 |
| 踢人下线 | 容易（删 Session） | 难（需黑名单） |
| CSRF（跨站请求伪造） | 有风险 | 天然免疫 |
| XSS（跨站脚本攻击） | HttpOnly 可防 | localStorage 易被读 |

---

## 八、一句话总结

> **认证解决"你是谁"，授权解决"你能干嘛"；前端做体验，后端做安全。**
