## 二、基础概念：为什么需要鉴权？

HTTP 是**无状态**协议——服务器不会记得你上一次请求过什么。但业务上我们需要：

- 知道**你是谁**（认证 Authentication）
- 知道**你能做什么**（授权 Authorization）

于是就有了各种"给请求打上身份标签"的方案。

## 三、Cookie 鉴权（最经典的方案）

### 3.1 核心流程

```
① 用户 POST /login   →  账号密码
② 服务器校验通过     →  生成 sessionId，存入服务端 Session 库（内存/Redis）
③ 响应 Set-Cookie: sessionId=xxx; HttpOnly
④ 浏览器自动保存 Cookie
⑤ 之后每次请求      →  浏览器自动带上 Cookie
⑥ 服务器用 sessionId 去 Session 库查用户 → 认出你
```

### 3.2 Cookie 的关键属性详解

服务器在响应头里写 `Set-Cookie` 时，可以带一堆配置项，它们决定了这个 Cookie 什么时候能被带、被谁读、能不能防攻击。

一个真实的响应头长这样：

```
Set-Cookie: sessionId=abc123; Domain=example.com; Path=/; Max-Age=3600; HttpOnly; Secure; SameSite=Lax
```

| 属性 | 作用 | 举例 / 说明 |
|------|------|------|
| **Name=Value** | Cookie 的键值对 | `sessionId=abc123` |
| **Domain** | 允许哪些域名带上这个 Cookie | 设为 `example.com` 时，`a.example.com`、`b.example.com` 都能带；不设默认是当前域名（不含子域）|
| **Path** | 只在匹配路径下才发送 | `Path=/admin` 时，`/admin/*` 请求会带，`/user/*` 不会 |
| **Expires / Max-Age** | 过期时间 | `Max-Age=3600`（秒）；不设则是**会话 Cookie**，关浏览器就没了 |
| **HttpOnly** | 禁止 JS 通过 `document.cookie` 读取 | 防止 XSS 脚本偷 Cookie；⚠️ 只是防"读"，浏览器发请求还是会自动带 |
| **Secure** | 只在 HTTPS 下才发送 | 防止明文传输被抓包 |
| **SameSite** | 控制**跨站**请求要不要带 Cookie（防 CSRF 的核心） | 见下方详解 |

#### 3.2.1 SameSite 三种取值

这是防 CSRF 最重要的一个属性。所谓"跨站"，就是**当前页面的站点** ≠ **请求目标站点**（注意是"站点 site"，不是"源 origin"，端口/子域一般算同站）。

| 取值 | 行为 | 场景 |
|------|------|------|
| **Strict**（严格） | 完全不允许跨站带 Cookie | 从别的网站点链接跳过来，都不带 Cookie，用户会"看似未登录"。适合银行等极敏感场景 |
| **Lax**（宽松，现代浏览器默认值） | 顶层导航（点链接、地址栏输入）跨站可以带；但 iframe、img、`fetch`、表单 POST 都不带 | 大多数网站合理的默认，既能防 CSRF，又不影响正常跳转 |
| **None** | 完全允许跨站带 Cookie | 必须同时设 `Secure`，否则浏览器拒收。用于第三方嵌入场景（如广告 SDK、跨站 SSO） |

**举例**：你登录了 `bank.com`，然后打开一个恶意页面 `evil.com`，里面有：
```html
<img src="https://bank.com/transfer?to=hacker&amount=1000" />
```
- `SameSite=Lax`：浏览器不带 `bank.com` 的 Cookie → 转账失败 ✅
- `SameSite=None`：浏览器带上 Cookie → 转账成功 ❌（CSRF 成功）

#### 3.2.2 跨域时的 Cookie（配合 CORS）

跨域请求默认**不会**带 Cookie，需要三处配合：

```js
// ① 前端
fetch('https://api.example.com/user', { credentials: 'include' })
// axios.defaults.withCredentials = true
```

```
// ② 后端响应头
Access-Control-Allow-Origin: https://web.example.com   // 不能是 *
Access-Control-Allow-Credentials: true
```

```
// ③ Set-Cookie 属性
Set-Cookie: token=xxx; SameSite=None; Secure
```

三者缺一不可，很多"登录后接口 401"的坑都出在这里。

### 3.3 缺点

| 问题 | 说明 |
|------|------|
| 服务端有状态 | 分布式部署要共享 Session（黏性会话 / Redis 集中存储） |
| 跨域麻烦 | Cookie 默认不跨域，`withCredentials` + CORS 才行 |
| 移动端/小程序不友好 | 原生 App、跨端环境没有浏览器的 Cookie 机制 |
| CSRF 风险 | 因为会"自动携带"，恶意站点也能诱导发起请求（可用 SameSite 缓解） |

## 四、请求头鉴权（Token / JWT，当前主流）

### 4.0 先厘清概念：Token vs JWT

很多人把这俩混着说，其实是不同层面的东西：

| 概念 | 是什么 |
|------|-------|
| **Token（令牌）** | 一个泛称，指"一串代表身份的字符串"。可以是 UUID、随机串、JWT 都行。 |
| **JWT（JSON Web Token）** | Token 的一种**具体实现规范**，特点是自包含 + 签名。 |

**举例**：
- 「不透明 Token」（Opaque Token）：随机字符串 `a3f9e2c1b4d5...`，服务端必须去 Redis 查才能知道对应哪个用户。
- 「JWT Token」：`xxx.yyy.zzz`，服务端**验签**就能直接从里面读出用户信息，不用查库。

主流用 JWT，因为无状态、性能好。但缺点是"没法撤销"——下面细讲。

### 4.1 核心流程

```
① 用户 POST /login   →  账号密码
② 服务器校验通过     →  签发一个 Token（通常是 JWT）
③ 前端把 Token 存起来（内存 / localStorage / Cookie）
④ 之后每次请求      →  手动带上   Authorization: Bearer <token>
⑤ 服务器校验签名     →  取出用户信息，无需查库
⑥ 返回业务数据
```

**`Bearer` 是什么？** HTTP 认证规范里的一种方案，直译"持票人"——意思是"谁拿着这个 token 谁就是本人"。所以千万别泄漏。

### 4.2 JWT（JSON Web Token，JSON 网络令牌）深入

#### 4.2.1 结构：三段式

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9  .  eyJ1aWQiOjEsImV4cCI6MTcwMH0  .  签名字符串
        Header（头部）                          Payload（载荷）              Signature（签名）
```

三段用 `.` 拼接，每段都是 **Base64URL 编码**（不是加密！可以直接 https://jwt.io 解开看）。

#### 4.2.2 Header（头部）

指定算法和 token 类型：

```json
{
  "alg": "HS256",   // 签名算法：HMAC-SHA256（对称）/ RS256（非对称）
  "typ": "JWT"
}
```

- **HS256**：对称加密，签名和验签用同一个密钥。简单，但所有服务都要拿到密钥，微服务多的话不安全。
- **RS256**：非对称，认证中心用私钥签，业务方用公钥验。私钥不外泄，适合多服务场景。

#### 4.2.3 Payload（载荷）

放业务数据，字段分两类：

**RFC 7519 规定的标准字段（Claims）**：

| 字段 | 全称 | 含义 |
|------|------|------|
| `iss` | issuer | 签发人（认证中心域名） |
| `sub` | subject | 主题（通常是 userId） |
| `aud` | audience | 接收方 |
| `exp` | expiration time | 过期时间戳（秒） |
| `iat` | issued at | 签发时间戳 |
| `nbf` | not before | 生效时间（在此之前无效） |
| `jti` | JWT ID | token 唯一 ID（做黑名单用） |

**业务自定义字段**：

```json
{
  "sub": "10086",
  "name": "张三",
  "role": "admin",
  "exp": 1720000000
}
```

⚠️ **Payload 是明文的**，任何人都能解开看。**不要放密码、身份证号、手机号**等敏感信息。

#### 4.2.4 Signature（签名）—— 防篡改的核心

```
Signature = HMAC-SHA256(
  Base64URL(Header) + "." + Base64URL(Payload),
  secret
)
```

签名把"头部 + 载荷 + 密钥"哈希在一起。任何人改一个字，签名就对不上了。**服务端只要用同一个密钥重算一遍签名对比即可，不需要查库**。

#### 4.2.5 手写一个 JWT（伪代码）

```js
const header  = base64url({ alg: 'HS256', typ: 'JWT' })
const payload = base64url({ sub: '10086', exp: Date.now()/1000 + 3600 })
const signature = base64url(HMAC_SHA256(header + '.' + payload, SECRET))

const jwt = `${header}.${payload}.${signature}`
// eyJhbGciOi...eyJzdWIiOi...abcd1234
```

服务端验证：

```js
function verify(jwt) {
  const [header, payload, sig] = jwt.split('.')
  const expected = base64url(HMAC_SHA256(header + '.' + payload, SECRET))
  if (sig !== expected) throw '签名不对，被篡改了'

  const data = JSON.parse(base64urlDecode(payload))
  if (data.exp < Date.now()/1000) throw 'token 过期'
  return data
}
```

### 4.3 JWT 优缺点（面试高频）

#### ✅ 优点

1. **无状态**：服务端不存 Session，只要验签就行。天然适合分布式、微服务、Serverless。
2. **自包含**：Payload 里已经有用户信息，省一次查库。
3. **跨端友好**：Web / App / 小程序都能用。
4. **跨域简单**：放在请求头里，不受 Cookie 同源策略约束。

#### ❌ 缺点

1. **无法主动踢人**：Token 一旦签发，在过期前都有效。用户改密码后老 token 依然能用。
   - **解决办法**：短期 accessToken + 黑名单（记录被撤销的 `jti`）。
2. **不能续期**：Token 是死的，快过期时怎么办？
   - **解决办法**：双 Token（accessToken + refreshToken，见 4.5）。
3. **Payload 明文**：不能放敏感数据。
   - **解决办法**：真需要加密就用 **JWE**（JSON Web Encryption），实际很少用。
4. **体积大**：JWT 比 sessionId 长得多，每次请求都带，浪费带宽。
5. **XSS 风险**：如果放 localStorage，被脚本注入就完蛋。
   - **解决办法**：放内存 + refreshToken 放 HttpOnly Cookie。

### 4.4 Token 存哪里？（三种存储位置对比）

Token 拿到手之后要存起来，等下一次请求时再读出来放进请求头。存的位置直接决定了它的安全性：

| 存储位置 | 生命周期 | JS 能读吗 | 关闭浏览器后 | XSS 风险 | CSRF 风险 |
|----------|---------|----------|------------|---------|-----------|
| **localStorage**（本地存储） | 永久，直到手动清 | ✅ 能读 | 还在 | ⚠️ 高（脚本可偷） | ✅ 低（不自动带） |
| **sessionStorage**（会话存储） | 当前标签页 | ✅ 能读 | 没了 | ⚠️ 高 | ✅ 低 |
| **Cookie (HttpOnly)** | 由服务端属性决定 | ❌ 读不到 | 看 Max-Age | ✅ 低 | ⚠️ 高（自动带） |
| **内存变量**（如 JS 变量 / Redux / Pinia） | 页面存活期间 | ✅ 能读（仅当前页） | 没了，刷新也没 | ⚠️ 中（需页面已被注入） | ✅ 低 |

> "放内存"就是**存在一个 JS 变量里**（比如 `let accessToken = xxx` 或 Vuex/Redux/Pinia 的 store）。
> 优点：刷新/关页就没了，XSS 也很难在极短时间内窃取；缺点：**页面一刷新就丢**，需要用 refreshToken 重新拿。

### 4.5 无感刷新（双 Token 方案）—— 详解

#### 4.5.1 为什么要两个 Token？

单 Token 有个两难：
- 有效期短 → 用户老掉线，体验差
- 有效期长 → 一旦泄漏，风险巨大

双 Token 把"日常鉴权"和"续命"拆开：

| Token | 用途 | 有效期 | 存哪 | 谁在带 |
|-------|------|-------|------|-------|
| **accessToken**（访问令牌） | 每次业务请求都带 | 短（15 分钟 ~ 2 小时） | 内存 / localStorage | 前端手动放 `Authorization` |
| **refreshToken**（刷新令牌） | 只用来换新的 accessToken | 长（7 天 ~ 30 天） | **HttpOnly Cookie**（推荐） | 浏览器自动带 |

**为什么 refreshToken 放 HttpOnly Cookie？** 它是"续命钥匙"，一旦被 JS 读到偷走，攻击者可以无限续期。放 HttpOnly Cookie 后：
- JS 读不到（防 XSS）
- 只在调 `/refresh` 接口时自动带（`Path=/refresh`）
- 配 `SameSite=Strict` 防 CSRF

#### 4.5.2 完整流程图

```
┌────────────┐                                    ┌────────────┐
│   前端      │                                    │   后端      │
└─────┬──────┘                                    └─────┬──────┘
      │  ① POST /login {账号,密码}                       │
      │ ─────────────────────────────────────────────▶ │
      │                                                │
      │  ② 200 { accessToken }                         │
      │     Set-Cookie: refreshToken=xxx; HttpOnly     │
      │ ◀───────────────────────────────────────────── │
      │                                                │
      │  【前端】把 accessToken 存内存                    │
      │                                                │
      │  ③ GET /api/user                              │
      │     Authorization: Bearer <accessToken>       │
      │ ─────────────────────────────────────────────▶ │
      │                                                │
      │  ④ 15 分钟后过期，返回 401                        │
      │ ◀───────────────────────────────────────────── │
      │                                                │
      │  ⑤ POST /refresh（Cookie 自动带 refreshToken）  │
      │ ─────────────────────────────────────────────▶ │
      │                                                │
      │  ⑥ 200 { accessToken: 新的 }                    │
      │ ◀───────────────────────────────────────────── │
      │                                                │
      │  ⑦ 用新 accessToken 重放 ③ 的请求                │
      │ ─────────────────────────────────────────────▶ │
      │                                                │
      │  ⑧ 200 { user: ... } （用户完全无感知）           │
      │ ◀───────────────────────────────────────────── │
```

#### 4.5.3 "放内存"怎么取出来用？

其实就是普通的 JS 变量。以 axios + Pinia 为例：

```js
// stores/auth.js — 用 Pinia 存 accessToken
import { defineStore } from 'pinia'

export const useAuthStore = defineStore('auth', {
  state: () => ({
    accessToken: ''       // 就是一个变量，页面刷新会没
  }),
  actions: {
    setToken(t) { this.accessToken = t },
    clearToken() { this.accessToken = '' }
  }
})
```

```js
// api/http.js — axios 拦截器统一读/写
import axios from 'axios'
import { useAuthStore } from '@/stores/auth'

const http = axios.create({
  baseURL: '/api',
  withCredentials: true   // 让 refreshToken 的 Cookie 能被带
})

// 请求拦截：从内存拿 accessToken 放进请求头
http.interceptors.request.use(config => {
  const auth = useAuthStore()
  if (auth.accessToken) {
    config.headers.Authorization = `Bearer ${auth.accessToken}`
  }
  return config
})
```

#### 4.5.4 无感刷新的响应拦截器（含并发处理）

难点：如果同一时刻有 3 个请求都过期了 401，你不能刷 3 次。要**排队**：

```js
let isRefreshing = false        // 正在刷新的锁
let waitQueue = []              // 等待新 token 的请求队列

http.interceptors.response.use(
  res => res,
  async error => {
    const { config, response } = error
    const auth = useAuthStore()

    // 只在 401 且不是刷新请求本身时处理
    if (response?.status !== 401 || config.url === '/refresh') {
      return Promise.reject(error)
    }

    // 场景一：已经有别的请求在刷新了 → 排队等新 token
    if (isRefreshing) {
      return new Promise(resolve => {
        waitQueue.push((newToken) => {
          config.headers.Authorization = `Bearer ${newToken}`
          resolve(http(config))      // 拿到新 token 后重发
        })
      })
    }

    // 场景二：第一个进入的请求负责去刷新
    isRefreshing = true
    try {
      // Cookie 里的 refreshToken 会自动带上
      const { data } = await http.post('/refresh')
      auth.setToken(data.accessToken)

      // 唤醒队列里所有等待的请求
      waitQueue.forEach(cb => cb(data.accessToken))
      waitQueue = []

      // 重发当前这个请求
      config.headers.Authorization = `Bearer ${data.accessToken}`
      return http(config)
    } catch (e) {
      // refreshToken 也过期 → 只能重新登录
      auth.clearToken()
      window.location.href = '/login'
      return Promise.reject(e)
    } finally {
      isRefreshing = false
    }
  }
)
```

#### 4.5.5 刷新页面之后怎么办？

内存里的 accessToken 没了，但 refreshToken 还在 HttpOnly Cookie 里。所以：

```js
// main.js / app 初始化时
async function bootstrap() {
  try {
    const { data } = await http.post('/refresh')   // Cookie 自动带
    useAuthStore().setToken(data.accessToken)
  } catch {
    // refreshToken 也过期了，跳登录页
  }
  app.mount('#app')
}
```

这样刷新页面用户依旧是登录态，"无感"就体现在这里。

#### 4.5.6 面试话术版总结

> 双 Token 的核心思想是**用短期的 accessToken 做业务鉴权、用长期的 refreshToken 做续期**。accessToken 放内存里刷新即失，安全性高；refreshToken 放 HttpOnly Cookie，JS 偷不到，只在调 `/refresh` 时由浏览器自动携带。响应拦截器里判断 401 就去调刷新接口拿新 token 重发请求，并发场景下用一个锁 + 队列保证只刷新一次。整个过程用户无感知。

## 五、前端路由守卫（不同用户不同页面）

**⚠️ 前端鉴权只是 UX，不是安全边界**。真正的权限校验必须在后端。前端做的事只是"不给你看菜单"，接口该拦还得后端拦。

路由守卫涉及 Vue Router 的完整导航解析流程、React Router 的 `loader`、Next.js 的 middleware / Server Component、动态路由注册、按钮级权限、常见死循环等话题，独立成文：

👉 [05-路由守卫.md](./05-路由守卫.md)

一句话概括：**middleware/loader/beforeEach 拦「能不能进」，动态路由和菜单决定「能看到什么」，按钮级权限决定「能操作什么」**。

## 六、进阶话题

### 6.1 CSRF（Cross-Site Request Forgery，跨站请求伪造）攻击与防御

- **原理**：恶意站点让你的浏览器发起带 Cookie 的请求。
- **防御**：
  - `SameSite=Lax/Strict` Cookie
  - CSRF Token（表单里带一个后端签发的随机值）
  - 关键操作二次确认（如短信验证码）

### 6.2 XSS（Cross-Site Scripting，跨站脚本攻击）攻击与防御

- **原理**：注入脚本，读取 localStorage 里的 Token。
- **防御**：
  - 转义所有用户输入（框架默认帮你做了）
  - CSP（Content-Security-Policy，内容安全策略）响应头
  - Token 尽量不放 localStorage

### 6.3 单点登录 SSO（Single Sign-On）

一次登录，多个系统通用。详细流程（CAS、OAuth 2.0 授权码、PKCE、OIDC、SLO 等）见独立文档：

👉 [04-SSO单点登录.md](./04-SSO单点登录.md)

一句话概括：**把认证剥离到独立的认证中心，通过重定向 + 一次性票据在多个业务系统间传递身份**。

### 6.4 OAuth 2.0 四种授权模式（简表）

| 模式 | 场景 |
|------|------|
| 授权码 Authorization Code | 最常用，Web 应用 |
| 隐式 Implicit（已废弃） | 曾用于纯前端 SPA，现改用授权码 + PKCE |
| 密码 Password | 高信任场景（自家 App） |
| 客户端 Client Credentials | 服务器之间调用 |

详细流程见 [04-SSO单点登录.md](./04-SSO单点登录.md)。

## 七、方案对比速查表

| 维度 | Cookie + Session | Token / JWT |
|------|------------------|-------------|
| 状态 | 服务端有状态 | 服务端无状态 |
| 存储位置 | 服务端 Session 库 | 客户端 |
| 携带方式 | 浏览器自动 | 前端手动 `Authorization` |
| 跨域 | 麻烦 | 简单 |
| 多端支持 | 弱 | 强 |
| 踢人下线 | 容易（删 Session） | 难（需黑名单） |
| CSRF（跨站请求伪造） | 有风险 | 天然免疫 |
| XSS（跨站脚本攻击） | HttpOnly 可防 | localStorage 易被读 |
