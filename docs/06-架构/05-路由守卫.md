# 前端路由守卫（Route Guard）

> 从 [03-鉴权.md](./03-鉴权.md) 拆出来的独立主题。鉴权解决"你是谁"，路由守卫解决"这个页面你能不能进"。

## 一、常见面试题

1. **前端路由守卫的作用是什么？前端做的鉴权真的能"防"住用户吗？**
2. **Vue Router 有哪几种守卫？完整的导航解析流程（Navigation Resolution Flow）是怎样的？**
3. **`beforeEach`、`beforeResolve`、`afterEach` 有什么区别？**
4. **动态路由（`addRoute`）怎么实现"按角色显示菜单"？刷新页面为什么会 404？怎么解决？**
5. **React Router v6 里没有 `beforeEach`，怎么实现路由守卫？**
6. **`loader` / `action` / `Navigate` / `useNavigate` 在权限控制里各自扮演什么角色？**
7. **Next.js（App Router）里怎么做路由级鉴权？`middleware.ts`、Server Component、`layout.tsx` 各适合什么场景？**
8. **前端路由守卫经常出现"死循环重定向"，怎么排查？**
9. **RBAC（Role-Based Access Control，基于角色）和 ABAC（Attribute-Based Access Control，基于属性）有什么区别？**
10. **按钮级权限（v-permission 指令 / `<Permission>` 组件）怎么设计？**

## 二、为什么需要路由守卫？

HTTP 无状态、SPA（Single Page Application，单页应用）又把"页面切换"变成了 JS 里的一次 `history.pushState`——**没有一次真正的服务器请求**。这意味着：

- 用户可以在地址栏直接敲 `/admin`，前端如果不拦，就把管理员页面渲染出来了。
- 菜单如果对所有人一样，普通用户能看到"用户管理"按钮，点进去接口报 403，体验极差。
- 未登录用户打开 `/order/detail`，应该被弹回登录页，登录后再跳回来。

路由守卫就是**在路由跳转的"缝隙"里插入一段逻辑**，用来：

1. 判断是否登录 → 未登录跳 `/login`
2. 判断是否有权限 → 无权限跳 `/403`
3. 动态注册路由 / 加载数据 / 埋点

### ⚠️ 最重要的一句话

> **前端路由守卫只是 UX（User Experience，用户体验）层的兜底，不是安全边界。**

用户 F12 打开控制台改一下 `store.user.role = 'admin'`，或者直接改 URL，理论上都能"绕过"前端守卫。**真正的权限校验必须在后端**（接口级）。前端做的事只有两件：

- 不让用户看到"根本没权限"的入口
- 提前把没权限的跳转拦下来，避免调接口才发现 403

## 三、权限的三种粒度

| 粒度 | 表现 | 实现位置 |
|------|------|---------|
| **页面级**（Route-Level） | 某些路由只有登录用户 / 特定角色能访问 | 路由守卫 |
| **菜单级**（Menu-Level） | 侧边栏、导航栏根据角色动态渲染 | 用户信息返回菜单树后过滤/动态注册 |
| **按钮级**（Element-Level） | "删除""审批"等按钮根据权限点显示或禁用 | 指令 / 组件 / Hook |

三者的数据源通常来自后端登录接口返回的 `permissions`（权限点列表）或 `menus`（菜单树）。

## 四、权限模型：RBAC vs ABAC

在写代码前，先想清楚"权限"到底是什么。

### 4.1 RBAC（Role-Based Access Control，基于角色的访问控制）

- **角色**是权限的集合。用户绑定一个或多个角色。
- 例：`admin` 角色拥有"用户管理""订单管理"权限；`user` 角色只有"我的订单"权限。
- 优点：直观、后台好维护。
- 缺点：粒度粗。想给"某个部门经理只看本部门订单"，RBAC 就吃力了。

```js
// 前端只需要判断 user.role 是否在允许列表
if (route.meta.roles?.includes(user.role)) { /* pass */ }
```

### 4.2 ABAC（Attribute-Based Access Control，基于属性的访问控制）

- 权限判断基于**用户属性 + 资源属性 + 环境属性**的组合规则。
- 例：`user.department === resource.department && time < 18:00`。
- 优点：灵活、能表达复杂场景。
- 缺点：规则引擎复杂，前端一般不承担。

### 4.3 前端最常见的落地：权限点（Permission Point）

RBAC 的一种精细化实现——把"能做什么"抽成一个个字符串标识，登录时后端把用户拥有的权限点列表全返回：

```json
{
  "user": { "id": 1, "name": "Alice" },
  "permissions": [
    "user:list", "user:create",
    "order:list", "order:export"
  ]
}
```

页面守卫、菜单渲染、按钮显隐，全都基于这个字符串数组判断。

## 五、Vue Router 路由守卫

### 5.1 完整的导航解析流程（Full Navigation Resolution Flow）

Vue Router 官方文档里给出了一个 12 步的"守卫触发顺序"，考试常考：

```
① 导航被触发（url 变化 / router.push）
② 在失活的组件里调用 beforeRouteLeave
③ 调用全局的 beforeEach
④ 在重用的组件里调用 beforeRouteUpdate（如动态路由参数变化）
⑤ 调用路由配置里的 beforeEnter
⑥ 解析异步路由组件（懒加载）
⑦ 在被激活的组件里调用 beforeRouteEnter
⑧ 调用全局的 beforeResolve
⑨ 导航被确认
⑩ 调用全局的 afterEach
⑪ 触发 DOM 更新
⑫ 调用 beforeRouteEnter 中传给 next 的回调（此时组件实例已挂载）
```

关键区别：

| 钩子 | 时机 | 常见用途 |
|------|------|---------|
| **beforeEach** | 每次导航开始时 | 登录校验、权限判断 |
| **beforeResolve** | 所有组件内守卫和异步路由都解析完之后 | 需要数据全部就绪时的最后一次拦截（如埋点前置） |
| **afterEach** | 导航确认之后 | 不能取消跳转；用来关 loading、改 title、埋点 |

### 5.2 基础实现：登录 + 角色

```js
// router/guard.js
router.beforeEach(async (to, from, next) => {
  const auth = useAuthStore()

  // 白名单：登录页、404 页直接放行
  if (['/login', '/404'].includes(to.path)) return next()

  // 1) 未登录
  if (!auth.accessToken) {
    return next({ path: '/login', query: { redirect: to.fullPath } })
  }

  // 2) 已登录但没拉过用户信息（刷新页面场景）
  if (!auth.userInfo) {
    try {
      await auth.fetchUserInfo()      // 里面会 setPermissions
      await registerDynamicRoutes()   // 见 5.3
      return next({ ...to, replace: true })  // 关键：重新进入以命中新注册的路由
    } catch {
      auth.clearToken()
      return next('/login')
    }
  }

  // 3) 页面级角色校验
  if (to.meta.roles && !to.meta.roles.some(r => auth.userInfo.roles.includes(r))) {
    return next('/403')
  }

  next()
})
```

**注意第 2 步为什么要 `next({ ...to, replace: true })`**：因为你刚 `addRoute` 注册的路由，当前这次导航是查不到的，Vue Router 会走到"未匹配到的 catch-all"。必须**重新触发一次导航**，让路由表里的新路由被匹配到。这是"刷新页面白屏 / 404"的最常见原因。

### 5.3 动态路由（`addRoute`）：按角色注册

思路：**静态路由只放公共页面（登录、404、首页），业务路由登录后动态注册**。

```js
// router/index.js
const staticRoutes = [
  { path: '/login', component: () => import('@/views/Login.vue') },
  { path: '/', component: Layout, children: [{ path: '', component: Home }] },
  { path: '/:pathMatch(.*)*', component: NotFound }
]

export const router = createRouter({ history: createWebHistory(), routes: staticRoutes })
```

```js
// 后端返回的菜单/路由字符串 → 前端组件的映射
const asyncRoutesMap = {
  '/admin/user': () => import('@/views/admin/User.vue'),
  '/admin/role': () => import('@/views/admin/Role.vue'),
  '/order/list': () => import('@/views/order/List.vue')
}

export async function registerDynamicRoutes() {
  const auth = useAuthStore()
  const menus = await api.getMenus()       // 后端根据当前用户返回

  menus.forEach(menu => {
    const component = asyncRoutesMap[menu.path]
    if (!component) return                  // 后端返回了前端没有的页面，跳过
    router.addRoute('Layout', {             // 挂到 Layout 下作为子路由
      path: menu.path,
      component,
      meta: { title: menu.title, permissions: menu.permissions }
    })
  })

  auth.setMenus(menus)
}
```

**为什么用"字符串 → 组件"的映射，而不是让后端直接返回组件路径？**
> 因为 Vite / Webpack 的动态 `import()` 必须能被静态分析（Static Analysis）。`import(backendPath)` 会导致打包工具**打包整个 views 目录**，失去代码分割意义，还有安全风险。

### 5.4 一个常见的死循环坑

```js
router.beforeEach((to, from, next) => {
  if (!isLogin()) next('/login')   // ⚠️ 如果 /login 也没配白名单，会无限跳
})
```

规则：**任何"跳转到 X"的分支里，都要先判断当前 `to.path === X`，或者用白名单排除**。

## 六、React Router 路由守卫

React Router **没有 `beforeEach`** 这种全局钩子（哲学差异：React 更倾向于"用组件表达一切"）。守卫要通过**包装组件** + **`loader`（v6.4+）**来实现。

### 6.1 包装组件（Wrapper Component）方案

```jsx
// ProtectedRoute.jsx
function ProtectedRoute({ children, roles }) {
  const { user, loading } = useAuth()
  const location = useLocation()

  if (loading) return <Spinner />
  if (!user) {
    // 保留原目标，登录后跳回
    return <Navigate to="/login" state={{ from: location }} replace />
  }
  if (roles && !roles.some(r => user.roles.includes(r))) {
    return <Navigate to="/403" replace />
  }
  return children
}

// 路由表
<Routes>
  <Route path="/login" element={<Login />} />
  <Route path="/admin" element={
    <ProtectedRoute roles={['admin']}><AdminLayout /></ProtectedRoute>
  }>
    <Route path="user" element={<UserList />} />
  </Route>
</Routes>
```

登录页拿到 `location.state.from` 就能跳回原页面：

```jsx
const { state } = useLocation()
const navigate = useNavigate()
const onLogin = async () => {
  await login()
  navigate(state?.from?.pathname ?? '/', { replace: true })
}
```

### 6.2 Data Router（v6.4+）方案：`loader`

`loader` 在**路由匹配成功、组件渲染之前**执行，抛出 `redirect()` 就能拦截跳转，比包装组件更接近 Vue 的 `beforeEnter`。

```jsx
import { redirect, createBrowserRouter } from 'react-router-dom'

async function requireAuth(request) {
  const user = await fetchCurrentUser()
  if (!user) {
    const url = new URL(request.url)
    throw redirect(`/login?redirect=${encodeURIComponent(url.pathname)}`)
  }
  return user
}

const router = createBrowserRouter([
  {
    path: '/admin',
    loader: async ({ request }) => {
      const user = await requireAuth(request)
      if (!user.roles.includes('admin')) throw redirect('/403')
      return user
    },
    element: <AdminLayout />
  }
])
```

`loader` 的优势：
- **无闪烁**（No Flash）：组件根本不渲染就跳走了，用户看不到"先出现管理员页 → 又被踢回来"。
- 数据和守卫合一：登录校验和拉数据在同一个 `loader` 里，不需要 `useEffect` 里再拉一次。

### 6.3 全局守卫：`useNavigation` + Route Match

如果坚持要全局钩子的感觉，可以在最顶层 layout 里用 `useMatches()` 收集匹配到的所有路由的 `handle`，自己实现"守卫链"。多数项目不需要，包装组件 + `loader` 已经够用。

## 七、Next.js（App Router）路由守卫

Next.js 的场景比较特殊——**很多逻辑可以在服务端跑**，前端守卫的位置就多了几种。

### 7.1 `middleware.ts`：边缘运行时（Edge Runtime）级拦截

在 Vercel Edge 或 Node.js 中间件层执行，**请求还没到页面 / RSC（React Server Component）就被拦下**。适合登录校验、AB 测试、多语言重定向。

```ts
// middleware.ts
import { NextResponse } from 'next/server'

export function middleware(req) {
  const token = req.cookies.get('accessToken')?.value
  const { pathname } = req.nextUrl

  if (pathname.startsWith('/admin') && !token) {
    const url = req.nextUrl.clone()
    url.pathname = '/login'
    url.searchParams.set('redirect', pathname)
    return NextResponse.redirect(url)
  }
}

export const config = { matcher: ['/admin/:path*', '/order/:path*'] }
```

限制：middleware 跑在**边缘运行时**，不能用 Node.js 独有 API，也**不能访问数据库**（应该只做 Cookie/JWT 的轻量校验）。

### 7.2 Server Component 里直接校验

```tsx
// app/admin/layout.tsx
import { redirect } from 'next/navigation'
import { getCurrentUser } from '@/lib/auth'

export default async function AdminLayout({ children }) {
  const user = await getCurrentUser()  // 服务端可以直接查数据库
  if (!user) redirect('/login')
  if (!user.roles.includes('admin')) redirect('/403')
  return <>{children}</>
}
```

优势：能做**真正的权限校验**（因为跑在服务端），既是 UX 也是安全边界的一部分。middleware 做粗筛，layout 做细颗粒。

### 7.3 Client Component 里的守卫

如果页面是 `'use client'`，思路和 React Router 一样——包装组件或者 `useEffect` 里判断后 `router.replace('/login')`。这里更容易出现"闪一下再跳"，建议尽量把守卫上移到 layout / middleware。

## 八、按钮级权限

### 8.1 Vue 自定义指令 `v-permission`

```js
// directives/permission.js
import { useAuthStore } from '@/stores/auth'

export default {
  mounted(el, binding) {
    const auth = useAuthStore()
    const need = binding.value          // 字符串或数组
    const perms = Array.isArray(need) ? need : [need]
    const ok = perms.some(p => auth.permissions.includes(p))
    if (!ok) el.parentNode?.removeChild(el)   // 直接从 DOM 移除，比 v-if 更彻底
  }
}

// main.js
app.directive('permission', permissionDirective)
```

```html
<button v-permission="'user:delete'">删除</button>
<button v-permission="['order:export', 'order:admin']">导出</button>
```

**为什么用 `removeChild` 而不是 `display:none`？** 因为按钮在 DOM 里，用户改 CSS 就能看到，虽然点了接口也会 403，但"能看到"本身就是信息泄漏。

### 8.2 React `<Permission>` 组件 / Hook

```jsx
export function usePermission(need) {
  const { permissions } = useAuth()
  const list = Array.isArray(need) ? need : [need]
  return list.some(p => permissions.includes(p))
}

export function Permission({ need, fallback = null, children }) {
  return usePermission(need) ? children : fallback
}

// 用法
<Permission need="user:delete"><Button>删除</Button></Permission>
```

## 九、常见坑与排查清单

| 症状 | 原因 | 解决 |
|------|------|------|
| 刷新页面白屏 / 404 | 动态路由还没注册就触发了导航 | `beforeEach` 里注册后 `next({ ...to, replace: true })` |
| 无限重定向 | 白名单没放开登录页，或权限判断分支缺少 `to.path` 判断 | 加白名单 / 判断当前是不是目标页 |
| 登录后没跳回原页面 | 没保存 `redirect` query / `location.state.from` | 跳登录时带上，登录成功后读出来 |
| 菜单显示了但点进去 403 | 前端菜单和后端接口权限不一致 | 菜单也由后端返回，前后端同一份权限源 |
| 按钮 `v-if` 藏了但用户改 CSS 能看到 | 用了 `display:none` | 用指令直接 `removeChild` |
| 权限接口慢，页面先闪出来 | 守卫里没等异步完成 | 用 `loader`（React）/ `await` in `beforeEach`（Vue）/ Server Component（Next） |
| 多标签页登录状态不同步 | 一个标签退出了，另一个还是登录态 | `storage` 事件监听 localStorage 变化 |

---

## 十、一句话总结

> **路由守卫是三层结构：middleware/loader/beforeEach 拦「能不能进」，动态路由和菜单决定「能看到什么」，按钮级权限决定「能操作什么」；三层都只是 UX，接口鉴权才是真安全。**
