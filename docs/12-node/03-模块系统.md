# 03 — 模块系统

> 学习目标：理解 CommonJS 和 ES Module 的本质区别、require 的完整加载流程、以及工程中的常见配置。

---

## 1. 为什么需要模块系统

没有模块系统时，所有 JS 代码通过 `<script>` 标签引入，存在的问题：

- **全局作用域污染**：变量互相覆盖
- **依赖顺序依赖**：必须手动保证 script 标签的加载顺序
- **命名冲突**：不同文件不能有同名变量

模块系统解决了这三个问题：**每个文件是独立作用域，显式声明导入导出。**

---

## 2. CommonJS

Node.js 最早采用的模块规范，所以也叫 CJS。

### 2.1 语法

```js
// 导出
module.exports = { foo, bar };
exports.foo = foo;        // exports 是 module.exports 的引用

// 导入
const { foo, bar } = require('./module');
```

### 2.2 关键特性

- **同步加载**：`require` 是同步的，执行到它时模块必须已经加载完
- **运行时加载**：`require` 可以放在 `if` 里、函数里，动态决定加载什么
- **值拷贝**：导出的是值的**拷贝**，模块内部改变原始值，导入方拿不到新值

```js
// counter.js
let count = 0;
module.exports = { count, increment: () => { count++; } };

// main.js
const { count, increment } = require('./counter');
increment();
console.log(count); // 0 —— 拿到的是拷贝，原始值变了但拷贝不变
```

### 2.3 `exports` vs `module.exports`

```js
// ✅ 正确——修改 module.exports 的属性
exports.foo = 1;

// ❌ 错误——给 exports 重新赋值，切断了和 module.exports 的引用
exports = { foo: 1 };  // 无效！导出的是 module.exports 不是 exports

// ✅ 正确——替换整个导出对象
module.exports = { foo: 1 };
```

**原理：** `exports` 只是 `module.exports` 的一个引用。`require` 最终返回的是 `module.exports`。给 `exports` 重新赋值相当于把这个引用指向了新对象，原来的 `module.exports` 没变。

## 3. ES Module

JavaScript 官方标准（ES2015），Node.js 12+ 稳定支持。

### 3.1 语法

```js
// 命名导出
export const foo = 1;
export function bar() {}

// 默认导出
export default class MyClass {}

// 导入
import { foo, bar } from './module.js';
import MyClass from './module.js';
import * as all from './module.js';
```

### 3.2 关键特性

- **异步加载**：`import` 是异步的，支持顶层 await
- **编译时静态分析**：`import` 语句会被提升到模块顶部，不能放在 `if` 里
- **只读引用**：导出的是值的**只读引用**（live binding），模块内部变了导入方也能拿到

```js
// counter.js
export let count = 0;
export function increment() { count++; }

// main.js
import { count, increment } from './counter.js';
increment();
console.log(count); // 1 —— 引用，拿到了变化后的值
```

### 3.3 Node.js 中如何启用 ESM

| 方式 | 说明 |
| --- | --- |
| `.mjs` 扩展名 | 永远按 ESM 处理 |
| `.cjs` 扩展名 | 永远按 CJS 处理 |
| `package.json` 设 `"type": "module"` | `.js` 文件按 ESM 处理 |
| `package.json` 无 `type` 或 `"type": "commonjs"` | `.js` 文件按 CJS 处理 |

## 4. CommonJS vs ES Module 对比

| 维度 | CommonJS | ES Module |
| --- | --- | --- |
| 语法 | `require` / `module.exports` | `import` / `export` |
| 加载时机 | 运行时动态 | 编译时静态 |
| 加载方式 | 同步 | 异步 |
| 导出值 | 值的**拷贝** | 值的**只读引用**（live binding） |
| 顶层 this | `→ exports`（即 `{}`） | `→ undefined` |
| 动态导入 | `require` 本身支持 | `import()` 函数（返回 Promise） |
| 循环引用 | 可能拿不完整值 | 可能拿不完整值 |

**面试重点：值拷贝 vs 只读引用。** 给出 counter 的例子就能证明你理解。

## 5. `require` 的加载流程

```
require('xxx')
    │
    ▼
1. 路径解析
    ├── 是核心模块（fs/http/path）？ → 直接加载
    ├── 以 / ./ ../ 开头？ → 按路径加载
    └── 包名？ → node_modules 逐级查找
    │
    ▼
2. 缓存检查
    └── require.cache 中有？ → 直接返回缓存（不重新执行）
    │
    ▼
3. 文件定位
    └── 读 package.json → main 字段 → 入口文件
        若无 package.json → 尝试 index.js / index.json / index.node
    │
    ▼
4. 编译执行
    └── 包装成: (function(exports, require, module, __filename, __dirname) { … })
        然后执行
    │
    ▼
5. 返回 module.exports
```

### 5.1 模块缓存

```js
// a.js
console.log('a 被执行');
module.exports = { time: Date.now() };

// main.js
const a1 = require('./a'); // 打印 "a 被执行"
const a2 = require('./a'); // 不打印，直接返回缓存
console.log(a1 === a2);    // true —— 同一个对象

// 清除缓存
delete require.cache[require.resolve('./a')];
```

模块是**单例**的：同一个模块只会被执行一次，后续 `require` 拿到同一个缓存对象。

### 5.2 循环引用

```js
// a.js
console.log('a 开始');
exports.a = 1;
const b = require('./b');
console.log('a 结束，b.done =', b.done);

// b.js
console.log('b 开始');
exports.done = false;
const a = require('./a'); // a 还没执行完，拿到不完整的导出
console.log('b 中的 a =', a); // { a: 1 } —— 只有已执行的导出
exports.done = true;
console.log('b 结束');
```

循环引用时，`require` 拿到的是"到目前为止已执行的导出"，可能不完整。解决方案：**避免循环引用**，或把 `require` 放在函数内部延迟加载。

## 6. `node_modules` 查找规则

以 `require('lodash')` 为例，从当前文件所在目录开始：

```
当前文件: /project/src/utils/helper.js
查找路径:
  /project/src/utils/node_modules/lodash       ← 先查当前目录
  /project/src/node_modules/lodash             ← 查不到往上一级
  /project/node_modules/lodash                 ← 再往上一级
  /node_modules/lodash                         ← 直到根目录
```

这就是 `npm install` 的包不需要写完整路径的原因。

## 7. `package.json` 的模块相关字段

### `main`
包的入口文件，`require('pkg')` 时找到的就是这个。默认 `index.js`。

```json
{ "main": "lib/index.js" }
```

### `exports`（Node 12.7+）
更精确的导出控制，推荐替代 `main`：

```json
{
  "exports": {
    ".": {
      "require": "./lib/index.cjs",
      "import": "./lib/index.mjs"
    },
    "./helpers": "./lib/helpers.js"
  }
}
```

**关键区别：** `exports` 声明的路径之外，其他子路径无法被 `require`——更安全。`main` 没有这个限制。

### `type`
```json
{ "type": "module" }   // .js 文件按 ESM 处理
```
不设或 `"type": "commonjs"` 时，`.js` 文件按 CJS 处理。`.mjs` / `.cjs` 扩展名不受此字段影响。

## 8. `__dirname` 和 `__filename` 的 ESM 替代

CommonJS 自动注入：
```js
console.log(__dirname);   // 当前文件所在目录的绝对路径
console.log(__filename);  // 当前文件的绝对路径
```

ES Module 中不存在这两个变量，需要手动转换：

```js
import { fileURLToPath } from 'url';
import { dirname } from 'path';

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
```

Node 21+ 简化：
```js
console.log(import.meta.dirname);   // 直接可用
console.log(import.meta.filename);  // 直接可用
```

---

## 9. 快速回顾

1. CJS：同步、运行时、值拷贝；ESM：异步、编译时、只读引用
2. `require` 流程：路径解析 → 缓存检查 → 文件定位 → 编译执行 → 返回 exports
3. `module.exports` 和 `exports` 不是一回事——给 `exports` 赋新值无效
4. 模块是单例，缓存 key 是文件绝对路径
5. `node_modules` 逐级向上查找
6. `package.json` 中 `main` 定入口，`exports` 更精确更安全，`type` 控制模块规范
