# 08 — 错误处理与性能

> 学习目标：建立完善的错误处理体系，掌握内存泄漏排查方法和性能优化策略。

---

## 1. 错误处理体系

### 1.1 四层兜底

```
┌─────────────────────────────────────┐
│ ① try/catch + async/await           │  ← 业务代码层
├─────────────────────────────────────┤
│ ② 框架全局错误中间件                  │  ← 请求层兜底
├─────────────────────────────────────┤
│ ③ process.on('unhandledRejection')  │  ← Promise 未 catch
├─────────────────────────────────────┤
│ ④ process.on('uncaughtException')   │  ← 同步异常最后防线
│    （记日志 + process.exit(1)）       │     ⚠️ 不能恢复！
└─────────────────────────────────────┘
```

### 1.2 每一层的写法

**① 业务代码 try/catch：**

```js
// ❌ 没有错误处理
async function getUser(id) {
  const user = await db.query('SELECT * FROM users WHERE id = ?', [id]);
  return user;
}

// ✅ 正确
async function getUser(id) {
  try {
    const user = await db.query('SELECT * FROM users WHERE id = ?', [id]);
    if (!user) throw new NotFoundError('User not found');
    return user;
  } catch (err) {
    logger.error('getUser failed', { id, error: err.message });
    throw err; // 重新抛出，交给上层处理
  }
}
```

**② 框架全局错误中间件：**

```js
// Express 全局错误中间件（四个参数）
app.use((err, req, res, next) => {
  logger.error(err.message, { stack: err.stack, url: req.url });
  res.status(err.status || 500).json({
    error: process.env.NODE_ENV === 'production'
      ? 'Internal Server Error'   // 生产环境不暴露详细信息
      : err.message                // 开发环境返回具体错误
  });
});

// Koa
app.use(async (ctx, next) => {
  try {
    await next();
  } catch (err) {
    ctx.status = err.status || 500;
    ctx.body = { error: err.message };
    ctx.app.emit('error', err, ctx); // 触发 app 的 error 事件
  }
});
```

**③ & ④ 兜底事件：**

```js
// Promise reject 没有 catch 时触发
process.on('unhandledRejection', (reason, promise) => {
  logger.error('Unhandled Rejection', { reason });
  // Node 15+ 默认会终止进程，这里只需记日志
});

// 同步异常最后防线
process.on('uncaughtException', (err) => {
  logger.error('Uncaught Exception', { stack: err.stack });
  // ⚠️ 绝对不能 try/catch 继续跑！
  // 此时 Node.js 状态不可预测
  // 正确做法：记日志 → 优雅关闭 → 靠 PM2 自动重启
  server.close(() => {
    process.exit(1);
  });

  // 如果 server.close 在超时内未完成，强制退出
  setTimeout(() => process.exit(1), 10000);
});
```

### 1.3 为什么 `uncaughtException` 不能恢复

`uncaughtException` 发生时，错误发生在**不可预期的位置**。无法确定：
- 哪些变量已被修改
- 哪些回调还在队列中
- 内存是否已被破坏

此时的状态不可预测，继续运行可能导致更严重的问题（数据损坏、错误的响应）。

## 2. 内存管理

### 2.1 process.memoryUsage()

```js
const mem = process.memoryUsage();
console.log({
  heapTotal: `${(mem.heapTotal / 1024 / 1024).toFixed(2)} MB`,   // 堆总大小
  heapUsed: `${(mem.heapUsed / 1024 / 1024).toFixed(2)} MB`,     // 堆已使用
  rss: `${(mem.rss / 1024 / 1024).toFixed(2)} MB`,               // 常驻内存
  external: `${(mem.external / 1024 / 1024).toFixed(2)} MB`,     // Buffer 等堆外内存
});
```

- **heapUsed 持续上涨不下跌** → 内存泄漏
- **rss 远大于 heapTotal + external** → 可能有 Buffer 或原生模块泄漏

### 2.2 V8 垃圾回收

**分代回收：**

| 分代 | 默认大小 | 算法 | 特点 |
| --- | --- | --- | --- |
| 新生代（Young） | ~32MB（64位） | **Scavenge**（复制） | 快，存活对象少时会很快 |
| 老生代（Old） | ~1.4GB（64位） | **Mark-Sweep & Mark-Compact** | 慢，处理长期存活的对象 |

**GC 过程：**

```
新生代：
  From空间（有对象） → Scavenge → To空间（存活对象）
      ↓                            ↓
    死的丢弃                    活太久的晋升到老生代

老生代：
  Mark-Sweep：标记可达对象 → 清除不可达对象
  Mark-Compact（碎片严重时）：标记 → 移动对象整理内存
```

**GC 触发条件：**
- 新生代空间快满时 → Scavenge
- 老生代空间不足时 → Mark-Sweep
- `global.gc()` 手动触发（需 `--expose-gc` 启动参数）

### 2.3 `--max-old-space-size`

```bash
# 限制老生代堆大小（MB）
node --max-old-space-size=2048 app.js   # 2GB 上限
node --max-old-space-size=4096 app.js   # 4GB 上限
```

**作用：**
- 限制 Node.js 的堆内存上限
- 内存泄漏时能更快触发 OOM（而不是把服务器内存吃光）
- 大内存应用需要调大此值

## 3. 内存泄漏排查

### 3.1 确认泄漏

```js
// 定时打印内存，观察趋势
setInterval(() => {
  const { heapUsed } = process.memoryUsage();
  console.log(`${new Date().toISOString()} Heap: ${(heapUsed / 1024 / 1024).toFixed(2)} MB`);
}, 5000);

// 输出示例（有泄漏）：
// 14:00:00 Heap: 120 MB
// 14:00:05 Heap: 145 MB
// 14:00:10 Heap: 172 MB  ← 只涨不跌
// 14:00:15 Heap: 198 MB
```

### 3.2 抓 Heap Snapshot

**方法 1：Chrome DevTools**

```bash
node --inspect app.js
# 打开 chrome://inspect → 连接 → Memory 面板 → Take snapshot
```

**方法 2：heapdump 模块**

```js
const heapdump = require('heapdump');
// 在两个时间点分别 dump
heapdump.writeSnapshot('./snapshot-1.heapsnapshot');
// ... 等待一段时间
heapdump.writeSnapshot('./snapshot-2.heapsnapshot');
```

在 Chrome DevTools Memory 面板加载两个快照 → 选择 Comparison 视图 → 看哪些对象在增长。

### 3.3 四种常见泄漏原因

| 原因 | 示例 | 修复 |
| --- | --- | --- |
| **全局变量/闭包** | 把大对象挂在 `global` 上 | 不用时置 `null`，避免无意挂全局 |
| **EventEmitter 监听器** | 反复 `on` 但从不 `off` | 用完 `off` 或用 `once`，销毁时 `removeAllListeners` |
| **定时器未清除** | `setInterval` 忘了 `clearInterval` | 组件销毁时清除所有定时器 |
| **Stream 未销毁** | 创建了没 pipe 也没 destroy | 错误/结束时调用 `stream.destroy()` |

### 3.4 排查工具

| 工具 | 用途 |
| --- | --- |
| `node --inspect` + Chrome DevTools | Heap Snapshot、Allocation Timeline |
| `clinic doctor` | 一键诊断（CPU/内存/Event Loop 延迟） |
| `0x` / `flamegraph` | CPU 火焰图 |
| `heapdump` | 编程方式抓 Heap Snapshot |
| `process.memoryUsage()` | 代码内定时打印内存 |

## 4. 性能优化

### 4.1 优化方向

| 方向 | 策略 |
| --- | --- |
| **减少 I/O** | 合并查询、缓存热点数据（Redis）、批量写入 |
| **减少计算** | 避免在主线程做 CPU 密集任务、用 Worker Threads |
| **减少 GC 压力** | 避免频繁创建大对象、复用对象（对象池）、大文件用 Stream |
| **减少锁竞争** | Node.js 没有多线程锁，但要注意避免阻塞 Event Loop |

### 4.2 阻塞 Event Loop 的常见原因

```js
// ❌ 同步文件操作（CLI 可以，服务端不行）
const data = fs.readFileSync('big-file.txt');

// ❌ 大量正则计算
const result = hugeString.match(/complex-pattern/g);

// ❌ JSON 大对象序列化
JSON.stringify(veryLargeObject);

// ❌ 大循环
for (let i = 0; i < 1e9; i++) { /* CPU 密集 */ }
```

**检测 Event Loop 延迟：**

```js
const { monitorEventLoopDelay } = require('perf_hooks');
const h = monitorEventLoopDelay();
h.enable();
// ... 运行一段时间
console.log(h.mean / 1e6, 'ms'); // Event Loop 平均延迟
h.disable();
```

### 4.3 性能检查清单

- [ ] 所有文件 I/O 使用异步 API 或 Stream？
- [ ] 数据库查询有合理的索引和 limit？
- [ ] 静态资源有 CDN / 缓存头？
- [ ] 大 JSON 响应用了 Stream 而不是 `res.json(bigObject)`？
- [ ] 请求中有没有不必要的同步阻塞操作？
- [ ] 日志是否异步写入（不要同步写文件）？
- [ ] 是否合理设置了 `--max-old-space-size`？

## 6. 快速回顾

1. 错误处理四层：业务 try/catch → 中间件 → unhandledRejection → uncaughtException
2. `uncaughtException` **不能恢复**——记日志 + 优雅退出 + PM2 重启
3. V8 分代 GC：新生代 Scavenge（快），老生代 Mark-Sweep（慢）
4. 内存泄漏：确认 → 抓快照对比 → 四种常见原因
5. 性能优化核心：避免阻塞 Event Loop、减少 GC 压力、减少 I/O
