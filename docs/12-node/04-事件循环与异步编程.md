# 04 — 事件循环与异步编程

> 学习目标：搞懂谁在执行代码、任务怎么排队、以及为什么代码不是"从上到下逐行跑"的。

---

## 1. 先搞懂谁在干活：V8 + libuv

你写的 `app.js` 跑起来之后，是两个东西在配合工作：

```
你的 JS 代码
    │
    ▼
┌──────────────┐
│   V8 引擎     │  职责：解析 JS、执行 JS
│  (Google 造)  │  只管"跑代码"这一件事
└──────┬───────┘
       │  遇到 setTimeout / fs.readFile / http.request 这类 Node API
       │  V8 不认识这些，交给 libuv
       ▼
┌──────────────┐
│    libuv     │  职责：执行 Node API、管理线程池、调度任务
│  (Node 造)   │  它把不同的任务分配到不同的"阶段"去执行
└──────────────┘
```

-**V8**：负责执行你写的 JavaScript 代码
-**libuv**：负责执行 Node.js 的 API（定时器、文件读写、网络请求），并把回调安排到正确的时机执行

事件循环就是 libuv **调度这些回调的机制**。

---

## 2. 事件循环的核心规则（只有两条）

理解事件循环只需要记住两条规则：

> **规则一：** 回调分阶段执行。定时器回调在 timers 阶段执行，I/O 回调在 poll 阶段执行，setImmediate 回调在 check 阶段执行……
>
> **规则二：** 每个阶段之间，都会清空微任务队列（process.nextTick → Promise）。

画出来就是这样：

```
同步代码全部执行完
│
├── 清空微任务：process.nextTick → Promise
│
├── ┌─────────┐
│   │ timers  │  setTimeout / setInterval 到期的回调
│   └─────────┘
├── 清空微任务：process.nextTick → Promise
│
├── ┌───────────────────┐
│   │ pending callbacks │  上一轮 poll 中延迟的 I/O 回调（见下方解释）
│   └───────────────────┘
├── 清空微任务：process.nextTick → Promise
│
├── ┌───────────────┐
│   │ idle, prepare │  系统内部使用，跳过
│   └───────────────┘
├── 清空微任务：process.nextTick → Promise
│
├── ┌─────────┐     ┌─────────────────────────┐
│   │  poll   │◄────│  incoming: connections, │
│   │  ★核心  │     │  data, I/O 回调          │
│   └─────────┘     └─────────────────────────┘
├── 清空微任务：process.nextTick → Promise
│
├── ┌─────────┐
│   │ check   │  setImmediate 回调
│   └─────────┘
├── 清空微任务：process.nextTick → Promise
│
├── ┌───────────────────┐
│   │ close callbacks   │  socket.on('close') 等
│   └───────────────────┘
├── 清空微任务：process.nextTick → Promise
│
└── 回到 timers，继续下一轮循环
```

> **记住这个节奏：** 阶段 → 微任务 → 阶段 → 微任务 → 阶段 → 微任务……像个三明治，每个阶段中间夹一层微任务。

---

## 3. 每个阶段干了什么

**① timers 阶段**
执行 `setTimeout` 和 `setInterval` 中**已经到时间**的回调。不是精确准时的——poll 阶段如果忙，timers 可能延迟。

**② pending callbacks 阶段**
执行**上一轮 poll 阶段遗留下来的**特定 I/O 回调。

什么时候会产生？比如在 poll 阶段处理 TCP 连接时，操作系统返回了一个错误——这类错误回调不会在当前 poll 阶段立刻执行，而是被 libuv 推迟到**本轮的 pending callbacks 阶段**去处理。

你日常写的 `fs.readFile`、`http.request` 这些回调**都在 poll 阶段执行**，不经由 pending callbacks。只有系统层面的特定错误/事件回调才会走这里——所以你写业务代码时几乎感知不到它，但它确实在每一轮循环中运行。

**③ idle, prepare 阶段**
libuv 内部使用，开发者无法调用，不需要关心。

**④ poll 阶段（最重要）**
事件循环的"主战场"。做了两件事：
- 执行 I/O 回调（文件读取完成、网络请求返回等都是这里）
- 如果 poll 队列是空的：
  - 有 `setImmediate` → 去 check 阶段
  - 有到期的 timer → 回 timers 阶段
  - 都没有 → 在这里**等着**，直到新任务到来

> **等等，那不是永远不会结束？**
>
> 不是。libuv 内部维护了一个"活跃句柄计数"（active handles）——每当你创建 `http.createServer`、`fs.createReadStream`、`setTimeout` 时，计数 +1；当句柄关闭/销毁/回调执行完时，计数 -1。
>
> **当计数归零时，poll 阶段不会再等——事件循环直接退出，进程结束。**
>
> ```js
> console.log('hello');
> // 跑完这行，没有 server、没有 timer、没有打开的流
> // 活跃句柄 = 0 → poll 不等待 → 进程退出，打印 "hello" 后立即结束
> ```
>
> ```js
> http.createServer((req, res) => { res.end('ok'); }).listen(3000);
> // server 一直监听 → 活跃句柄 = 1 → poll 永远等待新连接 → 进程不会退出
> ```
>
> 这就是为什么服务器程序一直跑着不退出，而脚本跑完就退出的原因。

### 3.2 在 poll 阶段的回调里调了 setTimeout，会怎样？

这是一个很细的追问，但面试官可能拿它来试你"是不是真写过 Node.js"。

场景：你在 poll 阶段执行一个 I/O 回调（比如 `fs.readFile` 的回调），里面调了 `setTimeout(fn, 0)`。此时还在 poll 阶段，接下来怎么处理？

**回答分两步：**

**第一步：poll 先把手头的事干完。** 你注册的 `setTimeout` 不会中断当前 poll 阶段的执行——poll 先把队列里所有的 I/O 回调都执行完，一个不留。

**第二步：poll 干完后，判断下一步去哪。** 它的判断顺序是：

```
poll 队列清空后
    │
    ├── 有 setImmediate 回调？ → 去 check 阶段
    │
    ├── 有到期的 timer？ → 回 timers 阶段
    │
    └── 都没有 → 等着（或退出）
```

**关键细节：** `setTimeout(fn, 0)` 的最小延迟是 **1ms**，而 poll 阶段处理回调通常只要几微秒。所以当你刚注册完 `setTimeout(fn, 0)` 后、poll 去检查"有没有到期的 timer"时，**大概率那 1ms 还没到**——它不算"到期的 timer"。

所以实际走的分支是：poll 检查 → setImmediate？有的话去 check → check 跑完 → 回到 timers → 这时 1ms 早过了 → 执行 `setTimeout` 的回调。

**一句话总结：** poll 回调里注册的 `setTimeout(fn, 0)`，至少要等到下一轮 timers 才执行，不会在当前 poll 阶段执行。但 `setImmediate` 可以在当前轮的 check 阶段立即执行——这就是为什么 I/O 回调里 `setImmediate` 一定先于 `setTimeout(fn, 0)`。**

用一个代码验证：

```js
const fs = require('fs');

fs.readFile(__filename, () => {
  // 这个回调在 poll 阶段执行

  setTimeout(() => console.log('timeout'), 0);   // 注册定时器
  setImmediate(() => console.log('immediate'));    // 注册 immediate

  console.log('callback done');
});

// 输出：
//   callback done    ← poll 中的同步代码
//   immediate        ← poll 结束 → check 阶段 → setImmediate 先跑
//   timeout          ← check 结束 → 下轮 timers → setTimeout 才轮到
```

> 即使你写的是 `setTimeout(fn, 0)`，它的实际延迟也不是 0——Node.js 限制最小 1ms。这意味着它永远不会在当前轮的 timers 阶段执行（因为当前轮的 timers 阶段早就过了），要等下一轮。

**⑤ check 阶段**
执行 `setImmediate` 的回调。

**⑥ close callbacks 阶段**
执行关闭事件的回调，如 `socket.on('close')`、`stream.destroy()` 等。

### 3.1 为什么面试和教程里常说 pending callbacks "可以跳过"？

不是这个阶段不存在，而是**你写的业务代码永远不会走到这里**。

| 你在代码里写的 | 回调在哪个阶段执行 |
|---|---|
| `fs.readFile(file, cb)` | poll 阶段 |
| `setTimeout(fn, 0)` | timers 阶段 |
| `setImmediate(fn)` | check 阶段 |
| `http.createServer((req, res) => {})` | poll 阶段 |

pending callbacks 处理的是 libuv 内部的、系统层面的回调——比如上一轮 poll 中某个 TCP 句柄出错，错误回调被推迟到这里执行。你写的 `fs.readFile` 回调不会出现在这里。

**结论：** 面试时知道六个阶段的名字就够了，pending callbacks 不需要展开。但如果面试官追问"它到底是干嘛的"，答案就是——处理上一轮 poll 中延迟到本轮的特定系统回调，业务代码不经过这里。

## 4. 微任务：process.nextTick 和 Promise

微任务**不属于任何阶段**。它在**每个阶段切换时**清空。

```
一个阶段的回调全部执行完
    │
    ▼
清空所有 process.nextTick 回调（优先级最高）
    │
    ▼
清空所有 Promise.then / queueMicrotask 回调
    │
    ▼
进入下一个阶段
```

**清空的规则：清到队列为空为止。** 如果在执行 nextTick 时又注册了新的 nextTick，新加的也会被立刻清掉——直到队列真的空了，才进入下一步。

```js
process.nextTick(() => {
  console.log('A');
  process.nextTick(() => console.log('B')); // B 也会在本次清空中执行
});
process.nextTick(() => console.log('C'));

// 输出：A → C → B
// 不是 A → B → C，因为 nextTick 按队列 FIFO 顺序清空，
// 第二轮注册的 B 排在 C 后面
```

**`process.nextTick` 和 `Promise` 的优先级：**

```js
Promise.resolve().then(() => console.log(1));
process.nextTick(() => console.log(2));

// 输出：2 → 1
// nextTick 永远在 Promise 之前
```

## 5. setImmediate 和 setTimeout(fn, 0) 的区别

| | setTimeout(fn, 0) | setImmediate |
|---|---|---|
| 执行阶段 | timers 阶段 | check 阶段 |
| 实际延迟 | ≥ 1ms | 当前 poll 之后立即 |
| 在 I/O 回调中 | 下一轮的 timers | 当前轮的 check |

**在主模块里直接调用，谁先谁后不确定：**

```js
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));

// timeout 和 immediate 谁先谁后？不一定。
// 因为事件循环启动需要时间，启动快 setTimeout 还没到期就先执行 immediate
```

**在 I/O 回调里调用，immediate 一定先：**

```js
const fs = require('fs');

fs.readFile(__filename, () => {
  // I/O 回调在 poll 阶段执行
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
});

// 输出：
//   immediate  ← poll 结束 → check → setImmediate 先执行
//   timeout    ← 下一轮的 timers 才轮到
```

## 5. 浏览器事件循环

### 5.1 浏览器的事件循环结构

浏览器没有"六个阶段"的划分，而是按**宏任务（MacroTask）+ 微任务（MicroTask）**组织：

```
┌──────────────────────────────────────────────┐
│  1. 取一个宏任务执行                            │
│  2. 宏任务执行完毕                              │
│  3. 清空微任务队列（执行期间新加的微任务也一并清空）   │
│  4. 可能执行渲染（如果需要）                       │
│      ├── requestAnimationFrame 回调             │
│      ├── 重新计算样式（Style）                    │
│      ├── 布局（Layout）                         │
│      └── 绘制（Paint）                          │
│  5. 回到 1，取下一个宏任务                        │
└──────────────────────────────────────────────┘
```

**浏览器宏任务：** `setTimeout`、`setInterval`、`I/O`、`UI Rendering`、`postMessage`、`MessageChannel`

**浏览器微任务：** `Promise.then`、`MutationObserver`、`queueMicrotask`

### 5.2 经典执行顺序示例

#### 基础版

```js
console.log('1');

setTimeout(() => console.log('2'), 0);

Promise.resolve()
  .then(() => console.log('3'))
  .then(() => console.log('4'));

console.log('5');

// 输出：1 → 5 → 3 → 4 → 2
```

**为什么是这个顺序？**

事件循环的核心就是一句话：**同步代码先跑完，再清微任务，再取下一个宏任务。**

```
第一轮：整个 <script> 就是一个宏任务
  ├── 同步代码：console.log('1')        → 输出 1
  ├── 同步代码：setTimeout(...)          → 注册宏任务，扔进宏任务队列
  ├── 同步代码：Promise.resolve()
  │     .then(3)                        → 注册微任务，扔进微任务队列
  │     .then(4)                        → 还没注册！等第一个 then 执行后才加
  ├── 同步代码：console.log('5')        → 输出 5
  │
  ├── 当前宏任务结束，开始清空微任务队列：
  │   ├── 第一个 then：输出 3
  │   │   然后 .then(4) 注册到微任务队列 ← 此时追加
  │   └── 第二个 then：输出 4
  │       （微任务队列空了，本轮结束）
  │
  └── 取下一个宏任务：setTimeout 回调    → 输出 2
```

> **关键点：** `.then().then()` 链不是一次性全部注册。第一个 `.then` 执行后才会注册第二个 `.then`——但因为微任务会"持续清空直到队列为空"，所以 3 和 4 在同一轮全部执行完，不会中间插入宏任务。

---

#### 进阶版（async/await）

这道题你出过错——专门拆开每一步解释。

```js
async function async1() {
  console.log('async1 start');   // ③ → 同步
  await async2();                 // 调用 async2，然后暂停
  console.log('async1 end');     // ⑨ → 微任务
}

async function async2() {
  console.log('async2');          // ④ → 同步
  await Promise.resolve();        // 暂停，后面的代码变微任务
  console.log('async2 end');     // ⑦ → 微任务
}

console.log('script start');      // ① → 同步

setTimeout(function() {
    console.log('setTimeout');    // ⑩ → 宏任务
}, 0);

async1();                          // ② → 同步调用

new Promise(function(resolve) {
    console.log('promise1');       // ⑤ → 同步！（executor 同步执行）
    resolve();
}).then(function() {
    console.log('promise2');       // ⑧ → 微任务
});

console.log('script end');         // ⑥ → 同步

// 输出：
//   script start
//   async1 start
//   async2
//   promise1
//   script end
//   async2 end
//   async1 end
//   promise2
//   setTimeout
```

**逐行推演（这是你需要的部分）：**

```
┌──────┬─────────────────────────────────┬──────────────────────────────────────────┐
│ 步骤 │ 代码                            │ 发生了什么                                │
├──────┼─────────────────────────────────┼──────────────────────────────────────────┤
│  ①   │ console.log('script start')     │ 同步执行 → 输出 script start              │
│  ②   │ setTimeout(...)                 │ 回调扔进宏任务队列                        │
│  ③   │ async1() —— 函数调用            │ 同步进入 async1 函数体                     │
│  ④   │ console.log('async1 start')     │ 同步执行 → 输出 async1 start             │
│  ⑤   │ await async2() —— 调用 async2   │ 同步进入 async2 函数体                     │
│  ⑥   │ console.log('async2')           │ 同步执行 → 输出 async2                   │
│  ⑦   │ await Promise.resolve()         │ await = 后面的代码包装成 .then()            │
│      │                                 │ async2 的剩余代码 → 扔进微任务队列 ①       │
│      │                                 │ async2 返回一个 pending Promise            │
│  ⑧   │ （回到 async1 的 await）         │ async1 的剩余代码要等 async2 的 Promise     │
│      │                                 │ 但目前 async2 的 Promise 还是 pending       │
│      │                                 │ 所以 async1 的剩余代码 → 暂不注册！          │
│  ⑨   │ new Promise(executor)           │ ⚠️ executor 是同步执行的！                 │
│      │ console.log('promise1')         │ 同步执行 → 输出 promise1                  │
│      │ resolve()                       │ .then 回调 → 扔进微任务队列 ②              │
│  ⑩   │ console.log('script end')       │ 同步执行 → 输出 script end                │
├──────┼─────────────────────────────────┼──────────────────────────────────────────┤
│      │ 同步代码全部执行完毕               │ 开始清空微任务队列                         │
│      │                                 │ 当前微任务队列：[①async2 end, ②promise2]   │
├──────┼─────────────────────────────────┼──────────────────────────────────────────┤
│  ⑪   │ console.log('async2 end')       │ 取出微任务 ① 执行 → 输出 async2 end       │
│      │                                 │ async2 执行完毕，它的 Promise resolve 了    │
│      │                                 │ ★ async1 的 await async2() 终于等到了！      │
│      │                                 │ async1 剩余代码 → 追加到微任务队列 ③        │
│      │                                 │ 现在微任务队列：[②promise2, ③async1 end]   │
│  ⑫   │ console.log('promise2')         │ 取出微任务 ② 执行 → 输出 promise2          │
│  ⑬   │ console.log('async1 end')       │ 取出微任务 ③ 执行 → 输出 async1 end       │
│      │                                 │ 微任务队列空了                              │
├──────┼─────────────────────────────────┼──────────────────────────────────────────┤
│      │ 取下一个宏任务                    │                                          │
│  ⑭   │ console.log('setTimeout')       │ 输出 setTimeout                          │
└──────┴─────────────────────────────────┴──────────────────────────────────────────┘
```

**三个最容易出错的点：**

**陷阱 ①：`new Promise(fn)` 里的 `fn` 是同步执行的！**

```js
new Promise(function(resolve) {
    console.log('A');  // 同步！立刻执行
    resolve();
});
console.log('B');
// 输出：A → B（不是 B → A）
```

很多人以为"Promise 是异步的"所以 executor 也是异步——不是。`new Promise` 的 executor 参数是**同步调用**的。只有 `.then()` / `.catch()` 的回调才是微任务。

**陷阱 ②：`await` 的"暂停"不等于"后面的代码立刻进微任务队列"**

```js
await async2();        // async2() 返回一个 Promise
console.log('end');    // 等 async2 的 Promise resolve 了，这行才进微任务队列
```

`await` 后面的代码什么时候进微任务队列？**当被 await 的 Promise resolve 的时候。** 不是立刻。

在本题中：
- `await async2()` → async2 返回的 Promise 初始是 pending
- async2 在步骤 ⑪ 才执行完，Promise 才 resolve
- 所以 async1 的剩余代码（`async1 end`）在步骤 ⑪ 才追加到微任务队列
- 这导致它排在了 `promise2` 的后面

**陷阱 ③：`await Promise.resolve()` 后面的代码也是微任务**

```js
async function async2() {
  console.log('async2');          // 同步
  await Promise.resolve();        // ← 后面的代码 = 微任务
  console.log('async2 end');     // 微任务！不是同步！
}
```

`await` 的本质是 `.then()`：
```js
// 这两段等价
await Promise.resolve();
console.log('async2 end');

// ↓ 等价于 ↓

Promise.resolve().then(() => {
  console.log('async2 end');
});
```

所以 `async2 end` 在步骤 ⑪（微任务阶段）才输出，而不是在步骤 ⑥ 后面立刻输出。

### 5.3 Node.js vs 浏览器事件循环对比

| | Node.js | 浏览器 |
|---|---|---|
| 结构 | 六个阶段循环 | 宏任务 → 微任务 → 可能渲染 |
| 微任务时机 | 每个**阶段切换时**清空 | 每个**宏任务之后**清空 |
| 独有 API | `process.nextTick`、`setImmediate` | `requestAnimationFrame`、`MutationObserver` |
| 渲染管线 | 无关 | 微任务后、下个宏任务前（由浏览器调度，非每轮都渲染） |
| setTimeout 最小延迟 | ~1ms | ~4ms（嵌套超过 5 层后） |

**关键差异：**

1. **微任务清空时机不同。** Node.js 在一个阶段的宏任务全部执行完后、进入下一个阶段前清空微任务。浏览器在每个宏任务执行完后立刻清空微任务。
2. **任务队列粒度不同。** Node.js 按阶段把宏任务分类（timers 队列、poll 队列、check 队列），浏览器只有一个宏任务队列（按注册顺序执行）。
3. **独有 API。** `process.nextTick` 只有 Node.js 有；`requestAnimationFrame` 和 `MutationObserver` 只有浏览器有。
4. **渲染。** 浏览器事件循环要协调渲染管线，Node.js 不需要。

### 5.4 一道面试真题

```js
// 问：这段代码在浏览器和 Node.js 中输出是否相同？

console.log(1);

setTimeout(() => {
  console.log(2);
  Promise.resolve().then(() => console.log(3));
}, 0);

setTimeout(() => {
  console.log(4);
  Promise.resolve().then(() => console.log(5));
}, 0);

Promise.resolve().then(() => console.log(6));

console.log(7);

// 答案：相同。输出都是 1 → 7 → 6 → 2 → 3 → 4 → 5
//
// 如果加上 process.nextTick / rAF 就会不同，但纯 setTimeout + Promise 的行为一致。
// 这说明宏任务 + 微任务的嵌套关系在两种环境中是通用的。
```

## 6. 异步编程演进

Node.js 的异步编程经历了四个阶段：

### Callback（回调地狱）
```js
fs.readFile('a.txt', (err, dataA) => {
  if (err) return console.error(err);
  fs.readFile('b.txt', (err, dataB) => {
    if (err) return console.error(err);
    fs.readFile('c.txt', (err, dataC) => {
      // 回调地狱
    });
  });
});
```

### Promise
```js
function readFilePromise(path) {
  return new Promise((resolve, reject) => {
    fs.readFile(path, (err, data) => err ? reject(err) : resolve(data));
  });
}

readFilePromise('a.txt')
  .then(dataA => readFilePromise('b.txt'))
  .then(dataB => readFilePromise('c.txt'))
  .catch(console.error);
```

### async / await
```js
const fs = require('fs/promises');

async function readAll() {
  try {
    const dataA = await fs.readFile('a.txt');
    const dataB = await fs.readFile('b.txt');
    const dataC = await fs.readFile('c.txt');
    return [dataA, dataB, dataC];
  } catch (err) {
    console.error(err);
  }
}
```

**关键理解：** `await` 后面的代码相当于包在 `.then()` 里——它是微任务。

### EventEmitter / Stream
```js
const stream = fs.createReadStream('big-file.dat');
stream.on('data', chunk => { /* 处理每一块 */ });
stream.on('end', () => console.log('done'));
```

## 7. 常见陷阱

### 陷阱 1：`nextTick` 递归导致饿死

```js
// ❌ 永远不会进入事件循环的下一个阶段
function recursiveNextTick() {
  process.nextTick(recursiveNextTick);
}
recursiveNextTick();
```

### 陷阱 2：在 `nextTick` 中抛出的错误

```js
process.nextTick(() => { throw new Error('oops'); });
// 这个错误无法被外层的 try/catch 捕获
// 因为 nextTick 回调在调用栈清空后才执行
```

### 陷阱 3：`setTimeout(fn, 0)` 的真实延迟

`setTimeout(fn, 0)` 实际上不会是 0ms。Node.js 中最小延迟是 1ms（浏览器中是 4ms）。如果需要真的"尽快"执行，用 `setImmediate` 或 `process.nextTick`。

## 10. 快速回顾

1. **两个角色**：V8 执行 JS → libuv 执行 Node API + 调度回调
2. **核心节奏**：阶段 → 清微任务 → 阶段 → 清微任务 → ……（三明治结构）
3. 六个阶段只需记住三个：**timers**（定时器）、**poll**（I/O，核心）、**check**（setImmediate）
4. 微任务优先级：**nextTick > Promise**，且清空会清到队列空为止
5. I/O 回调中 `setImmediate` 一定先于 `setTimeout(fn, 0)`
6. 浏览器 vs Node.js：阶段循环 vs 宏/微配对；独有 API 不同；渲染有无
