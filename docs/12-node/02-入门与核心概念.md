# 02 — 入门与核心概念

> 学习目标：搞清楚 Node.js 到底是什么、底层怎么跑、为什么这么设计。

---

## 1. Node.js 是什么

Node.js 是一个基于 Chrome **V8 引擎**的 JavaScript **运行时环境**。

- **不是语言**：没有发明新语法，跑的就是标准 JavaScript
- **不是框架**：不像 Express/Koa 那样提供开发范式，它只负责"执行"
- **不是浏览器**：没有 DOM、没有 window，但有文件系统、网络、进程

一句话：**Node.js = 让 JavaScript 在服务器端运行的能力。**

---

## 2. V8 引擎

V8 是 Google 用 C++ 为 Chrome 开发的 JavaScript 引擎。

**工作流程：**

```
JS 源码 → 解析为 AST → 编译为字节码 → JIT 编译为机器码 → 执行
```

- **JIT（Just-In-Time）编译**：先快速编译执行，发现热点代码（频繁执行的函数）后再深度优化
- Node.js 直接内嵌 V8，JS 执行性能和 Chrome 一致
- V8 还负责**垃圾回收**（GC）——自动管理内存

> **类比：V8 是 Node.js 的心脏，负责把 JS 代码变成 CPU 能执行的机器指令。**

---

## 3. Libuv

Libuv 是 Node.js 的底层 C 库，最初为 Node.js 开发。

| 职责 | 说明 |
| --- | --- |
| **事件循环** | 调度所有异步任务，Node.js 的"调度中心" |
| **非阻塞 I/O** | 文件读写、网络请求走异步路径 |
| **线程池** | 默认 4 线程，处理文件 I/O、DNS 查询等阻塞操作 |

**平台适配：** libuv 封装了各操作系统的异步机制——Linux 用 epoll，macOS 用 kqueue，Windows 用 IOCP。上层 Node.js 代码不感知平台差异。

> **类比：libuv 是神经系统，负责协调所有 I/O 操作的调度。**

---

## 4. Node.js 底层架构

```
┌──────────────────────────────────────┐
│         Node.js 内置模块               │  ← JS 层
│   fs, http, path, stream, crypto …   │
├──────────────────────────────────────┤
│         Binding 层（C++）              │  ← 桥接层
├──────────────────┬───────────────────┤
│     V8 引擎       │      libuv        │  ← 底层 C/C++
│  （执行 JS+GC）    │  （事件循环+异步I/O）│
└──────────────────┴───────────────────┘
```

- **内置模块**：`require('fs')` 能直接用的那层，JavaScript 实现
- **Binding**：让 JS 代码能调用底层的 C++ 功能（如 `fs.readFile` → C++ 的文件操作）
- **V8 + libuv**：真正干活的两块 C/C++ 代码

---

## 5. 非阻塞 I/O

### 什么是 I/O

I/O（Input/Output）= 程序与外部世界的数据交换。包括：读文件、发网络请求、查数据库、读键盘输入等。

### 阻塞 vs 非阻塞

**阻塞 I/O（传统模型）：**

```
发起读文件请求 → [等待磁盘返回数据] → 拿到数据 → 继续执行
                 ↑ 这段时间 CPU 空闲，但代码动不了
```

**非阻塞 I/O（Node.js）：**

```
发起读文件请求 → 立即继续执行 → 磁盘返回数据后通过回调通知
                 ↑ 等待期间 CPU 可以处理其他请求
```

### 为什么这很重要

服务端大部分时间不是在"计算"，而是在"等待"——等数据库返回、等文件读取、等网络响应。非阻塞 I/O 让等待的时间也能利用起来处理其他请求。

### 底层实现

- **网络 I/O**：libuv 使用系统级的 epoll/kqueue/IOCP，操作系统直接通知"数据到了"
- **文件 I/O**：操作系统没有完美的异步文件 API，所以 libuv 用**线程池**模拟异步——把文件操作放到线程池中执行，完成后通知主线程

---

## 6. 事件驱动

事件驱动 = "先注册监听，事件发生时执行回调"。

**流程：**

```
1. 发起异步操作（如读文件），注册回调函数
2. 操作交给 libuv 处理
3. 主线程继续执行其他代码
4. 操作完成 → libuv 将回调放入事件队列
5. 事件循环检测到事件 → 执行回调
```

这和浏览器的事件模型本质相同——浏览器监听 click/scroll，Node.js 监听 I/O 完成/连接建立。

---

## 7. 单线程模型

```
┌───────────────────────────────┐
│     主线程（JS 执行）           │  你的 JS 代码都在这里跑
│     一个 Event Loop            │
└──────────┬────────────────────┘
           │ 把耗时 I/O 委托出去
           ▼
┌───────────────────────────────┐
│      libuv 线程池（4线程）      │  文件读写、DNS 查询等
└───────────────────────────────┘
```

**分两层理解：**

| 层 | 线程数 | 说明 |
| --- | --- | --- |
| JS 执行层 | 1 个 | 同一时间只有一段 JS 在跑，无锁问题 |
| I/O 处理层 | 多线程 | libuv 线程池 + 操作系统异步机制 |

**额外能力：**
- **Worker Threads**（Node 12+）：真正的多线程执行 JS，适合 CPU 密集型任务
- **child_process / cluster**：多进程，每个进程有独立的 V8 实例和 Event Loop

---

## 8. Node.js 和浏览器的区别

| 维度 | 浏览器 JS | Node.js |
| --- | --- | --- |
| 运行位置 | 浏览器 | 服务器 / 终端 |
| 全局对象 | `window` | `global` |
| DOM | ✅ | ❌ |
| 文件系统 | ❌（安全沙箱） | ✅ `fs` 模块 |
| 网络 | `fetch` / `XMLHttpRequest` | `http` / `https` / `net` |
| 进程控制 | ❌ | ✅ `process` 对象 |
| 模块系统 | ES Module | CJS + ESM |
| 事件类型 | 用户交互（click、scroll） | 系统事件（I/O 完成、连接） |
| 主要用途 | 页面交互 | 后端逻辑、工具脚本、CLI |

**关键理解：语法一样，API 不同。** 哪些 API 可用取决于运行环境提供了什么。

---

## 9. Node.js 适合 / 不适合什么

| 擅长 | 不擅长 |
| --- | --- |
| Web 服务 / API 网关 | CPU 密集型计算（视频转码、图像处理） |
| 实时通信（WebSocket） | 大量数学运算 |
| 命令行工具 / 自动化脚本 | 需要多线程共享内存的场景 |
| SSR 中间层 | |
| 代理服务器 | |
| 爬虫 / 数据流处理 | |

**CPU 密集任务的替代方案：** Worker Threads、child_process 拆到子进程、或者换更适合的语言（Rust/Go）处理后通过 API 调用。

---

## 10. 快速回顾

1. Node.js = V8 + libuv + 内置模块，让 JS 跑在服务端
2. 非阻塞 I/O 让等待时间也能利用起来
3. 事件驱动 = 注册回调 → 异步执行 → 事件通知 → 执行回调
4. JS 主线程单线程，但底层 libuv 有线程池 + Worker Threads 支持
5. 语法和浏览器一样，但 API 不同（fs/http/process vs DOM/fetch）
