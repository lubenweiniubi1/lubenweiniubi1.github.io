## 一、基础概念

### 1. Node.js 是什么？

> Node.js 是一个基于 Chrome V8 引擎的 JavaScript **运行时环境**。它让 JavaScript 不再局限于浏览器，可以在服务器端执行。核心优势是**非阻塞 I/O + 事件驱动模型**，这使得它在高并发场景下表现非常出色，特别适合 I/O 密集型的 Web 服务和 API 网关。

### 2. Node.js 和 JavaScript 是什么关系？和浏览器里的 JS 有什么区别？

> 两层关系。第一层：JavaScript 是**编程语言**，Node.js 是**运行时环境**——就像 Java 和 JVM 的关系。第二层：浏览器和 Node.js 是两种不同的运行时环境，语法完全相同，但提供的 API 不同。浏览器给了 `window`、DOM、`localStorage`，面向页面交互；Node.js 给了 `global`、`fs`、`http`、`process`，面向系统级操作。模块系统也不同——浏览器现代开发用 ES Module，Node.js 同时支持 CJS 和 ESM。

### 3. Node.js 的底层架构是什么样的？

> 分三层。最上层是**内置模块**（fs、http、path 等），用 JavaScript 编写。中间是 **Binding 层**（C++），桥接 JS 和底层。最底层两块——**V8 引擎**负责执行 JS 代码，**libuv** 负责事件循环和异步 I/O。这个分层设计让 Node.js 既有 JS 的开发效率，又有 C/C++ 的性能。

### 4. Node.js 的优缺点是什么？适合和不适合什么场景？

> **优点：** ① 非阻塞 I/O + 事件驱动，高并发下资源开销低；② 前后端统一语言（JavaScript），全栈成本低；③ npm 生态巨大，几乎任何需求都有现成方案；④ 开发效率高，从代码到运行的反馈循环很短。
>
> **缺点：** ① 不适合 CPU 密集型计算——单线程执行 JS，大量计算会阻塞事件循环；② 回调嵌套历史遗留问题（虽然 async/await 已解决）；③ 动态类型语言，大型项目维护成本较高。
>
> **擅长：** Web 服务、API 网关、实时通信（WebSocket）、SSR 中间层、CLI 工具、爬虫。
> **不擅长：** 视频/图片处理、大量数学运算、需要多线程共享内存的场景（此类场景用 Worker Threads 或换语言）。

## 二、核心机制

### 5. Node.js 是不是单线程？

> 答案分两层。**JS 执行层是单线程的**——同一时间只有一段 JS 在跑，无锁问题。但**底层不是**——libuv 维护了默认 4 线程的线程池处理 I/O，且 Node 12+ 有 Worker Threads 可以开多线程执行 JS。准确说法：JS 主线程是单线程，但 Node.js 运行时整体是多线程的。

| 展开 | JS 单线程 → libuv 线程池 → Worker Threads |
| 收束 | "主线程单线程，运行时整体多线程" |

### 6. 为什么 Node.js 适合处理高并发？

> 传统多线程模型每个请求一个线程，线程等 I/O 时被阻塞，大量并发时上下文切换开销巨大。Node.js 用**单线程 + 非阻塞 I/O + 事件循环**替代——遇到 I/O 交给 libuv 异步处理，主线程立刻回头处理下一个请求。I/O 完成后通过回调通知。这模式让一个线程能扛几万并发连接，资源开销远低于多线程方案。

### 7. 描述 Node.js 事件循环（Event Loop），和浏览器的事件循环有什么区别？

> ⭐ **最高频题，没有之一。通常会追问他俩的区别。**

**先讲谁在干活：**

> 代码跑起来后是两个东西配合：**V8 引擎**负责解析和执行 JS 代码；**libuv** 负责执行 Node API（定时器、文件读写、网络），并把回调安排到正确的时机执行。事件循环就是 libuv 调度这些回调的机制。

**Node.js 的核心规则：**

> 回调分阶段执行，**每个阶段之间清空微任务**。节奏是这样的：
>
> ```
> 同步代码跑完
>   ↓ 清微任务（nextTick → Promise）
> ① timers      ← setTimeout / setInterval
>   ↓ 清微任务
> ② pending     ← 系统内部 I/O（跳过）
>   ↓ 清微任务
> ③ idle        ← 系统内部（跳过）
>   ↓ 清微任务
> ④ poll ★核心  ← I/O 回调（读文件、网络请求等）
>   ↓ 清微任务
> ⑤ check       ← setImmediate
>   ↓ 清微任务
> ⑥ close       ← socket.on('close')
>   ↓ 清微任务
> 回到 ①，继续下一轮
> ```
>
> 你只需要关注三个：**timers**（定时器）、**poll**（I/O 回调，最重要）、**check**（setImmediate）。微任务是每个阶段之间的"夹层"——nextTick 优先级高于 Promise。

**浏览器的事件循环：**

> 浏览器没有"六个阶段"的划分，而是**一个宏任务 → 清空所有微任务 → 可能渲染 → 下一个宏任务**的循环。宏任务包括 `setTimeout`、`I/O`、`postMessage`；微任务包括 `Promise.then`、`MutationObserver`。

**核心区别：**

| | Node.js | 浏览器 |
|---|---|---|
| 结构 | 六个阶段循环 | 宏任务 → 微任务 → 渲染 |
| 微任务时机 | 每个**阶段切换时**清空 | 每个**宏任务之后**清空 |
| 独有 API | `process.nextTick`、`setImmediate` | `requestAnimationFrame`、`MutationObserver` |
| 渲染 | 无关 | 微任务之后、下个宏任务之前 |

> **关键记忆点：** Node.js = 阶段 × 微任务的三明治结构；浏览器 = 宏任务 + 微任务的配对结构。面试官追问区别时，重点讲微任务清空时机的不同和各自独有的 API。

| 段 | 内容 |
|---|---|
| 定义 | "V8 执行 JS + libuv 调度回调，阶段之间夹微任务" |
| 展开 | Node.js 六阶段流程图 → 浏览器宏/微任务 → 对比表 |
| 收束 | "三明治 vs 配对，独有 API 不同" |

### 8. `process.nextTick`、`setImmediate`、`setTimeout(fn, 0)` 的区别？

> ⭐ **必考题**

三者触发时机完全不同：

| API | 触发阶段 | 优先级 | 注意 |
| --- | --- | --- | --- |
| `process.nextTick` | 每个阶段切换时 | 最高（＞ Promise） | 微任务 |
| `Promise.then` | 每个阶段切换时 | 次高（＜ nextTick） | 微任务 |
| `setTimeout(fn,0)` | timers 阶段 | 中 | 实际延迟 ≥ 1ms |
| `setImmediate` | check 阶段 | 低 | poll 阶段后 |

> 同级调用栈中的执行顺序：**nextTick → Promise → setTimeout(fn,0) → setImmediate**。
>
> 特殊场景：在 poll 阶段内部调用 `setImmediate`，会在当轮的 check 阶段执行，比下轮 timers 的 `setTimeout(fn,0)` 更早。

| 段 | 内容 |
|---|---|
| 定义 | "触发时机完全不同" |
| 展开 | 逐一表格对比 + 执行顺序 + poll 阶段特殊场景 |
| 收束 | "记住顺序 + 特殊情况" |

### 9. 非阻塞 I/O 的底层原理？

> Node.js 的非阻塞 I/O 靠 **libuv** 实现。当代码发起 I/O 操作时，libuv 根据平台选择最优的异步方式——网络 I/O 用系统级的 epoll（Linux）/ kqueue（macOS）/ IOCP（Windows），文件 I/O 用线程池。操作被提交给系统后，主线程继续执行。I/O 完成后，系统通知 libuv，libuv 将回调放入事件循环的对应阶段执行。核心思想：**主线程不直接等 I/O，而是把等待开销转嫁给操作系统。**

| 段 | 内容 |
|---|---|
| 定义 | "libuv + 平台异步机制" |
| 展开 | 网络 I/O（epoll等）vs 文件 I/O（线程池）的不同路径 |
| 收束 | "等待开销转嫁给 OS" |

### 11. `require` 的加载机制是什么？模块缓存是什么？

> ⭐ **高频追问**

加载分四步：
>
> 1. **路径解析**：核心模块 → 相对/绝对路径 → 包名（node_modules 逐级查找）
> 2. **缓存检查**：查 `require.cache`，已加载则**直接返回缓存**，不重新执行
> 3. **包装执行**：文件内容包进 `(function(exports, require, module, __filename, __dirname) { … })` 并执行
> 4. **返回 module.exports**（注意不是 exports——`exports = xxx` 无效）
>
> 模块是**单例**——同一模块只执行一次。但循环引用时可能拿到不完整的导出值。

|---|---|
| 定义 | "四步：解析→缓存→执行→返回" |
| 展开 | 逐步展开 + 缓存机制 + exports 陷阱 |
| 收束 | "单例 + 循环引用风险" |

### 12. Node.js 的模块查找顺序是什么？

> 以 `require('lodash')` 为例，从当前文件所在目录开始：
>
> 1. 查是否是**核心模块**（fs、http、path 等）
> 2. 从当前目录的 `node_modules/lodash` 开始，逐级向上查
> 3. 找到后，读 `package.json` 的 `"main"` 字段确定入口文件
> 4. 如果没有 `package.json`，尝试 `index.js` / `index.json` / `index.node`
> 5. 如果当前目录没有，继续向上级目录查 `node_modules`，直到根目录
>
> 这就是为什么项目里 `npm install` 后不需要写完整路径。

### 13. `package.json` 中 `main`、`exports`、`type` 字段的作用？

> - **`main`**：指定包的入口文件，`require('pkg')` 时找到的就是这个文件。默认 `index.js`。
> - **`exports`**（Node 12.7+）：更精确的导出控制。可以指定不同条件（`require` vs `import`）导出不同文件，还能限制子路径访问（未在 exports 中声明的路径无法被 require）。
> - **`type`**：值为 `"module"` 时，`.js` 文件按 ES Module 处理；值为 `"commonjs"` 或不设时，按 CommonJS 处理。`.mjs` / `.cjs` 扩展名不受此影响。

| 展开 | main 的默认行为 → exports 的条件导出 → type 的影响范围 |
| 收束 | "exports 是未来趋势，更安全更灵活" |

### 14. `__dirname` 和 `__filename` 在 ES Module 中怎么替代？

> CommonJS 中每个模块自动注入 `__dirname` 和 `__filename`。ES Module 中没有这两个变量，需要用 `import.meta.url` 替代：
>
> ```js
> import { fileURLToPath } from 'url';
> import { dirname } from 'path';
>
> const __filename = fileURLToPath(import.meta.url);
> const __dirname = dirname(__filename);
> ```
>
> Node 21+ 可以直接用 `import.meta.dirname` 和 `import.meta.filename`，不再需要手动转换。

### 15. Buffer 是什么？什么时候用？

> Buffer 是 Node.js 处理**二进制数据**的类，**堆外内存分配**（不经过 V8 堆），基于 `ArrayBuffer`。常见场景：读文件（`fs.readFile()` 默认返回 Buffer）、TCP 流、编码转换（`Buffer.from(str).toString('base64')`）、文件上传接收。核心优势：绕过 V8 堆大小限制，处理大文件时不会触发频繁 GC。

### 16. Stream（流）是什么？pipe 和背压怎么理解？

> ⭐ **考察核心模块理解深度**

Stream 是流式数据抽象，分四种：**Readable / Writable / Duplex / Transform**。核心优势是内存友好——不把整个文件读进内存，分块处理。
>
> `pipe` 把可读流和可写流连起来。当可读速度快于可写时，数据在内存中堆积——**背压**。处理方式：可写流的 `write()` 返回 `false` 时暂停可读流，等 `drain` 事件后恢复。
>
> 实际例子：`fs.createReadStream → gzip → fs.createWriteStream`，文件再大内存也稳定。

| 展开 | pipe 机制 → 背压 → drain 事件 |
| 收束 | gzip 管道实际例子 |

### 17. EventEmitter 是什么？怎么手动实现一个？

> EventEmitter 是 Node.js **事件驱动**的核心实现（`events` 模块），发布-订阅模式。Node 中大部分能发事件的模块都继承自它。
>
> 手动实现的核心逻辑：
> ```js
> class EventEmitter {
>   constructor() { this._events = {}; }
>   on(event, listener) {
>     (this._events[event] ||= []).push(listener);
>     return this;
>   }
>   emit(event, ...args) {
>     this._events[event]?.forEach(fn => fn(...args));
>     return this;
>   }
>   once(event, listener) {
>     const wrapper = (...args) => { this.off(event, wrapper); listener(...args); };
>     return this.on(event, wrapper);
>   }
>   off(event, listener) {
>     this._events[event] = this._events[event]?.filter(fn => fn !== listener);
>     return this;
>   }
> }
> ```
> 面试时重点讲清楚 `once` 的实现思路（包装函数，触发后自删）。

| 段 | 内容 |
|---|---|
| 定义 | "发布-订阅模式的核心实现" |
| 展开 | 给出 on/emit/once/off 实现 + once 的包装思路 |
| 收束 | "once 是高频考点——包装函数自删" |

### 18. `fs` 模块的 API 有几种调用方式？怎么选？

> fs 模块有四套 API：
>
> | 方式 | 示例 | 适用场景 |
> | --- | --- | --- |
> | 同步 | `fs.readFileSync` | CLI 工具、初始化脚本 |
> | 回调 | `fs.readFile(path, cb)` | 老代码兼容 |
> | Promise | `fs.promises.readFile` | 新代码推荐 |
> | Stream | `fs.createReadStream` | 大文件 |

> 选择策略：**小文件用 Promise API，大文件用 Stream，初始化用 Sync，生产服务端永远不要用 Sync**——同步操作会阻塞事件循环。
> Node 20+ 支持 `const fs = require('fs/promises')` 的简写方式。

| 段 | 内容 |
|---|---|
| 定义 | "四套 API：同步、回调、Promise、Stream" |
| 展开 | 表格对比 + 选择策略 |

### 19. `http.createServer` 的请求处理流程是什么？

> 当客户端发起 HTTP 请求时：
>
> 1. Node.js 底层（net 模块）建立 TCP 连接
> 2. HTTP parser 解析请求报文，构建 `IncomingMessage`（req）和 `ServerResponse`（res）对象
> 3. 触发 `request` 事件，传入 req 和 res
> 4. `createServer` 的回调被调用，开始执行业务逻辑
> 5. 业务代码通过 `res.writeHead` / `res.end` 返回响应
>
> 关键理解：req 是 **Readable Stream**（可读请求体），res 是 **Writable Stream**（可写响应体）。这也是为什么可以直接 `pipe` 一个文件到 res。

| 收束 | "stream 本质让 pipe 成为可能" |

### 20. Express 和 Koa 中间件的洋葱模型是什么？

> ⭐ **后端面试必问**

中间件是函数链。Express 中 `(req, res, next)` 通过调用 `next()` 将控制权交给下个中间件。Koa 中是 `async (ctx, next)`，`await next()` **之前**的代码在请求阶段执行，**之后**的代码在响应阶段执行。
>
> 洋葱模型的价值：**请求前后逻辑分离**。比如鉴权中间件——进时验证 token，出时记录耗时，两块逻辑在一个函数里但执行时机不同。
>
> Koa 的实现更优雅——不需要手动区分同步异步，async/await 天然支持洋葱模型。

| 段 | 内容 |
|---|---|
| 定义 | "函数链 + next() 传递控制权" |
| 展开 | Express 模型 → Koa async/await → 洋葱模型图解 |
| 收束 | "请求前后逻辑分离是关键价值" |

### 21. 怎么自己实现一个简单的中间件引擎？（compose 函数）

> ⭐ **高频手写题**

Koa 的 `koa-compose` 核心实现：
>
> ```js
> function compose(middlewares) {
>   return function (ctx) {
>     let index = -1;
>     function dispatch(i) {
>       if (i <= index) return Promise.reject(new Error('next() called multiple times'));
>       index = i;
>       const fn = middlewares[i];
>       if (!fn) return Promise.resolve();
>       try {
>         return Promise.resolve(fn(ctx, () => dispatch(i + 1)));
>       } catch (err) {
>         return Promise.reject(err);
>       }
>     }
>     return dispatch(0);
>   };
> }
> ```
>
> 关键点：递归调用 `dispatch` 实现洋葱模型、`i <= index` 防止多次调用 `next()`、返回 Promise 保证异步中间件兼容。

| 段 | 内容 |
|---|---|
| 定义 | "递归 dispatch 实现洋葱模型" |
| 展开 | 核心代码 + 三个关键点 |
| 收束 | "compose 是 Koa 的灵魂" |

### 22. 跨域（CORS）在 Node.js 中怎么处理？

>

### 23. cookie、session、JWT 三种鉴权方案怎么选？

> | 方案 | 原理 | 优点 | 缺点 |
> | --- | --- | --- | --- |
> | **Cookie** | 浏览器自动携带，服务端直接读 | 简单，浏览器原生支持 | 容量小（4KB），CSRF 风险 |

> **实际选择**：传统 Web 应用用 Session + Cookie；前后端分离 / 移动端 / 微服务用 JWT；需要即时失效的场景（如支付）用 Session 或 JWT + Redis 黑名单。

## 六、进程与部署

### 24. cluster 模块的原理？PM2 做了什么？

> ⭐ **高频场景题**

`cluster`：主进程 fork 多个子进程（Worker），共享一个端口。主进程通过 IPC 或 Round-robin 分发连接。每个子进程是独立 V8 实例，有独立的 Event Loop。
>
> PM2 基于 cluster，额外提供：进程守护（挂了自动重启）、零停机 reload（逐个重启 Worker）、日志管理、多环境配置。`pm2 start app.js -i max` 按 CPU 核心数启动。
>
> 核心限制：**进程间不共享内存**，通信靠 IPC 或 Redis 等外部存储。

| 定义 | "cluster 多进程 + PM2 进程管理" |
| 展开 | cluster 原理 → PM2 增值功能 → 内存不共享 |
| 收束 | "IPC 或 Redis 解决进程间通信" |

### 25. `child_process` 的 `fork`、`spawn`、`exec` 有什么区别？

> | 方法 | 用途 | 特点 |
> | --- | --- | --- |
> | `spawn` | 执行命令，流式输出 | 返回 Stream，适合长时间运行、大量输出 |
> | `exec` | 执行命令，Buffer 输出 | 有 maxBuffer 限制（默认 1MB），适合短命令 |

> **选择策略：** 调用其他 Node 脚本用 `fork`（默认 IPC）；长时间运行、输出大的命令用 `spawn`；简单的 shell 命令用 `exec`。

### 26. 子进程间怎么通信（IPC）？

> - **`fork` 自带 IPC**：父进程 `child.send(msg)`，子进程 `process.on('message')`。底层是 JSON 序列化 + 管道传输。
> - **`spawn` / `exec`**：通过 `stdio` 管道（stdin/stdout）传递数据。
> - **多进程间共享**：cluster 的 Worker 间不能直接通信，常用方案是 Redis Pub/Sub、进程间消息转发（Worker → Master → 另一个 Worker）。

### 28. 如何排查 Node.js 内存泄漏？

> ⭐ **考察线上排障能力**

三步排查法：
>
> 1. **确认泄漏**：定时打 `process.memoryUsage().heapUsed`，看内存是否只涨不跌。或用 `--max-old-space-size` 限制堆大小触发 OOM。
> 2. **抓 Heap Snapshot**：用 Chrome DevTools Memory 面板或 `heapdump` 模块，不同时间点抓两份快照，对比增量。
> 3. **定位原因**——四个常见元凶：
>    - 闭包/全局变量持有大对象
>    - EventEmitter 监听器未移除（`setMaxListeners` 警告是信号）
>    - 定时器未清除
>    - Stream 未 destroy
>
> 生产环境用 `clinic doctor` 或 `--inspect` + Chrome DevTools 远程调试。

### 29. `uncaughtException` 和 `unhandledRejection` 的区别？

> - **`uncaughtException`**：同步代码或 async 函数中未被 try/catch 捕获的错误。**不应在回调里恢复进程**——此时 Node 状态不可预测，正确做法是记录日志 + `process.exit(1)`，靠 PM2 自动重启。
> - **`unhandledRejection`**：Promise reject 后没有 `.catch` 处理。Node 15+ 默认行为是**直接终止进程**（之前只打印警告）。
>
> 正确的多层兜底：业务层 try/catch → 框架全局错误中间件 → `unhandledRejection` → `uncaughtException`（仅日志 + 退出）。

| 段 | 内容 |
|---|---|
| 定义 | "两个事件各自的触发条件" |
| 展开 | 为什么不能恢复 + Node 15+ 行为变化 |
| 收束 | "四层兜底，最后一层只记日志然后退出" |

### 30. V8 的垃圾回收机制是什么？怎么优化？

> V8 使用**分代回收**：
>
> | 分代 | 大小 | 算法 | 特点 |
> | --- | --- | --- | --- |
> | 新生代 | ~32MB | Scavenge（复制） | 快，存活对象少 |
> | 老生代 | ~1.4GB | Mark-Sweep & Mark-Compact | 慢，存活对象多 |
>
> 优化策略：
> - 避免频繁创建短生命周期的大对象（触发频繁 GC）
> - 大文件用 Stream 而不是 `readFileSync`（避免一次性占满堆）
> - 合理设 `--max-old-space-size`（默认 ~1.4GB，大应用可能需要调大）
> - 避免内存泄漏（见上题）
> - 用 `--expose-gc` + `global.gc()` 在开发环境手动触发 GC 做测试

| 段 | 内容 |
|---|---|
| 定义 | "分代回收：新生代 Scavenge + 老生代 Mark-Sweep" |
| 展开 | 表格对比 + 五条优化建议 |
| 收束 | "用 Stream、控对象生命周期、合理设堆上限" |
