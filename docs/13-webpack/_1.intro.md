## 一、早期困境：无模块化时代（2009年之前）

### 1.1 全局变量污染问题
```javascript
// 早期开发方式
var username = '张三';
var password = '123456';

function login() {
    console.log(username + '登录');
}

function logout() {
    console.log(username + '登出');
}
```
**问题分析**：
- 所有变量都挂在`window`对象上
- 多个文件同时使用`username`变量会导致覆盖
- 代码量大时命名冲突频发

### 1.2 依赖管理混乱
```html
<!-- 依赖顺序必须手动维护 -->
<script src="jquery.js"></script>
<script src="plugin1.js"></script> <!-- 依赖jquery -->
<script src="plugin2.js"></script> <!-- 依赖plugin1 -->
<script src="app.js"></script>      <!-- 依赖plugin2 -->
```
**痛点**：
- 依赖关系靠注释说明
- 加载顺序错误导致运行时错误
- 无法实现按需加载

## 二、初级解决方案（2005-2009年）

### 2.1 命名空间模式（Namespace）
```javascript
// 基础命名空间
var MyProject = {};
MyProject.ModuleA = {
    data: '数据',
    method: function() {}
};

// 嵌套命名空间
var MyProject = MyProject || {};
MyProject.Utils = MyProject.Utils || {};
MyProject.Utils.String = {
    trim: function(str) {
        return str.trim();
    }
};
```
**优缺点**：
- ✅ 减少全局变量数量
- ❌ 本质仍是全局变量，只是层级更深
- ❌ 无法真正实现私有成员

### 2.2 立即执行函数表达式（IIFE）
```javascript
// 基础IIFE
(function() {
    var privateVar = '私有变量';
    function privateFunc() {
        console.log(privateVar);
    }
    window.publicVar = '公开变量';
})();

// 带参数的IIFE
(function($) {
    $.fn.myPlugin = function() {
        // 插件代码
    };
})(jQuery);

// 模块模式
var Counter = (function() {
    var count = 0;
    return {
        increment: function() {
            return ++count;
        },
        decrement: function() {
            return --count;
        },
        getCount: function() {
            return count;
        }
    };
})();
```
**核心原理**：
- 利用闭包实现私有作用域
- 通过`return`暴露公共接口
- 解决了变量污染问题

## 三、模块化规范爆发期（2009-2015年）

### 3.1 CommonJS（2009）- Node.js的模块化标准

#### 3.1.1 历史背景
- 2009年Node.js诞生，需要服务端模块化方案
- Ryan Dahl参考Python的`import`机制设计
- 采用同步加载，适合文件系统环境

#### 3.1.2 核心语法
```javascript
// module.js
// 单个导出
exports.name = '模块名';
exports.func = function() {};

// 整体导出
module.exports = {
    name: '模块名',
    func: function() {}
};

// 导入
var module = require('./module.js');
console.log(module.name);
```

#### 3.1.3 执行机制

node 只会在启动的时候同步加载模块，执行的时候是不需要的。浏览器不是，大量同步请求出现会阻塞。
```javascript
// require的执行过程
function require(moduleName) {
    // 1. 解析路径
    var resolvedPath = path.resolve(moduleName);
    
    // 2. 检查缓存
    if (Module._cache[resolvedPath]) {
        return Module._cache[resolvedPath].exports;
    }
    
    // 3. 创建模块对象
    var module = new Module(resolvedPath);
    
    // 4. 加载并执行
    module.load();
    
    // 5. 缓存模块
    Module._cache[resolvedPath] = module;
    
    return module.exports;
}
```

#### 3.1.4 循环依赖处理

CommonJS 通过**模块缓存**机制处理循环依赖。当 `require` 遇到循环引用时，不会无限递归，而是**直接返回当前已执行部分的 `exports` 对象**（一个"半成品"）。

```javascript
// a.js
console.log('a.js 开始执行');
exports.done = false;
var b = require('./b.js');
console.log('在a中，b.done = ' + b.done);
exports.done = true;
console.log('a.js 执行完毕');

// b.js
console.log('b.js 开始执行');
exports.done = false;
var a = require('./a.js');
console.log('在b中，a.done = ' + a.done);
exports.done = true;
console.log('b.js 执行完毕');

// c.js（入口文件）
console.log('c.js 开始执行');
var a = require('./a.js');
var b = require('./b.js');
console.log('在c中，a.done = ' + a.done);
console.log('在c中，b.done = ' + b.done);
```

**执行流程分析（以 `node c.js` 为例）：**

```
1. require('./a.js') → a 模块不在缓存中，创建模块对象，开始执行 a.js
   │
   ├─ console.log → "a.js 开始执行"
   ├─ exports.done = false
   ├─ require('./b.js') → b 模块不在缓存中，创建模块对象，开始执行 b.js
   │   │
   │   ├─ console.log → "b.js 开始执行"
   │   ├─ exports.done = false
   │   ├─ require('./a.js') → ⚠️ a 模块已经在缓存中了！
   │   │   但 a 只执行到一半（exports.done 还是 false）
   │   │   直接返回缓存的 a.exports → { done: false }
   │   ├─ console.log → "在b中，a.done = false"
   │   ├─ exports.done = true
   │   └─ console.log → "b.js 执行完毕"
   │   b 模块完成，b.exports = { done: true }
   │
   ├─ console.log → "在a中，b.done = true"  ← b 此时已经完整
   ├─ exports.done = true                    ← a 的 done 被更新为 true
   └─ console.log → "a.js 执行完毕"
   a 模块完成，a.exports = { done: true }

2. require('./b.js') → b 已在缓存中，直接返回 { done: true }

3. console.log → "在c中，a.done = true"
4. console.log → "在c中，b.done = true"
```

**实际输出：**
```
c.js 开始执行
a.js 开始执行
b.js 开始执行
在b中，a.done = false
b.js 执行完毕
在a中，b.done = true
a.js 执行完毕
在c中，a.done = true
在c中，b.done = true
```

**关键要点：**
- `b.js` 中拿到的 `a` 是一个**不完整的"半成品"**——`a.done` 还是 `false`
- `c.js` 中拿到的 `a` 才是**最终结果**——`a.done` 已经是 `true`
- 核心原理：`require` 先检查缓存，缓存命中直接返回，**不管模块是否执行完毕**
- **避免踩坑**：不要在模块顶层依赖循环引用的值来做条件判断，应该把 `require` 放到函数内部延迟执行，或者把对导入值的访问也放到函数内部

#### 3.1.5 优点与局限
- ✅ 语法简洁，符合直觉
- ✅ 服务端同步加载效率高
- ❌ 浏览器端无法直接使用（需要打包工具）
- ❌ 无法实现异步加载

### 3.2 AMD（Asynchronous Module Definition）- 浏览器端异步方案

#### 3.2.1 历史背景
- 2009年James Burke在RequireJS中提出
- 解决浏览器端异步加载问题：先异步加载所有依赖，再执行回调

#### 3.2.2 核心语法
```javascript
// 定义模块：依赖前置——先声明所有依赖，加载完成后才执行工厂函数
define(['jquery'], function($) {
    return {
        init: function() {
            $('body').append('<div>Hello</div>');
        }
    };
});

// 使用模块
require(['module1'], function(module1) {
    module1.init();
});
```

#### 3.2.3 优点与局限
- ✅ 异步加载，天然适合浏览器
- ✅ 依赖前置，执行前就能知道完整依赖图
- ❌ 语法繁琐，依赖数组和回调参数必须一一对应
- ❌ 条件分支中用不到的依赖也会提前加载（浪费请求）

### 3.3 CMD（Common Module Definition）- 国产优化方案

#### 3.3.1 历史背景
- 2011年玉伯（支付宝）在Sea.js中提出
- 针对AMD的改进：**依赖就近**——用到时才 `require`，而不是提前全部声明

#### 3.3.2 核心语法
```javascript
// 定义模块：依赖就近——require 写在用到的位置
define(function(require, exports, module) {
    var $ = require('jquery');  // 用的时候才引入
    
    exports.init = function() {
        console.log($);
    };
});

// 使用模块
seajs.use(['module1'], function(module1) {
    module1.init();
});
```

#### 3.3.3 与AMD的核心区别

| 特性 | AMD | CMD |
|------|-----|-----|
| 依赖时机 | 依赖前置，提前加载 | 依赖就近，用到才加载 |
| 语法风格 | `define(['dep'], function(dep) {})` | `define(function(require) { ... })` |
| 条件分支 | 不用的依赖也会加载 | 不走到分支就不加载 |

#### 3.3.4 优点与局限
- ✅ 依赖就近，按需加载，更符合直觉
- ❌ Sea.js 生态已停止维护，已被 ESM 全面取代

### 3.4 UMD（Universal Module Definition）- 通用模块定义

#### 3.4.1 设计理念
- 兼容AMD、CommonJS和全局变量
- 让模块可以在多种环境中使用
- 适合开发第三方库

#### 3.4.2 核心实现
```javascript
(function(root, factory) {
    if (typeof define === 'function' && define.amd) {
        // AMD环境
        define(['jquery'], factory);
    } else if (typeof exports === 'object') {
        // CommonJS环境
        module.exports = factory(require('jquery'));
    } else {
        // 浏览器全局变量
        root.MyLibrary = factory(root.jQuery);
    }
}(this, function($) {
    // 模块实现
    var MyLibrary = {
        version: '1.0.0',
        init: function() {
            console.log('初始化');
        }
    };
    return MyLibrary;
}));
```

#### 3.4.3 简化版本
```javascript
(function(root, factory) {
    if (typeof module === 'object' && module.exports) {
        module.exports = factory();
    } else if (typeof define === 'function' && define.amd) {
        define(factory);
    } else {
        root.MyLibrary = factory();
    }
}(this, function() {
    return {
        name: 'MyLibrary'
    };
}));
```

## 四、标准化时代：ES6 Module（2015年至今）

### 4.1 历史背景
- 2015年ES6（ES2015）正式发布，JavaScript 语言原生支持模块化
- 统一了浏览器和服务器的模块标准

### 4.2 核心语法
```javascript
// 导出
export const name = '模块名';
export function func() {}
export default function() { console.log('默认导出'); }

// 导入
import myDefault, { name, func } from './module.js';
import * as myModule from './module.js';

// 动态导入（运行时按需加载）
const module = await import('./module.js');
```

### 4.3 核心特性一览

| 特性 | 说明 |
|------|------|
| **静态分析** | 编译时确定依赖关系 → 支持 Tree Shaking、Scope Hoisting |
| **值的绑定** | `import` 拿到的是实时引用，export 值变化后 import 方自动更新（对比 CJS 拿到的是副本） |
| **循环依赖** | 通过"构建→实例化→求值"三阶段处理，函数声明在实例化阶段即提升初始化（注意 `export const fn = () => {}` 会因 TDZ 报错） |
| **单例** | 同一个模块不管被 import 多少次，只执行一次 |

> 📎 详细内容见 [02-esm.md](./02-esm.md)

## 六、各规范对比总结

### 6.1 核心特性对比

| 特性 | CommonJS | AMD | CMD | ES6 Module |
|------|----------|-----|-----|------------|
| 加载方式 | 同步 | 异步 | 异步 | 静态/动态 |
| 适用环境 | Node.js | 浏览器 | 浏览器 | 全平台 |
| 语法简洁度 | 中等 | 复杂 | 简洁 | 简洁 |
| 静态分析 | ❌ | ⚠️ | ⚠️ | ✅ |
| Tree Shaking | ❌ | ❌ | ❌ | ✅ |
| 循环依赖 | ✅ | ✅ | ✅ | ✅ |
| 原生支持 | Node.js | ❌ | ❌ | ✅ |

### 6.2 使用场景建议

- **Node.js后端开发**：CommonJS（或ESM）
- **现代前端开发**：ES6 Module + 构建工具
- **老旧项目维护**：根据现有技术栈选择
- **第三方库开发**：UMD（兼容多种环境）
