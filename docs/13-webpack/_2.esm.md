# ES Module 核心特性详解

ES Module（ESM）是 JavaScript 官方的模块化标准（ES2015）。本文由浅入深梳理 ESM 的核心特性。

---

## 一、基础入门

### 1.1 基本语法

```javascript
// ========== 导出 ==========
// 命名导出
export const name = '模块名';
export function func() {}
export class MyClass {}

// 默认导出（一个模块只能有一个）
export default function() { console.log('默认导出'); }

// 重命名导出
export { name as myName };

// 重新导出（聚合模块）
export { default as OtherDefault } from './other.js';
export * from './other.js';

// ========== 导入 ==========
// 命名导入
import { name, func } from './module.js';
import { name as myName } from './module.js';    // 重命名

// 默认导入
import myDefault from './module.js';

// 混合导入
import myDefault, { name, func } from './module.js';

// 命名空间导入（导入全部）
import * as myModule from './module.js';
console.log(myModule.name);

// 仅执行（不导入任何内容，如 CSS 或 polyfill）
import './module.js';
```

### 1.2 在浏览器中使用

```html
<!-- 外部模块 -->
<script type="module" src="./app.js"></script>

<!-- 内联模块 -->
<script type="module">
    import { func } from './module.js';
    func();
</script>

<!-- 旧浏览器降级 -->
<script type="module" src="./app.js"></script>
<script nomodule src="./app-legacy.js"></script>
```

### 1.3 在 Node.js 中使用

```javascript
// 方式1：文件扩展名改为 .mjs
// app.mjs
import { func } from './module.js';

// 方式2：package.json 中声明 type
// package.json
{
    "type": "module"
    // 所有 .js 文件将被视为 ESM
}

// CommonJS 中想用 ESM → 两种方式

// 方式 A：动态 import()（所有 Node.js 版本通用）
async function loadESM() {
    const esmModule = await import('./esm-module.js');
    console.log(esmModule.namedExport);
}

// 方式 B：require()（Node.js 22+ 原生支持）
// 注意：ESM 模块不能有顶层 await，否则 require() 会报错
const esmModule = require('./esm-module.js');
console.log(esmModule.default);     // export default → .default
console.log(esmModule.namedExport); // export const → 直接属性
```

## 二、与普通脚本的差异

### 2.1 严格模式 — 自动启用

ESM 文件**自动处于严格模式**，无需手动写 `'use strict'`。

```javascript
// ESM 中自动生效
export function test() {
    console.log(this); // undefined（非严格模式下是 window）

    undeclaredVar = 10;  // ❌ ReferenceError
    var x = 10;
    delete x;            // ❌ SyntaxError
    var interface = 10;  // ❌ SyntaxError（保留字）
}
```

### 2.2 作用域隔离 — 模块级作用域

每个 ESM 文件拥有**独立作用域**，顶层变量不会挂到 `window`，同名也不冲突。

```javascript
// moduleA.js
const shared = 'moduleA';   // 模块私有，外部不可访问
export function getShared() { return shared; }

// moduleB.js
const shared = 'moduleB';   // 同名，互不干扰
export function getShared() { return shared; }

// app.js
import { getShared as getA } from './moduleA.js';
import { getShared as getB } from './moduleB.js';
console.log(getA()); // 'moduleA'
console.log(getB()); // 'moduleB'
```

### 2.3 延迟执行 — 默认 defer 行为

`<script type="module">` 默认相当于 `defer`：**不阻塞 HTML 解析**，等待 DOM 构建完成后**按文档顺序**执行。

| 类型 | 阻塞 HTML 解析 | 执行时机 | 顺序保证 |
|------|:---:|------|:---:|
| 普通 `<script>` | ✅ | 立即执行 | ✅ |
| `<script defer>` | ❌ | HTML 解析完成后 | ✅ |
| `<script async>` | ❌ | 下载完成后立即执行 | ❌ |
| `<script type="module">` | ❌ | HTML 解析完成后（同 defer） | ✅ |

**实际输出顺序示例：**

```html
<script>          console.log('1. 普通脚本'); </script>
<script type="module"> console.log('4. 模块脚本'); </script>
<script>          console.log('2. 普通脚本'); </script>
<script type="module"> console.log('5. 模块脚本'); </script>
<script>
    document.addEventListener('DOMContentLoaded', () => {
        console.log('6. DOMContentLoaded');
    });
</script>

<!-- 输出：1 → 2 → 4 → 5 → 6（模块脚本在普通脚本之后、DOMContentLoaded 之前） -->
```

## 三、核心机制

### 3.1 单例 — 模块只执行一次

同一个模块不管被 `import` 多少次，**只在第一次加载时执行**，后续 import 直接返回缓存的实例。

```javascript
// counter.js
console.log('模块执行');  // 只打印一次！
export let count = 0;
export function increment() { count++; }

// app1.js
import { count, increment } from './counter.js';
increment();
console.log(count); // 1

// app2.js
import { count } from './counter.js';
console.log(count); // 1（共享同一个实例，不是 0）
```

利用这一特性可以自然实现**全局状态管理**（如 store、配置对象等），无需手动写单例模式。

### 3.2 值的绑定（Live Binding）— 实时引用

`import` 拿到的不是值的副本，而是**导出值的实时引用**。导出方的值变化后，导入方**自动感知**。

```javascript
// counter.js
export let count = 0;
export function increment() { count++; }

// app.js
import { count, increment } from './counter.js';
console.log(count); // 0
increment();
console.log(count); // 1（自动更新！不是 0）
```

**与 CommonJS 的本质区别：**

```javascript
// ===== ES Module：解构出的值仍是实时引用 =====
// counter.mjs
export let count = 0;
export function increment() { count++; }

// app.mjs
import { count, increment } from './counter.mjs';  // import 解构 ≠ 拷贝
console.log(count); // 0
increment();
console.log(count); // 1 ✅ 自动更新——count 是实时绑定

// ===== CommonJS：解构出的原始值是副本 =====
// counter.js
exports.count = 0;
exports.increment = function() { exports.count++; };

// app.js
const { count } = require('./counter.js');    // 解构赋值 → 原始值拷贝！
const counter = require('./counter.js');       // 获取模块对象引用

console.log(count);        // 0
counter.increment();
console.log(count);        // 0 ❌ 解构出的 count 是拷贝，不会更新
console.log(counter.count); // 1 ✅ 通过对象引用访问，能看到最新值
```

**为什么会有这个区别？**

- CJS 的 `require()` 返回的是一个普通 JS 对象。`const { count } = obj` 解构原始值（number）时，拿到的是**值的拷贝**，和原对象脱钩。
- ESM 的 `import { count }` 虽然语法像解构，但底层是**模块绑定（module binding）**——`count` 直接指向导出模块的内存地址，不是拷贝。
- 如果你在 CJS 中始终通过 `counter.count` 访问（对象属性引用），表现确实和 ESM 一致——**但一旦解构就不同了，而 `import` 语法天然就是"解构"形式**，所以这个差异非常关键。

> **记忆口诀**：`import { x }` 是"给了你地址"（实时绑定），`const { x } = require()` 是"复印了一张"（值的副本）。

### 3.3 静态分析 — 编译时确定依赖

ESM 的 `import` 在**编译时**就能确定依赖关系，无需执行代码。

```javascript
// ✅ ESM：编译时就知道依赖
import { func } from './module.js';

// ❌ CJS：运行时才知道
const moduleName = './module.js';
const module = require(moduleName); // require 可以放在任何位置、接受动态参数
```

**编译时确定意味着：**

1. `import` 只能在模块**顶层**使用，不能放在 `if`、循环、函数内部（动态导入用 `import()` 代替）
2. 导入不存在的导出 → **编译时报错**（而不是运行时 `undefined`）
3. 打包工具可以构建完整的**模块依赖图** → Tree Shaking、Scope Hoisting 等优化的基础

## 四、进阶场景

### 4.1 循环依赖

两个模块互相 `import` 对方，初看像死循环，但 ESM 不会——它在**代码执行前**就做完了"接线"工作。

**从一个具体例子看全过程：**

```javascript
// a.js（入口）
import { bFunc } from './b.js';
export function aFunc() { console.log('aFunc 执行'); }
console.log('a.js 执行完毕');
bFunc();

// b.js
import { aFunc } from './a.js';
export function bFunc() { console.log('bFunc 执行'); }
console.log('b.js 执行完毕');
aFunc();
```

把 ESM 加载拆成三步，带着代码走一遍：

```
┌─ 第 1 步：解析（Parse）───────────────────────────────┐
│ 解析 a.js → 发现它依赖 b.js                            │
│ 解析 b.js → 发现它依赖 a.js（a.js 已解析过，跳过）       │
│ ✅ 两个模块的语法树都有了，但一行代码都还没执行             │
└──────────────────────────────────────────────────────┘
         ↓
┌─ 第 2 步：接线（Link）────────────────────────────────┐
│ 为 aFunc 分配内存 → function 声明 → 立刻初始化为函数     │
│ 为 bFunc 分配内存 → function 声明 → 立刻初始化为函数     │
│ import { aFunc } → 指向 aFunc 的内存地址（已经是有效函数）│
│ import { bFunc } → 指向 bFunc 的内存地址（已经是有效函数）│
│                                                        │
│ ⭐ 关键：两个函数此时都已经"活"了，可以被调用！            │
└──────────────────────────────────────────────────────┘
         ↓
┌─ 第 3 步：执行（Evaluate）────────────────────────────┐
│ 入口 a.js，深度优先：                                   │
│                                                        │
│ 开始执行 a.js → 先去执行它的依赖 b.js                    │
│   └→ 执行 b.js：                                       │
│        ├ b.js 依赖 a.js，但 a.js 状态为"执行中" → 跳过   │
│        ├ export function bFunc(){...} — 已接好线，跳过  │
│        ├ console.log     → "b.js 执行完毕"             │
│        └ aFunc()         → "aFunc 执行" ✅ 能调！       │
│                                                        │
│ 回到 a.js 继续：                                       │
│   ├ export function aFunc(){...} — 已接好线，跳过       │
│   ├ console.log     → "a.js 执行完毕"                  │
│   └ bFunc()         → "bFunc 执行" ✅                   │
└──────────────────────────────────────────────────────┘
```

**实际输出：**
```
b.js 执行完毕
aFunc 执行
a.js 执行完毕
bFunc 执行
```

**为什么函数能提前调用？** `export function` 是函数声明，在第 2 步"接线"时就被初始化了——代码还没执行，函数已经是有效的，所以循环依赖中互相调用不会报错。

---

**❌ 换一种写法就崩了：`export const fn = () => {}`**

`const` 的赋值发生在第 3 步（执行阶段），第 2 步时内存分配了但值是 **TDZ**：

```javascript
// a.js
import { bFunc } from './b.js';
export const aFunc = () => console.log('aFunc');  // ← 第 3 步才赋值
bFunc();

// b.js
import { aFunc } from './a.js';
export const bFunc = () => console.log('bFunc');
aFunc(); // 💥 ReferenceError: aFunc 在 TDZ
```

不止函数，**所有声明类型**在第 2 步（接线阶段）的表现都不一样。用一个统一场景来对比：

```javascript
// a.js（入口）
import { bFunc } from './b.js';
export function aFunc() { console.log('aFunc'); }   // 函数声明
export var aVar = 'a';                               // var
export let aLet = 'a';                               // let
export const aConst = 'a';                           // const 原始值
export const aArrow = () => console.log('aArrow');   // const 箭头函数
console.log('a.js 执行完毕');

// b.js
import { aFunc, aVar, aLet, aConst, aArrow } from './a.js';
export function bFunc() {
    console.log('aVar:', aVar);       // ?
    console.log('aLet:', aLet);       // ?
    console.log('aConst:', aConst);   // ?
    // aArrow();                      // ?
}
```

下表是 `bFunc()` 在第 3 步被 a.js 调用时，各导出的状态：

| 导出方式 | 第 2 步（接线）状态 | 第 3 步访问时 | 原因 |
|---------|-------------------|-------------|------|
| `export function` | ✅ 已初始化为函数 | 正常调用 | 函数声明在接线阶段被完整初始化 |
| `export var` | ⚠️ 已初始化为 `undefined` | 拿到 `undefined`，不报错 | `var` 有提升，初始值是 `undefined`（赋值还是第 3 步才执行） |
| `export let` | ⛔ TDZ | 💥 ReferenceError | `let` 声明提升但未初始化，在赋值前访问触发 TDZ |
| `export const` | ⛔ TDZ | 💥 ReferenceError | 同上，`const` 原始值在第 3 步赋值前处于 TDZ |
| `export const fn = ()=>{}` | ⛔ TDZ | 💥 ReferenceError | 同上，箭头函数本质上是通过 `const` 绑定——在赋值前函数还不存在 |

> **记忆方式**：第 2 步接线时，`function` 是"活"的，`var` 是 `undefined`（活但没值），`let`/`const` 是"死的"（碰就报错）。只有 `function` 在循环依赖中是安全的。

---

**CommonJS vs ESM 循环依赖对比：**

| 维度 | CommonJS | ES Module |
|------|----------|-----------|
| 处理机制 | 缓存已执行部分的 exports | 三阶段加载 + 函数提升 |
| 循环时拿到什么 | ⚠️ 可能是不完整的"半成品"（中间状态的值） | ✅ 接线阶段已初始化的绑定 |
| 推荐写法 | `require` 放函数内部延迟执行 | 用 `export function` 而非 `export const` |

### 4.2 动态导入 — 运行时按需加载

`import()` 函数在**运行时**返回 Promise，适用于代码分割、条件加载、懒加载。

```javascript
// 条件加载
if (isMobile) {
    const module = await import('./mobile.js');
    module.init();
}

// 懒加载（Vue Router 示例）
const routes = [
    { path: '/home', component: () => import('./views/Home.vue') },
    { path: '/about', component: () => import('./views/About.vue') },
];

// 变量路径（如国际化）
const module = await import(`./locales/${navigator.language}.js`);
```

> ⚠️ 动态导入**不支持 Tree Shaking**——运行时才能确定依赖，编译时无法分析哪些代码未使用。

### 4.3 Tree Shaking — 死代码消除

基于 ESM 的**静态分析**，打包工具在编译时识别并移除未被引用的代码（Dead Code Elimination）。

```javascript
// utils.js
export const usedFunc = () => console.log('used');
export const unusedFunc = () => console.log('unused');

// app.js
import { usedFunc } from './utils.js';
usedFunc();

// 打包后：unusedFunc 被移除 ✅
```

**三大生效条件：**
1. 使用 ESM 语法（`import { ... }` 而非 `require`）
2. 不能是动态导入（`import()` 运行时才确定依赖）
3. `package.json` 中 `"sideEffects": false`

**`sideEffects` 配置：**

```json
{
    "name": "my-library",
    "sideEffects": false,
    // 或标注有副作用的文件：
    "sideEffects": ["./src/polyfills.js", "*.css"]
}
```

| `sideEffects` | 含义 |
|:---|------|
| `false` | 所有模块都无副作用，打包工具可安全删除未使用代码 |
| `true`（默认） | 保守策略，不删除任何代码 |
| `["*.css"]` | 仅指定文件有副作用（CSS 的 import 本身是副作用） |

## 五、高级特性

### 5.1 顶层 await（ES2022）

模块顶层直接使用 `await`，无需包裹在 async 函数中。

```javascript
// data.js
const response = await fetch('/api/config');
export const config = await response.json();

// app.js
import { config } from './data.js';
console.log(config); // 等 data.js 的 await 完成后才执行
```

> ⚠️ 顶层 await 会**阻塞依赖链上所有模块**，直到 await 完成。注意对加载性能的影响。

### 5.2 导入断言（ES2022）

导入非 JS 资源时显式指定类型，防止服务器返回错误的 MIME 类型导致安全问题。

```javascript
import config from './config.json' assert { type: 'json' };
// 如果服务器返回非 JSON，直接报错，不会尝试当 JS 执行
```

### 5.3 导入映射（Import Maps）

在 HTML 中定义裸模块标识符（bare specifier）到实际 URL 的映射，让浏览器直接理解 `import React from 'react'`。

```html
<script type="importmap">
{
    "imports": {
        "react": "https://cdn.jsdelivr.net/npm/react@18/index.js",
        "@utils/": "/src/utils/"
    }
}
</script>

<script type="module">
    import React from 'react';              // → CDN 地址
    import { helper } from '@utils/helpers.js';  // → /src/utils/helpers.js
</script>
```

## 六、工程化配置

### 6.1 CORS 与 MIME 类型

浏览器中 ESM 默认启用 CORS 检查，且要求正确的 MIME 类型：

```
Content-Type: application/javascript   或 text/javascript
Access-Control-Allow-Origin: *        （跨域时必需）
```

> 同源请求不需要 CORS 头；CDN 引入的第三方 ESM 库需要 CDN 配置正确的 CORS 头。

### 6.2 模块标识符解析

```javascript
// 相对路径 → 相对于当前模块文件路径
import { func } from './module.js';
import { func } from '../parent.js';

// 绝对 URL → 直接发起 HTTP 请求
import { func } from 'https://cdn.example.com/lib.js';

// 裸标识符（bare specifier）→ 需要 import map（浏览器）或 node_modules 解析（打包工具 / Node.js）
import React from 'react';
```

### 6.3 package.json 关键字段

```json
{
    "name": "my-library",
    "type": "module",
    "main": "./dist/cjs/index.js",
    "module": "./dist/esm/index.js",
    "exports": {
        ".": {
            "import": "./dist/esm/index.js",
            "require": "./dist/cjs/index.js"
        }
    },
    "sideEffects": false
}
```

| 字段 | 作用 |
|------|------|
| `type: "module"` | Node.js 将 `.js` 视为 ESM |
| `main` | CJS 入口（`require` 时回退用） |
| `module` | ESM 入口（打包工具如 Webpack/Rollup **优先使用**） |
| `exports` | 条件导出——精确控制不同环境加载哪个文件，定义后外部只能访问 `exports` 列出的路径 |
| `sideEffects` | `false` = 所有模块无副作用，打包工具可安全 Tree Shaking |

## 七、对比与总结

### 7.1 ESM vs CommonJS 总结表

| 维度 | CommonJS | ES Module |
|------|----------|-----------|
| 语法 | `require` / `module.exports` | `import` / `export` |
| 加载时机 | 运行时、同步 | 编译时（静态）+ 运行时（动态 `import()`） |
| 值的绑定 | **副本** — require 后值不再变化 | **实时引用** — export 值变化后自动同步 |
| 循环依赖 | 返回"半成品"（已有部分的 exports） | 三阶段加载，函数声明在求值前已初始化 |
| Tree Shaking | ❌ 不支持 | ✅ 支持 |
| 顶层 this | 指向 `module.exports` | `undefined`（严格模式） |
| 环境 | Node.js（原生） | 浏览器 + Node.js（全平台） |

### 7.2 最佳实践

1. **新项目用 ESM** — JS 生态的事实标准
2. **库开发同时提供 ESM + CJS 双入口** — 通过 `exports` 条件导出
3. **循环依赖用 `export function`** — 避免 `export const fn = () => {}` 的 TDZ 陷阱
4. **`sideEffects: false`** — 让打包工具最大化 Tree Shaking 效果
5. **按需加载用 `import()`** — 减少首屏体积
