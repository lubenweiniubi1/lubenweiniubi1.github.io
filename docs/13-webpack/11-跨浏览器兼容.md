# 11 - 跨浏览器兼容

## 概念

### 1. 兼容问题到底"不兼容"什么？

写了一套代码，在不同浏览器上跑出不同结果——这是前端工程的经典问题。不兼容的根源可分为三个层面：

| 层面 | 问题 | 举例 |
|------|------|------|
| **语法层** | 新语法老浏览器解析失败 → 直接报错 | `const`、箭头函数、`??`、`?.`、`class` |
| **API 层** | 语法没问题，但运行时缺少方法/对象 | `Promise`、`Array.prototype.includes`、`fetch` |
| **样式层** | CSS 属性未实现或需要厂商前缀 | `grid`、`sticky`、`backdrop-filter` |

对应两套解决思路：

1. **语法问题 → 编译时转换（Transform）**：Babel 把新语法转成旧语法
2. **API 缺失 → 运行时补充（Polyfill）**：core-js 在全局/原型上注入缺失实现

### 2. 谁来决定"兼容到哪"？—— Browserslist

**Browserslist** 是所有构建工具（Babel、PostCSS、Autoprefixer）共同的"指挥棒"——它描述你支持哪些浏览器，工具链据此决定该做多少转换。

```json
// package.json
{
  "browserslist": [
    "> 0.5%",           // 全球使用率 > 0.5%
    "last 2 versions",  // 每个浏览器最新 2 版
    "not dead",         // 排除 24 个月无更新的浏览器
    "not ie 11"         // 排除 IE11
  ]
}
```

常用查询语法：

| 语法 | 含义 |
|------|------|
| `> 1%` | 全球使用率 > 1%（可加 `in CN` 限定区域） |
| `last 2 versions` | 每个浏览器最新 2 版 |
| `not dead` | 排除 24 个月无更新的浏览器 |
| `defaults` | `> 0.5%, last 2 versions, Firefox ESR, not dead` |

运行 `npx browserslist` 可查看当前配置覆盖的具体浏览器列表。

## JS 兼容（由浅入深）

### 2.1 核心工具链：Babel + core-js

JS 兼容的标准范式：

```
源码（ES2022+） → Babel（语法转换） → 目标环境代码
                       ↓
                  core-js（API 补丁注入）
```

- **Babel**：负责**语法层**——`const` → `var`、箭头函数 → `function`、`class` → `function + prototype`
- **core-js**：负责**API 层**——补充 `Promise`、`Array.from`、`Object.assign` 等

### 2.2 @babel/preset-env —— 一站式的方案

`@babel/preset-env` 把语法转换和 polyfill 注入打包在一起，你只需指定目标环境：

```js
// babel.config.js
module.exports = {
  presets: [
    ['@babel/preset-env', {
      // 推荐：不写 targets，自动读取 browserslist
    }]
  ]
};
```

**内部逻辑**：读取 browserslist → 对照 `@babel/compat-data`（基于 caniuse/MDN） → 只转换不被目标浏览器支持的语法/API → 避免多余转换，体积最优。

> **⚠️ 常见误区 1：以为 babel-loader 没生效**
>
> 刚配好 `@babel/preset-env`，发现 `const`、箭头函数都没被转译，就以为配置坏了。其实 babel 已经在工作——`browserslist` 默认值是 `defaults`（`> 0.5%, last 2 versions, Firefox ESR, not dead`），这些浏览器全都原生支持箭头函数和 `const`，所以 `preset-env` 按目标环境判断**不需要转译**。即使把范围放宽到 `> 0.25%`，覆盖的仍然是现代浏览器（2026 年全球只有不到 0.15% 的用户还在用 IE11 这类不支持 ES6 的浏览器），结果一样不转译。换成 `targets: "ie 11"` 就会看到全部转为 ES5——babel-loader 从头到尾都在正常工作。这不是 bug，是 preset-env 的智能优化：不为已支持的浏览器做多余转换。
>
> **⚠️ 常见误区 2：以为 babel-loader 只在生产模式生效**
>
> webpack 的 `mode`（`development` / `production` / `none`）只影响 webpack 内置行为（devtool 类型、Terser 压缩等），**不影响 loader 的执行**。babel-loader 在任何 mode 下都会运行。之所以开发模式的输出看起来"没转换"，是因为 webpack 开发模式默认使用 `eval` devtool，代码被包裹在 `eval("...")` 字符串里，表面上看代码似乎没变——实际上 babel 已经转换完了，只是 eval 里的内容看起来和源码一样（因为转译结果就是源码本身，modern targets 不需要改）。切到 `--mode none` 或把 targets 改成 `"ie 11"` 就能直观验证。

### 2.3 polyfill 注入策略 —— 最关键的选择

两种方案，解决"如何注入 polyfill"：

#### 方案 A：useBuiltIns（全局注入 → 适合应用）

```js
module.exports = {
  presets: [['@babel/preset-env', {
    useBuiltIns: 'usage',    // 自动扫描代码，按需注入
    corejs: '3.35',
  }]]
};
```

| useBuiltIns 值 | 行为 |
|:---|------|
| `false`（默认） | 不自动注入，需手动 `import 'core-js/stable'` |
| `'entry'` | 入口手动 `import 'core-js'`，Babel 按 targets 精确拆分 |
| `'usage'` | **推荐**——无需手动 import，自动扫描 + 按需注入 |

`'usage'` 编译示例：

```js
// 源码
const p = new Promise();
[1, 2].includes(1);

// 编译后 → Babel 自动插入精确的 polyfill import
import "core-js/modules/es.promise.js";
import "core-js/modules/es.array.includes.js";
const p = new Promise();
[1, 2].includes(1);
```

✅ 零配置、体积最优　❌ 污染全局（修改 `Array.prototype`）→ **不适合库开发**

#### 方案 B：transform-runtime（模块级注入 → 适合库）

```js
module.exports = {
  plugins: [['@babel/plugin-transform-runtime', { corejs: '3.35' }]]
};
```

编译后 polyfill 变成模块级引用，不污染全局：

```js
// 编译后
import _Promise from "@babel/runtime-corejs3/core-js-stable/promise";
const p = new _Promise();
```

✅ 不污染全局 → **适合库/组件开发**　❌ 体积稍大，实例方法需额外处理

| 场景 | 推荐方案 |
|------|---------|
| **应用开发** | `useBuiltIns: 'usage'` |
| **库/组件开发** | `transform-runtime` |

### 2.4 polyfill 的边界

| 类型 | 可 polyfill？ | 举例 |
|------|:---:|------|
| 静态方法 | ✅ | `Array.from()`、`Object.assign()` |
| 实例方法 | ✅（需污染原型） | `[].includes()`、`'str'.startsWith()` |
| 新语法 | ❌ 只能 transform | `const`、箭头函数、`async/await`、`?.` |
| 新的全局对象 | ✅ | `Promise`、`Map`、`Set`、`Symbol` |
| 引擎级行为 | ❌ 无法 polyfill | `Proxy`、`Reflect`（需引擎底层支持） |

**经验法则**：能通过"运行时注入一段 JS 代码"来弥补的 → 可 polyfill；需要修改 JS 引擎底层行为的 → 无法 polyfill。

### 2.5 差异化服务（Differential Serving）

旧浏览器（如 IE11）需要把代码全转成 ES5，体积暴涨 30%-50%。差异化服务的思路：**新浏览器用小包，旧浏览器用大包，互不拖累。**

```html
<!-- 现代浏览器加载 ES2017+ 轻量包 -->
<script type="module" src="/app-modern.js"></script>
<!-- 旧浏览器（不支持 type="module"）则加载 ES5 兼容包 -->
<script nomodule src="/app-legacy.js"></script>
```

Vite 通过 `@vitejs/plugin-legacy` 原生支持；Webpack 需要配置多套构建目标。

### 2.6 功能检测 vs UA 嗅探

```js
// ❌ UA 嗅探 —— 脆弱、可伪造
if (navigator.userAgent.includes('IE')) { /* ... */ }

// ✅ 功能检测 —— 只关心能力，不关心身份
if (!window.Promise) {
  await import('./polyfills/promise.js');
}
if (!('IntersectionObserver' in window)) {
  await import('intersection-observer');
}
```

## CSS 兼容（由浅入深）

### 3.1 为什么 CSS 也有兼容问题？

浏览器实现新 CSS 属性时会先加**厂商前缀**做实验性支持，等标准化后再去掉前缀：

```css
/* 手写：每个前缀都要写 */
.my-element {
  -webkit-user-select: none;
  -moz-user-select: none;
  user-select: none;          /* 标准写法放最后 */
}
```

### 3.2 Autoprefixer —— 自动前缀

**Autoprefixer** 根据 Browserslist 自动添加前缀，你只需写标准写法：

```css
/* 你写的 */
.my-element { user-select: none; }

/* Autoprefixer 输出 */
.my-element { -webkit-user-select: none; user-select: none; }
```

```js
// postcss.config.js
module.exports = {
  plugins: [require('autoprefixer')]
};
```

### 3.3 PostCSS 生态全景

PostCSS 是 CSS 处理框架，Autoprefixer 只是其中一个插件。类比 JS 中的 Babel：

| 插件 | 作用 | JS 类比 |
|------|------|---------|
| `autoprefixer` | 添加厂商前缀 | — |
| `postcss-preset-env` | 现代 CSS → 兼容 CSS | `@babel/preset-env` |
| `postcss-nested` | Sass-like 嵌套 | — |
| `cssnano` | CSS 压缩 | Terser |
| `postcss-import` | 内联 `@import` | — |

```js
// postcss.config.js —— 推荐配置
module.exports = {
  plugins: [
    require('postcss-preset-env')({
      stage: 3,                        // 只使用 Stage 3+ 的特性
      autoprefixer: { grid: true },
    }),
    require('cssnano')({ preset: 'default' }),
  ]
};
```

### 3.4 CSS 功能查询 —— @supports

类似 JS 功能检测，CSS 原生提供 `@supports`：

```css
/* 如果支持 grid → 用 grid */
@supports (display: grid) {
  .container {
    display: grid;
    grid-template-columns: repeat(3, 1fr);
  }
}

/* 不支持 → 自动落到 flexbox */
.container {
  display: flex;
  flex-wrap: wrap;
}
```

CSS 层叠本身就是 fallback 机制：旧样式写前面，新特性写在后面（或 `@supports` 块内），不支持的浏览器自动降级。

### 3.5 五大经典降级模式

```css
/* 1. 逻辑属性降级 */
.element {
  margin-left: 1rem;                  /* fallback */
  margin-inline-start: 1rem;          /* 支持逻辑属性的优先 */
}

/* 2. CSS 变量降级 */
.button {
  background: #007bff;                /* fallback */
  background: var(--primary, #007bff);
}

/* 3. Grid → Flexbox 降级 */
.layout { display: flex; flex-wrap: wrap; }
@supports (display: grid) {
  .layout { display: grid; grid-template-columns: repeat(auto-fill, minmax(250px, 1fr)); }
}

/* 4. aspect-ratio 降级 */
.box { width: 100%; height: 200px; }  /* fallback: 固定高度 */
@supports (aspect-ratio: 16/9) {
  .box { height: auto; aspect-ratio: 16/9; }
}

/* 5. backdrop-filter 降级 */
.overlay { background: rgba(0,0,0,0.6); }  /* fallback: 不透明遮罩 */
@supports (backdrop-filter: blur(10px)) {
  .overlay { background: rgba(0,0,0,0.25); backdrop-filter: blur(10px); }
}
```

### 3.6 CSS Reset / Normalize

消除浏览器默认样式差异是兼容第一步：

| 方案 | 理念 |
|------|------|
| **CSS Reset** | 把默认样式统一清零 |
| **Normalize.css** | 保留有用默认样式，只修正不一致 |
| **modern-normalize** | Normalize 的精简版，只保留核心修正 |

## HTML 兼容

### 4.1 语义元素和响应式图片

HTML5 语义标签（`<header>`、`<nav>`、`<section>` 等）在 IE9 以下不识别——使用 `html5shiv` 即可解决。

响应式图片自带 fallback 机制——旧浏览器忽略 `<picture>` / `<source>`，直接渲染内部的 `<img>`：

```html
<picture>
  <source srcset="image.avif" type="image/avif">
  <source srcset="image.webp" type="image/webp">
  <img src="image.jpg" alt="描述">  <!-- 最终 fallback -->
</picture>
```

## 构建工具集成

### 5.1 Webpack 完整兼容流水线

```
源文件 (.js / .css / .scss)
   │
   ├─→ JS  ─→ babel-loader ─→ 语法转换 + polyfill 注入
   │
   ├─→ CSS ─→ postcss-loader ─→ autoprefixer + 现代 CSS 降级
   │        ─→ css-loader ─→ 解析 @import / url()
   │        ─→ MiniCssExtractPlugin.loader ─→ 提取为独立文件
   │
   └─→ HTML ─→ html-webpack-plugin ─→ 注入打包后的资源
```

```js
// webpack.config.js —— 完整配置
module.exports = {
  module: {
    rules: [
      {
        test: /\.js$/, exclude: /node_modules/,
        use: {
          loader: 'babel-loader',
          options: {
            presets: [['@babel/preset-env', { useBuiltIns: 'usage', corejs: '3.35' }]]
          }
        }
      },
      {
        test: /\.css$/,
        use: [
          MiniCssExtractPlugin.loader,
          'css-loader',
          {
            loader: 'postcss-loader',
            options: {
              postcssOptions: {
                plugins: [
                  require('autoprefixer'),
                  require('postcss-preset-env')({ stage: 3 }),
                ]
              }
            }
          }
        ]
      }
    ]
  }
};
```

### 5.2 Vite 流水线

Vite 内置 PostCSS（只需提供 `postcss.config.js`），esbuild 处理语法转换但不做 polyfill，需 `@vitejs/plugin-legacy` 补位：

```js
import legacy from '@vitejs/plugin-legacy';
export default {
  plugins: [legacy({ targets: ['defaults', 'not ie 11'] })]
};
```

## 策略与实践

### 6.1 渐进增强 vs 优雅降级

| | 渐进增强 | 优雅降级 |
|---|---------|---------|
| **出发点** | 从最低基线开始，向上叠加 | 从最新特性开始，向下修补 |
| **思路** | "所有人都能用，好浏览器更好" | "先做最好，再修旧的" |
| **CSS** | 基础样式 → `@supports` 增强 | 新特性 → `@supports not` fallback |
| **JS** | 功能检测后加载增强 | Babel 编译 + polyfill |

两者不互斥——实际项目常混合：Babel/polyfill 做降级保底，`@supports` / 功能检测做增强。

### 6.2 兼容成本

| 兼容目标 | 体积增加 | 主要开销 |
|---------|:---:|------|
| 仅现代浏览器（ES2020+） | 基准 | 几乎不需要 transform |
| > 1% + last 2 versions | +5%~15% | 少量 polyfill |
| 加上 IE11 | +30%~50% | class 转换、大量 polyfill |
| 加上 IE9 | +100%+ | 全部 ES5 + DOM polyfill |

**决策**：看用户数据 → 评估商业影响 → 必要时用 differential serving 分流。

### 6.3 检查清单

| 环节 | 工具 |
|------|------|
| 确定兼容范围 | `.browserslistrc` / `package.json` |
| JS 语法转换 | `@babel/preset-env` |
| JS API polyfill | `core-js`（`useBuiltIns` 或 `transform-runtime`） |
| CSS 厂商前缀 | `autoprefixer` |
| 现代 CSS 降级 | `postcss-preset-env` |
| 样式归一化 | `modern-normalize` |
| 差异化服务 | `@vitejs/plugin-legacy` / Webpack 多配置 |
| 运行时检测 | `@supports`（CSS）、`'xxx' in window`（JS） |

> **一句话总结**：Browserslist 定范围 → Babel + core-js 处理 JS → PostCSS + Autoprefixer 处理 CSS → 功能检测兜底 → 按需 differential serving。
