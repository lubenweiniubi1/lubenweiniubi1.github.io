# 前端 Monorepo（单体仓库）模式

“**在前端 Monorepo（单体仓库）的架构中**”这句话指的是：**将多个前端项目或模块的源代码，统一放在同一个 Git 仓库（repository）里进行管理的一种开发模式**。

---

### 📌 什么是 Monorepo？

- **Mono** = 单一  
- **Repo** = Repository（代码仓库）

与之相对的是 **Multi-repo（多仓库）** 模式，即每个项目/包都有自己的独立仓库。

#### ✅ Monorepo 的典型结构示例：
```
my-frontend-monorepo/
├── packages/
│   ├── ui-components/     ← 公共 UI 组件库
│   ├── utils/             ← 工具函数库
│   ├── web-app/           ← 主站前端应用
│   └── admin-panel/       ← 后台管理系统
├── package.json
├── lerna.json
└── yarn.lock
```

所有子项目都放在一个仓库里，但各自有独立的 `package.json`，可单独发布为 npm 包。

---

### 💡 为什么前端要用 Monorepo？

1. **共享代码更方便**  
   比如 `utils` 和 `ui-components` 被 `web-app` 和 `admin-panel` 同时使用。在 Monorepo 中，可以直接引用本地代码（通过软链接），无需发布到 npm。

2. **统一依赖版本**  
   所有子项目使用同一份 `node_modules`（通过 Yarn Workspaces 提升依赖），避免不同项目依赖同一个库的不同版本导致冲突。

3. **原子化提交（Atomic Commits）**  
   一次 Git 提交可以同时修改多个相关包（例如：改了 `utils` 的 API，同时更新 `web-app` 的调用方式），保证代码一致性。

4. **简化 CI/CD 和脚本管理**  
   可以用 `lerna run test` 一键测试所有包，或只构建/部署变更过的包。

5. **团队协作更高效**  
   尤其适合大型前端团队维护多个相互关联的项目（如微前端、组件库 + 多个业务应用）。

---

### ⚠️ 注意：Monorepo ≠ 把所有东西塞进一个项目

Monorepo 强调的是 **“一个仓库，多个逻辑独立的包”**，而不是把所有代码写在一个 `src` 里。每个子包应保持清晰边界和独立性。

---

### 🔧 常见支持 Monorepo 的工具

| 工具 | 作用 |
|------|------|
| **Yarn Workspaces** | 管理依赖安装、本地软链接（核心基础设施） |
| **Lerna** | 版本管理、批量发布、运行跨包脚本 |
| **Nx** | 更强大的任务调度、缓存、依赖图分析（现代替代方案） |
| **pnpm + pnpm workspaces** | 类似 Yarn Workspaces，但更节省磁盘空间 |

---

### 总结

> “在前端 Monorepo 架构中” = **用一个 Git 仓库管理多个前端子项目/包，并借助工具（如 Yarn + Lerna）实现高效的依赖共享、版本控制和协作开发**。

这种模式已被 React、Babel、Angular、Vue 等大型开源项目广泛采用。


# 简单介绍

在前端 Monorepo（单体仓库）的架构中，Yarn Workspaces 和 Lerna 是两个非常经典的工具。为了让你更清晰地理解，我们可以这样简单区分它们的定位：**Yarn Workspaces 负责底层的“依赖安装与链接”，而 Lerna 负责上层的“版本管理与批量发布”**。

在现代的前端工程化实践中，通常会将两者结合使用（即 Yarn Workspaces + Lerna）。下面为你带来一份详细的实战教学。

### 📦 Yarn Workspaces：依赖管理与本地软链接

Yarn Workspaces 是 Yarn 自带的原生功能。它的核心作用是让多个子项目共享根目录的 `node_modules`（依赖提升），并自动为本地子包建立软链接（symlink），让你可以像引用 npm 包一样直接引用本地代码。

#### 1. 初始化项目结构
首先创建项目根目录，并初始化 `package.json`。
*注意：作为 workspace 的根目录，必须设置 `"private": true`，以防止根目录被意外发布到 npm。*

```bash
mkdir my-monorepo
cd my-monorepo
yarn init -y
```

修改根目录的 `package.json`，添加 `workspaces` 字段，指定子包存放的目录：
```json
{
  "name": "my-monorepo",
  "private": true,
  "workspaces": [
    "packages/*"
  ]
}
```

#### 2. 创建子包
在根目录下创建 `packages` 文件夹，并初始化两个子包（例如 `core` 和 `utils`）：

```bash
mkdir packages
cd packages
mkdir core utils

# 初始化 core 包
cd core
yarn init -y
# 记得修改 core/package.json 的 name 为 "@my-monorepo/core"

# 初始化 utils 包
cd ../utils
yarn init -y
# 记得修改 utils/package.json 的 name 为 "@my-monorepo/utils"
```

#### 3. 常用指令与跨包引用
回到项目根目录，运行 `yarn install`。Yarn 会自动扫描 `packages` 下的所有包，并将它们的公共依赖提升到根目录的 `node_modules` 中。

*   **为特定子包安装依赖：**
    `yarn workspace @my-monorepo/core add lodash`
*   **跨包引用（自动软链接）：**
    如果 `core` 包需要用到 `utils` 包的代码，只需在 `core` 的 `package.json` 中添加依赖：
    ```json
    "dependencies": {
      "@my-monorepo/utils": "1.0.0"
    }
    ```
    再次运行 `yarn install`，Yarn 会自动在 `core` 的 `node_modules` 中建立一个指向本地 `utils` 包的软链接，无需发布到 npm。
*   **在所有子包中运行命令：**
    在根目录的 `package.json` 中配置脚本：
    ```json
    "scripts": {
      "build": "yarn workspaces run build"
    }
    ```
    运行 `yarn build` 即可触发所有子包的构建命令。

### 🚀 Lerna：版本控制与批量发布

Lerna 是专为 Monorepo 设计的工具，它最强大的地方在于**自动化版本管理和发布流程**。它可以根据你的代码变更，自动判断哪些包需要升级版本号，并生成 CHANGELOG。

*注：Lerna 7.x 之后，官方推荐将底层的依赖安装工作完全交给 npm/yarn/pnpm 的 native workspaces 处理，Lerna 专注于版本发布。*

#### 1. 安装与初始化
在项目根目录安装 Lerna 并初始化：
```bash
npx lerna init
```
这会生成一个 `lerna.json` 配置文件。

#### 2. 核心工作流：Version 与 Publish
Lerna 的工作流通常结合 Git 提交规范（Conventional Commits，如 `feat: xxx`, `fix: xxx`）来使用。

*   **创建新包：**
    `npx lerna create @my-monorepo/cli`
*   **批量执行命令：**
    比如在所有的包里运行测试：`npx lerna run test`
*   **版本升级（最核心）：**
    `npx lerna version`
    运行此命令时，Lerna 会：
    1. 检查自上次发布以来，哪些包发生了变更。
    2. 提示你为这些变更的包升级版本号（遵循语义化版本规范 Major/Minor/Patch）。
    3. 自动更新对应包的 `package.json` 版本号。
    4. 自动生成或更新 `CHANGELOG.md`。
    5. 在 Git 中创建对应的 Tag 并提交。
*   **发布到 npm：**
    `npx lerna publish from-git`
    它会将刚刚打过 Tag 的版本推送到 npm 仓库。

### 🤝 强强联合：现代 Monorepo 最佳实践

在实际的企业级项目中，通常采用 **“Yarn Workspaces 管依赖，Lerna 管发布”** 的组合模式。

**配置步骤：**
1. 在根目录的 `package.json` 中开启 Yarn Workspaces（如上文所述）。
2. 在 `lerna.json` 中开启对 Yarn workspaces 的支持：
```json
{
  "$schema": "node_modules/lerna/schemas/lerna-schema.json",
  "version": "independent", 
  "npmClient": "yarn",
  "useWorkspaces": true 
}
```
*   `useWorkspaces: true`：告诉 Lerna 使用 Yarn 的 Workspaces 来处理依赖安装和软链接。
*   `version: "independent"`：表示各个子包独立管理版本号（适合组件库、工具库）；如果设为具体版本号（如 `"1.0.0"`），则是固定模式，所有包版本号同步（适合强耦合的系统）。

### 📌 总结对比

| 特性 | Yarn Workspaces | Lerna |
| :--- | :--- | :--- |
| **核心定位** | 依赖安装与管理 | 版本控制与发布生命周期 |
| **依赖提升(Hoist)** | ✅ 原生支持，自动提升公共依赖 | ❌ 已弃用，交由包管理器处理 |
| **本地软链接** | ✅ 自动建立 | ✅ (通过 `useWorkspaces` 调用 Yarn) |
| **自动化版本升级** | ❌ | ✅ (根据 Git 提交记录自动计算) |
| **生成 CHANGELOG** | ❌ | ✅ (集成约定式提交规范) |
| **批量执行脚本** | ✅ (`yarn workspaces run`) | ✅ (`lerna run`，过滤能力更强) |

建议你先从搭建一个简单的 Yarn Workspaces 项目开始，熟悉多包目录结构和依赖引用，再逐步引入 Lerna 来规范团队的版本发布流程。
