**single-spa** 是一个轻量级的**微前端框架**，被誉为微前端领域的开创性解决方案。以下是关于它的详细介绍：

## 一、核心概念

**single-spa**（读作"single spa"）是一个用于构建**微前端架构**的 JavaScript 框架，它允许将多个独立的前端应用整合到一个统一的主应用中，实现子应用之间的无缝切换与协同工作。

## 二、核心特性

### 1. **极简设计**
- 体积小于 **5KB**（gzip压缩后）
- 只专注于两件事：
  - **提供生命周期管理**：协调子应用的挂载和卸载
  - **路由劫持**：监听URL变化，匹配并执行对应子应用的生命周期

### 2. **多框架支持**
- 支持在同一页面上使用**多个不同的前端框架**（React、Vue、Angular等）
- 无需刷新页面即可切换不同框架的应用

### 3. **独立部署**
- 每个子应用可以独立开发、测试、部署
- 支持**增量升级**，无需重写现有应用

### 4. **懒加载优化**
- 延迟加载代码，改善初始加载时间
- 按需加载子应用，提升性能

## 三、工作原理

### 1. **路由劫持机制**
```javascript
// single-spa会劫持这些路由事件
- hashchange
- popstate  
- history.pushState
- history.replaceState

// 路由变化时触发 reroute() 函数
```

### 2. **生命周期管理**
single-spa定义了标准的子应用生命周期：
```javascript
// 子应用必须实现这些生命周期函数
export async function bootstrap(props) {
  // 初始化应用
}

export async function mount(props) {
  // 挂载应用到DOM
}

export async function unmount(props) {
  // 卸载应用，清理资源
}

export async function unload(props) {
  // 可选：完全卸载应用
}
```

### 3. **状态流转**
```
加载流程：toLoadPromise → toBootstrapPromise → toMountPromise
卸载流程：toUnmountPromise → toUnloadPromise
```

## 四、架构设计

### **主应用 + 子应用模式**
- **主应用（Root Application）**：负责子应用的调度和路由管理，业务逻辑尽量简单
- **子应用（Micro Applications）**：承担具体的业务逻辑，独立运行

```
┌─────────────────────────────────┐
│        主应用 (Root App)         │
│   - 路由管理                     │
│   - 子应用注册和调度             │
└────────────┬────────────────────┘
             │
    ┌────────┼────────┐
    ▼        ▼        ▼
┌──────┐ ┌──────┐ ┌──────┐
│子应用1│ │子应用2│ │子应用3│
│React │ │ Vue  │ │Angular│
└──────┘ └──────┘ └──────┘
```

## 五、与qiankun的关系

**qiankun是single-spa的增强版**：
- qiankun基于single-spa构建
- single-spa提供了核心的生命周期管理和路由劫持
- qiankun在此基础上增加了：
  - **样式隔离**（CSS沙箱）
  - **JS执行隔离**（快照沙箱）
  - **HTML Entry**支持
  - 更完善的通信机制

## 六、使用场景

### 适合使用single-spa的场景：
1. **大型复杂应用**：需要拆分成多个独立模块
2. **多团队协作**：不同团队使用不同技术栈
3. **渐进式重构**：逐步将旧系统迁移到新技术
4. **独立部署需求**：各模块需要独立发布和更新

### 不适合的场景：
1. **小型简单应用**：引入微前端会增加复杂度
2. **强耦合业务**：各模块间依赖紧密，难以拆分

## 七、优缺点分析

### ✅ 优点：
- 轻量级，学习成本低
- 灵活性高，支持多种框架
- 独立部署，降低发布风险
- 渐进式迁移，保护现有投资

### ❌ 缺点：
- **不处理样式隔离**：需要自行解决样式冲突
- **不处理JS隔离**：全局变量可能相互影响
- **需要额外工具**：通常需要配合SystemJS等模块加载器
- **调试复杂**：跨应用调试相对困难

## 八、快速开始示例

```javascript
// 主应用注册子应用
import { registerApplication, start } from 'single-spa';

registerApplication({
  name: '@org/navbar',
  app: () => System.import('@org/navbar'),
  activeWhen: ['/'],
});

registerApplication({
  name: '@org/dashboard',
  app: () => System.import('@org/dashboard'),
  activeWhen: ['/dashboard'],
});

start();
```
