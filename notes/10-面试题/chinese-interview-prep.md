## 2. 行为面试：STAR 法则答题要点

### Q：「讲讲你做过最有挑战的技术项目。」

**映射到：** 保时捷微信 → 抖音小程序迁移

**情境：** 保时捷经销商网络系统基于 TaroJS 在微信小程序上搭建。业务方想扩展到抖音生态去触达不同用户群。代码量大，不可能全量重写。

**任务：** 把微信小程序迁移到抖音小程序，最大化代码复用，保持功能完整。

**行动：**
- 先梳理了 Taro API 在抖音运行时上的兼容性问题
- 最大问题出在动画：抖音运行时的 CSS 动画和 requestAnimationFrame 处理方式不同，部分 Taro 原生 API 行为也不一致
- 用抖音兼容的方式重写了动画模块，组件层尽量保持不动
- 搭了一套共享代码结构，平台相关代码通过统一接口隔离

**结果：** 大约 70% 代码复用。这套迁移模式后来被其他团队参考复用，抖音版按计划上线。

**练习要点：** 准备好解释具体哪些 API 出了问题、你怎么解决的。面试官可能会追问 Taro 内部实现细节。

### Q：「讲一次你主导技术项目的经历。」

**映射到：** 跨平台组件库（保时捷项目）

**情境：** 四个平台（微信、抖音、Web Admin、H5）各自在写自己的 UI 组件。车辆卡片、经销商选择器、表单组件 —— 设计稿是同一套，实现却有四份。重复劳动拖慢了整体节奏。

**任务：** 搭建一套四个平台通用的组件库，放在 Monorepo 里，让各团队能各自贡献而不冲突。

**行动：**
- 选型 Taro + Lerna/Yarn Workspaces 做 Monorepo。Taro 本身支持多平台编译，Lerna 管包版本
- 先定组件 API 约定：每个组件在各个平台上接受哪些 props
- 从复用度最高的组件开始（车辆卡片、经销商选择器），逐步做了 15+ 个组件
- 配了 lint 规则和 code review 流程，规范其他团队的贡献
- 每个组件在需要时写平台适配层，但业务逻辑保持单一来源

**结果：** 四个平台不再各自重复造轮子。使用这些组件的功能开发明显提速，UI 层不用从头写。

**练习要点：** 准备好对比 Taro UI、Vant Weapp 这些方案，能说清楚为什么保时捷的设计体系值得自己造一套。

### Q：「代码审查中有分歧时你怎么处理？」

**映射到：** Thoughtworks 工程文化

**情境：** 在 Thoughtworks，code review 是强制环节。团队分布在澳洲、印度、中国，分歧经常出现：代码风格不同、TypeScript 严格程度不同、对抽象 vs 复制的看法不同。

**回答框架：**
1. **先看有没有规范。** ESLint 规则或团队约定能覆盖的，照做就完了。
2. **观点分歧，** 问一句「你这改动想解决什么问题？」很多时候问题对齐了，分歧就没了。
3. **真·设计分歧，** 别在 PR 评论区耗三天。写清楚两种方案的利弊，发团队频道公开聊。
4. **风格问题不阻塞。** 功能对、测试绿、代码能看懂，就 Approve 带建议，别卡人。

**面试时怎么讲（可以这样举例）：** 「记得有次 PR 有人想把工具函数抽成公共模块，我觉得太早。两个使用场景只是看起来相似，实际问题不同。我们最终各写各的，三个迭代后需求果然分叉了，我当时很庆幸没提前耦合。」

### Q：「描述一次你解决线上问题的经历。」

**映射到：** REA Group 事故响应

**情境：** 高流量房产列表页面出现线上问题，部分用户页面加载失败。

**任务：** 定位根因，在不影响平台其他部分的情况下部署修复。

**行动：**
- 先看监控和日志缩小范围：是特定地区？特定房产类型？特定浏览器？
- 锁定范围后在本地用相同数据条件复现
- 找到 bug，写了个复现的失败测试，修复，确认测试通过
- 用 feature flag 部署，出问题可以即时回滚
- 修复后写了根因分析，给这类错误加了监控

**结果：** 几小时内解决。事后复盘促使团队增加了针对该类错误的健康检查。

**如果被追问细节：** 「不是所有线上问题都是代码 bug。我也碰到过第三方 API 变更把集成搞挂的、缓存策略问题导致用户看到脏数据的。不管哪种类型，我的处理习惯都一样：先缩小范围 → 本地复现 → 带测试修 → 安全部署 → 加监控防复发。」

## 3. 技术深挖

### React

**Q：「讲讲 React Fiber 的架构，为什么它重要？」**

要点：
- Fiber 要解决的核心问题：同步渲染阻塞主线程。组件树一大就容易掉帧。
- 解决方案：Fiber 把组件树表示为 fiber 节点的链表，渲染可以被中断和恢复
- 关键概念：`requestIdleCallback`（或 Scheduler 的 polyfill）、优先级、reconciliation 的两个阶段（render vs. commit）
- 跟你的工作的关系：在保时捷做动画重的小程序时，理解调度机制能帮你避开卡顿
- 实践：`useTransition` 和 `useDeferredValue` 就是基于 Fiber 调度机制的 hook 层 API

**Q：「讲一下 React Hooks 内部实现。useState 到底怎么工作的？」**

要点：
- Hooks 依赖调用顺序。React 给每个 fiber 维护一个 hook 状态的链表。这就是为什么 hooks 不能写在条件里。
- `useState` 把状态存在队列里；`useEffect` 存在单独的链表中，commit 后执行
- 闭包陷阱：事件处理器中的过期状态，为什么 `useRef` + `.current` 能保持最新
- 性能：`useMemo`/`useCallback` —— 什么时候真有用，什么时候是过早优化

**Q：「你怎么对 React 应用做性能优化？」**

回答结构：
1. **先测。** React DevTools Profiler、Lighthouse、Web Vitals。没数据就别谈优化。
2. **常见问题：** 不必要的重渲染、打包体积大、render 中的重计算、图片未优化
3. **工具：** `React.memo`、`useMemo`、`useCallback`、代码分割（`React.lazy` + `Suspense`）、虚拟滚动、图片懒加载
4. **你的实例：** REA Group 的房产列表页有几百张卡片。可以聊虚拟滚动、图片优化、filter 状态变化时避免全量重渲染

### 跨平台（TaroJS 与小程序）

**Q：「解释 Taro 的架构。它怎么编译到不同平台？」**

要点：
- Taro 的思路：写 React/Vue 代码，编译到小程序模板（WXML、TTML 等）和各自运行时 API
- 编译管线：JSX → 小程序模板，逻辑层分离
- 运行时适配器模式：Taro 把平台 API 抽象成统一接口，每个平台有自己的适配器
- 你踩过的坑：动画性能差异、CSS 支持度不同、微信和抖音 API 行为不一致

**Q：「从开发者角度看，微信小程序和抖音小程序的主要区别？」**

基于你的实际经验：
- 运行时行为：抖音 JavaScript 运行时对定时器和动画的处理方式不同
- API 覆盖：不是所有微信 API 都有抖音对等版本；有些行为有细微差异
- CSS 支持：CSS 属性支持度不同，尤其是动画和 transform 相关
- 开发工具：调试体验不同；抖音 devtools 有进步但某些方面仍不如微信
- 生态：微信组件生态更大（WeUI、Vant Weapp）；抖音在追赶

### 工程化与 AI 辅助开发

**Q：「你是怎么设计 REA Group 的 Agent Skills 的？」**

1. **要解决的问题：** 开发流程是标准化的，但每个人执行方式不一样。代码审查反复发现同类问题。测试覆盖不统一。
2. **方案：** 不写一个大而全的脚本，而是把流程拆成可组合的技能模块：
   - Spec Skill：输入需求，输出有验收条件的结构化 spec
   - Unit Test Skill：读实现代码，生成覆盖正常路径和边界的 Jest 测试
   - Code Review Skill：对 diff 运行，按 ESLint 规则和团队约定检查，在人 review 前拦截问题
   - Issue Skill：发现 bug 时生成结构化报告：复现步骤、根因、修复建议
3. **效果：** PR 质量提升。review 耗时下降，因为机械问题被提前消灭。测试更统一，技能强制了相同模式。
4. **教训：** AI 生成的代码仍然需要人的判断。技能最适合做第一轮的工具。它处理机械劳动，让人专注设计和边界情况。

**Q：「你怎么看 AI 在软件开发中的现状？」**

- AI 对机械任务非常有用：样板代码、测试生成、文档、代码审查预检
- 差距：系统设计决策、利弊权衡、理解业务上下文 —— AI 还有很长的路
- 风险：过度依赖。如果开发者让 AI 替他思考，天花板会很快到来。核心能力正在从「写代码」转向「描述意图 + 验证输出」
- 个人观点：用 AI 来放大工程判断力，而不是取代它

## 4. 项目详解（每个 3 分钟叙事）

### 项目一：保时捷经销商网络系统

**背景（30 秒）：** 「保时捷需要一套面向全国经销商网络的数字化系统，几百家门店。销售、经理、售后都需要工具来管库存、做车辆推荐、跟销售流程。系统要支撑四个平台，因为不同角色用的设备和场景不一样。」

**你的角色（30 秒）：** 「我是前端负责人，负责架构选型、组件库策略、四个平台的质量基线。我还主导了微信到抖音的迁移，结果证明这是项目里最难的工程问题之一。」

**关键技术决策（60 秒）：**
- 「选了 TaroJS，因为它承诺一套代码多端运行。实际用下来它帮我们解决了大概 80% 的问题，剩下 20% 需要平台各自处理。」
- 「Monorepo 加组件库这个决定很关键。百人团队规模，如果各子团队各自造 UI，场面会很乱。组件库强制了一致性，省了大量时间。」

**成果（30 秒）：** 「四个平台，一个代码库，约 70% 复用。系统在保时捷全国经销商网络跑着。迁移方案被其他团队复用。」

**如果问重来一次怎么做：** 「前期应该在跨平台自动化测试上多投点时间。有些抖音特有的问题到 QA 阶段才发现，平台级测试覆盖上去了，能更早暴露。」

## 6. 常见坑和应对

### 坑 1：「你 2025 年 9 月离开 Thoughtworks，之后在做什么？」

**面试官实际在问：** 有空窗期？是不是被开了？

**你的回答：** 「我是主动离开的。REA Group 的项目在收尾，我想找一个能同时用到跨平台经验和 AI 工程实践的机会。这段时间我在沉淀技能：做做个人项目，关注 React 生态的变化，为下一次机会认真准备。」

态度上：正面、往前看、别绕。解释太多反而显得心虚。

### 坑 2：「七年两家公司，经验多样性够吗？」

**面试官实际在问：** 你见过足够多不同类型的问题吗？

**你的回答：** 「如果看具体项目的话，多样性其实挺够的。光在 Thoughtworks，我就从带百人团队的跨平台项目（保时捷），做到澳洲最大房产平台的全栈开发，再到搭建 AI 辅助开发工具链。再之前在小公司的经历也完全不一样：单兵作战、IoT、政务项目。覆盖面其实挺宽。」

### 坑 3：「你的英语怎么样？全英文环境有多适应？」

**面试官实际在问：** 沟通会有问题吗？

**你的回答：** 「在 REA Group 的时候，我每天早上和澳洲、印度的同事 standup、code review、架构讨论，全英文。英文文档和 spec 也是日常写的。不是母语水平，但够用，而且有实战经验。如果你们想验证一下，接下来我们可以切英文聊。」

语气上：自信，别急着证明。你做过他们担心的那件事，直接说事实就行。

### 坑 4：「你把 AI 辅助开发写得很突出。有些工程师觉得这是拐杖。」

**面试官实际在问：** 没 AI 你还能写代码吗？

**你的回答：** 「合理的问题。我写了七年代码，其中五年是在 AI 工具有用之前。我用 AI 干的都是机械活：生成测试框架、跑提交前检查、搭 spec 结构。架构决策、需要判断的事我不会交给 AI。可以把它理解成一个不会累的初级开发：吞吐量很高，但需要 review。」
