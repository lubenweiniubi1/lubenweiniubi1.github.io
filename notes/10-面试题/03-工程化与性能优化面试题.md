## 一、前端工程化（ESLint / Prettier / Husky）

### Q1: 你如何在前端项目中落地工程化规范？

**标准答案框架——完整 Lint 工作流：**

```
写代码 → 保存自动格式化 → git commit 触发 Husky
→ lint-staged 仅检查暂存文件 → commitlint 校验提交信息
→ 推送 → CI 最终守门
```

**工具链组合：**

| 工具 | 作用 | 触发时机 |
|------|------|----------|
| ESLint | 代码质量检查（未使用变量、潜在 bug） | 保存时 + pre-commit |
| Prettier | 代码风格统一（缩进、引号、分号） | 保存时自动格式化 |
| Husky | 管理 Git Hooks | `git commit` 时触发 |
| lint-staged | 仅检查 git 暂存文件 | pre-commit hook 内 |
| Commitlint | 提交信息规范校验 | commit-msg hook 内 |

### Q2: ESLint 代码检查的过程是什么？

1. **解析（Parse）**：将代码解析为 AST（抽象语法树）
2. **遍历（Traverse）**：深度优先遍历 AST 节点
3. **规则匹配（Lint）**：每个规则是一个 visitor，在进入/离开特定节点类型时执行检查
4. **报告（Report）**：收集所有违规项，统一报告

**追问：ESLint 9 的 Flat Config 有什么变化？**
- 使用 `eslint.config.js` 替代 `.eslintrc`
- 不再用 `extends`，改为数组形式的 `configs`
- 插件需要通过 `import` 引入

### Q3: 如何在团队中推行 Lint 规范？

**落地策略（分阶段）：**

1. **阶段一：本地自动化**（最小阻力）
   - 配置编辑器保存时自动格式化
   - 提供统一的 VSCode 配置 `.vscode/settings.json`
2. **阶段二：提交时卡口**
   - Husky + lint-staged 在 `pre-commit` 时自动修复
   - 只检查变更文件，速度可控
3. **阶段三：CI 守门**
   - CI 中执行全量 lint 作为最终门禁
   - 不通过则不允许合并
4. **阶段四：持续优化**
   - 定期 review 规则是否合理
   - 将高频问题写入团队文档

### Q6: 接口缓存策略有哪些？

**分层缓存策略：**

```js
// 1. HTTP 缓存（强缓存 + 协商缓存）
// Cache-Control: max-age=3600  // 强缓存 1 小时
// ETag / Last-Modified        // 协商缓存

// 2. 内存缓存（SWR 模式）
function useSWR(key, fetcher) {
  const cache = useRef(new Map());

  const { data, error } = useSWR(key, async (k) => {
    if (cache.current.has(k)) {
      return cache.current.get(k);  // 先返回缓存
    }
    const result = await fetcher(k);
    cache.current.set(k, result);   // 更新缓存
    return result;
  }, {
    revalidateOnFocus: true,        // 标签页聚焦时重新验证
    dedupingInterval: 2000,         // 2 秒内去重
  });

  return { data, error };
}

// 3. 请求去重
const pendingRequests = new Map();
async function fetchWithDedup(url) {
  if (pendingRequests.has(url)) {
    return pendingRequests.get(url);  // 复用进行中的请求
  }
  const promise = fetch(url).then(r => r.json());
  pendingRequests.set(url, promise);
  promise.finally(() => pendingRequests.delete(url));
  return promise;
}
```

## 三、防抖与节流（高频手写 + 场景题）

### Q7: 手写防抖函数（支持立即执行 + cancel）

```js
function debounce(fn, delay = 300, immediate = false) {
  let timer = null;

  const debounced = function (...args) {
    if (timer) clearTimeout(timer);

    if (immediate) {
      const callNow = !timer;
      timer = setTimeout(() => { timer = null; }, delay);
      if (callNow) fn.apply(this, args);
    } else {
      timer = setTimeout(() => {
        fn.apply(this, args);
        timer = null;
      }, delay);
    }
  };

  debounced.cancel = function () {
    clearTimeout(timer);
    timer = null;
  };

  return debounced;
}
```

### Q8: 手写节流函数（定时器版 + 时间戳版对比）

```js
// 时间戳版：第一次立即执行，停止后不执行最后一次
function throttle_timestamp(fn, delay = 300) {
  let lastTime = 0;
  return function (...args) {
    const now = Date.now();
    if (now - lastTime >= delay) {
      fn.apply(this, args);
      lastTime = now;
    }
  };
}

// 定时器版：第一次延迟执行，停止后会执行最后一次
function throttle_timer(fn, delay = 300) {
  let timer = null;
  return function (...args) {
    if (timer) return;
    timer = setTimeout(() => {
      fn.apply(this, args);
      timer = null;
    }, delay);
  };
}

// 综合版：第一次立即执行 + 停止后执行最后一次
function throttle(fn, delay = 300) {
  let lastTime = 0, timer = null;
  return function (...args) {
    const now = Date.now();
    const remaining = delay - (now - lastTime);

    if (remaining <= 0) {
      if (timer) { clearTimeout(timer); timer = null; }
      fn.apply(this, args);
      lastTime = now;
    } else if (!timer) {
      timer = setTimeout(() => {
        fn.apply(this, args);
        lastTime = Date.now();
        timer = null;
      }, remaining);
    }
  };
}
```

### Q9: 防抖和节流的区别？各自适用场景？

| 维度 | 防抖（Debounce） | 节流（Throttle） |
|------|-----------------|------------------|
| 行为 | 连续触发只执行最后一次（或第一次） | 固定时间间隔执行一次 |
| 适用场景 | 搜索输入联想、表单验证、resize 回调 | 滚动加载、鼠标拖拽、视频进度 |
| 抖音/头条案例 | 搜索框输入停顿后发请求 | 视频流滚动时固定频率加载下一批 |

## 四、虚拟列表（深度优化方案）

### Q10: 虚拟列表的实现原理

**核心思想**：只渲染可视区域内的 DOM 节点，用空白占位模拟滚动高度。

```
可视区域计算：
├── startIndex = Math.floor(scrollTop / itemHeight)
├── endIndex = Math.ceil((scrollTop + containerHeight) / itemHeight)
├── visibleData = allData.slice(startIndex, endIndex + overscan)
├── paddingTop = startIndex * itemHeight   （撑开上方空间）
└── paddingBottom = (total - endIndex) * itemHeight

关键事件：
├── scroll 事件 + requestAnimationFrame 节流
├── ResizeObserver 监听容器尺寸变化
└── IntersectionObserver 做懒加载
```

### Q11: 不定高列表项如何实现虚拟列表？

**两种方案：**

1. **预估高度 + 实测校准（react-window 做法）**
   ```
   - 预设一个预估高度（如 80px）
   - 实际渲染后通过 getBoundingClientRect() 获取真实高度
   - 更新高度缓存并重新计算偏移量
   ```

2. **ResizeObserver 动态监听（react-virtuoso 做法）**
   ```
   - 用 ResizeObserver 监听每个已渲染项的高度变化
   - 自动更新总高度和偏移量
   ```

### Q12: 虚拟列表的空白闪烁问题怎么解决？

- **overscan（缓冲区）**：在可视区域外多渲染 2-5 个 item
- **骨架屏占位**：给未渲染区域预置灰色骨架
- **min-height 兜底**：给每行设置最小高度，避免高度突变

### Q13: 如何检测和优化 Long Task？

```js
// 检测
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.duration > 50) {
      console.warn('Long Task:', entry);
    }
  }
});
observer.observe({ entryTypes: ['longtask'] });

// 优化：拆分长任务
function splitLongTask(tasks, chunkSize = 5) {
  let index = 0;
  function processChunk() {
    const end = Math.min(index + chunkSize, tasks.length);
    for (let i = index; i < end; i++) tasks[i]();
    index = end;
    if (index < tasks.length) {
      requestIdleCallback(processChunk); // 或 setTimeout
    }
  }
  requestIdleCallback(processChunk);
}
```

## 七、Webpack（高频必问）

### Q15: Webpack 的核心概念？Loader 和 Plugin 的区别？

**核心概念：**

| 概念 | 说明 |
|------|------|
| Entry | 打包入口，webpack 从这里开始构建依赖图 |
| Output | 打包产物输出路径和文件名 |
| Loader | 模块转换器，将非 JS 文件转为 webpack 能处理的模块 |
| Plugin | 插件，扩展 webpack 在构建各阶段的能力 |
| Mode | development / production / none，决定内置优化策略 |

**Loader vs Plugin 核心区别：**

- **Loader**：运行在模块编译阶段，对单个文件进行转换（如 `babel-loader` 转译 JSX，`css-loader` 处理 CSS 中的 `@import`/`url()`，`style-loader` 将 CSS 注入 DOM）。本质是一个函数，接收源文件内容，返回转换后的内容。
- **Plugin**：可以监听 webpack 整个生命周期中的钩子（hooks），在特定时机执行更复杂的任务（如打包优化 `TerserPlugin`、资源管理 `HtmlWebpackPlugin`、环境变量注入 `DefinePlugin`）。本质是一个带有 `apply` 方法的类。

**手写 Plugin 示例：**

```js
// 一个简单的插件：打包完成后输出文件列表
class FileListPlugin {
  apply(compiler) {
    compiler.hooks.emit.tapAsync('FileListPlugin', (compilation, callback) => {
      const fileList = Object.keys(compilation.assets)
        .map(name => `- ${name} (${compilation.assets[name].size()} bytes)`)
        .join('\n');

      compilation.assets['filelist.md'] = {
        source: () => `# Build Assets\n\n${fileList}`,
        size: () => fileList.length,
      };

      callback();
    });
  }
}

module.exports = FileListPlugin;
```

### Q16: Webpack 构建流程是怎样的？

```
初始化参数 → 创建 Compiler → 确定入口 → 编译模块 (Loader) → 完成模块编译 → 输出资源 → 输出完成

    make ───────── seal ──────── emit
```

**各阶段说明：**

1. **初始化参数**：从配置文件 + CLI 参数合并得到最终配置
2. **创建 Compiler**：`webpack()` 调用生成 Compiler 实例，它是整个构建过程的核心控制器
3. **确定入口**：从 `entry` 配置开始递归解析依赖
4. **编译模块**：匹配对应 Loader 对模块进行转换，生成 AST 并分析依赖，递归处理所有依赖模块
5. **完成模块编译**：所有模块完成编译，每个模块有了最终内容和依赖关系
6. **输出资源**：根据入口和模块间依赖关系，组装成 Chunk，每个 Chunk 对应一个 Bundle 文件
7. **输出完成**：根据 output 配置写入磁盘

### Q17: Webpack 构建优化（打包速度 + 打包体积）？

**构建速度优化：**

| 手段 | 说明 |
|------|------|
| `exclude` / `include` | 精确指定 loader 作用范围，避开 `node_modules` |
| 持久化缓存 | Webpack 5 内置，`cache.type = 'filesystem'` |

**打包体积优化：**

|------|------|
| Tree Shaking | 基于 ESM 的静态分析，删除未引用的导出 |

### Q18: Webpack 的 Tree Shaking 原理？什么情况下会失效？

**原理：**

1. **静态分析**：基于 ESM 的 `import`/`export` 语句是静态的（在编译时确定），webpack 遍历 AST 标记每个导出是否被使用
2. **`usedExports`**：标记未使用的导出为 `/* unused harmony export */`
3. **TerserPlugin**：最终压缩阶段删除未被使用的代码（DCE，Dead Code Elimination）

**失效情况：**

| 原因 | 说明 | 解决方案 |
|------|------|----------|
| CommonJS | `require()` / `module.exports` 是动态的，无法静态分析 | 改用 ESM |
| 副作用（Side Effects） | 某些模块导入时就执行全局代码（如 polyfill、CSS） | `package.json` 中设置 `"sideEffects": false` 或 `"sideEffects": ["./polyfill.js"]` |
| 动态引用 | `import(variable)` 无法在编译时确定导出 | 避免动态路径 |
| Babel 配置问题 | `@babel/preset-env` 的 `modules: "commonjs"` 会把 ESM 转成 CommonJS | 设置 `modules: false` 保留 ESM |

**`sideEffects` 字段的作用：**

```json
{
  "name": "my-library",
  "sideEffects": false   // 告诉 webpack: 整个包都没有副作用，未使用的导出可以安全删除
}
```

### Q19: Webpack module/chunk/bundle 的区别？

```
源文件（JS、CSS、图片...）
    │
    ▼
module（模块）  ← 每个源文件都是一个 module
    │
    ▼
chunk（代码块）  ← webpack 根据入口和代码分割规则将 module 分组
    │
    ▼
bundle（输出文件） ← chunk 经过编译和优化后输出到磁盘的文件
```

**定义：**

| 概念 | 说明 |
|------|------|
| **Module** | 项目中每一个文件（JS / CSS / 图片 / 字体等）都被视为一个模块 |
| **Chunk** | webpack 根据依赖关系将多个 module 组合成的代码块。一个 chunk 可能包含多个 module |
| **Bundle** | chunk 经过编译、压缩等处理后输出的文件，也就是我们最终在浏览器中加载的文件 |

**示例：**
```
entry: { main: './src/index.js', admin: './src/admin.js' }
splitChunks: { chunks: 'all' }

→ 生成 main.js（main chunk 的 bundle）
   admin.js（admin chunk 的 bundle）
   vendors~main~admin.js（公共依赖的 bundle）
```

### Q20: Webpack 5 模块联邦 (Module Federation) 原理和使用场景？

**原理：**

模块联邦允许多个独立构建的应用在运行时共享代码，实现微前端架构。

- **Host（宿主）**：加载远程模块的应用
- **Remote（远程）**：暴露模块给其他应用的应用
- **Shared（共享依赖）**：多个应用共享的库（如 React、Vue），避免重复加载

```js
// remote 应用配置
const { ModuleFederationPlugin } = require('webpack').container;

new ModuleFederationPlugin({
  name: 'app2',
  filename: 'remoteEntry.js',
  exposes: {
    './Button': './src/Button',
    './Dashboard': './src/Dashboard',
  },
  shared: { react: { singleton: true }, 'react-dom': { singleton: true } },
});

// host 应用配置
new ModuleFederationPlugin({
  name: 'app1',
  remotes: {
    app2: 'app2@http://localhost:3002/remoteEntry.js',
  },
  shared: { react: { singleton: true }, 'react-dom': { singleton: true } },
});
```

**使用场景：**
- 微前端架构：不同团队独立开发部署，运行时组合
- 大型应用拆分为多个子应用，按需加载
- 跨项目组件共享（如设计系统组件库）

### Q21: Webpack HMR (热更新) 原理？

```
文件变更 → 增量编译 → 推送 hash → 客户端请求 hot-update.json
    → 拉取新模块 → 模块替换 (module.hot.accept) → 更新 UI
```
https://www.bilibili.com/video/BV1kP41177wp?spm_id_from=333.788.videopod.episodes&vd_source=d66a6fb5cb08fa8db4dd3bf2bd839f71&p=51
https://www.bilibili.com/video/BV14q2LBKEGd/?spm_id_from=333.337.search-card.all.click&vd_source=d66a6fb5cb08fa8db4dd3bf2bd839f71
**详细流程：**

1. **文件变动**：监听的文件发生变化
2. **增量编译**：webpack 重新编译变更的模块，生成两个文件：
   - `hot-update.json`：描述本次更新的 hash 和变更模块列表
   - `hot-update.js`：新模块代码
3. **通知客户端**：通过 WebSocket 将编译 hash 推送到浏览器
4. **拉取更新**：客户端收到 hash 后，通过 JSONP 请求 `hot-update.json` 和 `hot-update.js`
5. **模块替换**：执行新模块代码，替换旧模块，调用 `module.hot.accept()` 触发更新回调
6. **视图更新**：如果使用 React Fast Refresh / Vue HMR，框架自动重新渲染受影响组件

### Q22: 手写一个简单的 Webpack Plugin？

下面展示一个完整的插件示例：生成打包文件清单。

```js
// file-list-plugin.js
class FileListPlugin {
  constructor(options = {}) {
    this.filename = options.filename || 'filelist.md';
  }

  apply(compiler) {
    // compiler.hooks.emit：在输出资源到 output 目录之前触发
    compiler.hooks.emit.tapAsync('FileListPlugin', (compilation, callback) => {
      // compilation.assets 包含所有即将输出的资源
      const lines = ['# Build File List\n', '| File | Size |', '| --- | --- |'];

      for (const [filename, asset] of Object.entries(compilation.assets)) {
        lines.push(`| ${filename} | ${asset.size()} bytes |`);
      }

      const content = lines.join('\n');

      // 向输出中添加一个新文件
      compilation.assets[this.filename] = {
        source: () => content,
        size: () => content.length,
      };

      callback();
    });
  }
}

module.exports = FileListPlugin;
```

**使用：**

```js
// webpack.config.js
const FileListPlugin = require('./file-list-plugin');

module.exports = {
  plugins: [
    new FileListPlugin({ filename: 'assets-report.md' }),
  ],
};
```

### Q23: Webpack 的 source-map 有哪些模式？各有什么区别？

| 模式 | 速度 | 质量 | 说明 |
|------|------|------|------|
| `eval` | 最快 | 差 | 每个 module 用 `eval()` 包裹，定位到文件行 |
| `cheap-source-map` | 快 | 一般 | 不带列信息，只有行信息 |
| `cheap-module-source-map` | 较慢 | 较好 | 带 loader 处理前的原始源代码行信息 |
| `source-map` | 最慢 | 最好 | 完整的原始源代码映射，生产可用 |
| `hidden-source-map` | 同 source-map | 最好 | 生成 map 文件但不引用，用于错误监控 |
| `nosources-source-map` | 同 source-map | 较好 | 有错误栈但看不到源代码内容，安全性好 |

**开发环境推荐：** `eval-cheap-module-source-map`

**生产环境推荐：**
- 公开项目：不生成 source-map 或 `nosources-source-map`
- 内部监控：`hidden-source-map`（source map 上传到监控平台）

## 八、Vite（高频必问）

### Q24: Vite 为什么快？和 Webpack 的核心区别？

| 维度 | Vite | Webpack |
|------|------|---------|
| **开发原理** | 基于原生 ESM，按需编译 | 全量打包后提供服务 |
| **冷启动** | 无需打包，直接启动 dev server | 需要先打包所有模块 |
| **热更新** | 只使失效模块对应的链，速度与项目大小无关 | 全量或增量打包，大项目变慢 |
| **构建工具** | 生产用 Rollup（开发用 esbuild） | 统一用 webpack |
| **依赖处理** | esbuild 预构建（Go 编写，极快） | 用 js 处理依赖 |

**为什么 Vite 开发时快？**

1. **免打包启动**：Vite 在开发时将依赖用 esbuild 预构建并缓存到 `node_modules/.vite`，源码直接以原生 ESM 方式提供给浏览器
2. **按需编译**：浏览器请求哪个模块才编译哪个模块，而非像 webpack 那样全量打包
3. **esbuild 预构建**：用 Go 编写的 esbuild 比 JS 编写的打包器快 10-100 倍

### Q25: Vite 的预构建 (Pre-Bundling) 做了什么？

**预构建发生在 `vite dev` 启动时，主要做两件事：**

1. **CommonJS → ESM 转换**
   - 浏览器原生只支持 ESM 模块
   - 将 `require()` 形式的依赖转为 `import`/`export` 形式

2. **合并散装模块**
   - 如 `lodash` 有几百个内部文件，如果不合并，浏览器会发出几百个 HTTP 请求
   - 预构建将它们合并为少数几个 ESM 模块，减少请求数量

**缓存机制：**
- 结果存储在 `node_modules/.vite` 目录
- 通过文件 hash 判断是否需要重新预构建
- 修改 `vite.config.js` 的 `optimizeDeps` 配置或 `package-lock.json` 变更时会触发重构建

### Q26: Vite 生产环境为什么用 Rollup 而不是 esbuild？

| 原因 | 说明 |
|------|------|
| **代码分割** | esbuild 的代码分割能力较弱，Rollup 更成熟 |
| **CSS 处理** | Rollup 的 CSS 插件生态更完善 |
| **产物质量** | Rollup 产物体积通常更小，优化更细致 |
| **插件生态** | Rollup 有丰富的官方和社区插件 |
| **稳定性** | Rollup 经历多年生产验证，esbuild 相对较新 |

**未来方向：** Vite 团队正在开发 **Rolldown**——用 Rust 重写的打包工具，目标是统一开发和生产构建工具链。

### Q27: Vite 和 Webpack 如何选择？

| 维度 | Vite | Webpack |
|------|------|---------|
| **开发体验** | 冷启动极快，HMR 即时 | 启动慢，大项目 HMR 延迟 |
| **生态成熟度** | 较新但已很完善 | 非常成熟的生态和社区 |
| **生产产物** | Rollup 打包，体积小 | 配置灵活，可精细控制 |
| **浏览器兼容** | 依赖 ESM，低版本浏览器需 polyfill | 兼容所有浏览器 |
| **大型项目** | 单体大项目可能遇到边界问题 | 久经考验，大型项目首选 |
| **微前端** | 需额外配置 | 模块联邦原生支持 |
| **SSR** | 原生 SSR 支持良好 | 需结合 next.js / nuxt |

**选择建议：**
- **新项目首选 Vite**：尤其是 Vue 3 / React + TypeScript 项目
- **大型存量项目**：如果已有 webpack 配置且运行良好，不必强行迁移
- **需要微前端**：Webpack 5 的 Module Federation 更成熟
- **需要兼容旧浏览器**：Webpack 或 Vite + `@vitejs/plugin-legacy`

### Q28: Vite 的 HMR 原理？

```
文件变更 → 服务端通知 → 浏览器通过 WebSocket 收到更新
    → 按模块依赖链精确失效 → 浏览器请求新模块 → 热替换
```

**核心原理：**

1. **原生 ESM 支持**：Vite 利用浏览器原生 ESM 能力，模块更新只需替换单个模块
2. **失效链追踪**：Vite 精确追踪模块的导入关系链，只对受影响模块进行更新
3. **esbuild 快速编译**：变更的源码用 esbuild 快速转译后推送

**对比 Webpack HMR：**

| 对比项 | Vite | Webpack |
|--------|------|---------|
| 更新粒度 | 模块级，精确失效 | chunk 级，批量更新 |
| 速度是否受项目大小影响 | 否，只编译变更模块 | 是，大项目更慢 |
| 依赖编译方式 | esbuild（Go） | webpack（JS） |
| 是否需要打包 | 不需要，直接提供 ESM | 需要，即使增量也要打包 |

## 九、Babel 与 AST

### Q29: Babel 的编译过程？

```
源代码 → Parse（解析）→ AST → Transform（转换）→ 新 AST → Generate（生成）→ 输出代码
```

**三个阶段详解：**

| 阶段 | 作用 | 核心工具 |
|------|------|----------|
| **Parse（解析）** | 将源代码解析为 AST（词法分析 → 语法分析） | `@babel/parser` |
| **Transform（转换）** | 遍历 AST，应用插件进行修改（新增/删除/替换节点） | `@babel/traverse` + `@babel/types` |
| **Generate（生成）** | 将转换后的 AST 重新生成为代码字符串 | `@babel/generator` |

**为什么需要这些阶段？**

```
// 源代码（你写的）
const sum = (a, b) => a + b;

// ↓ Parse → AST（数据结构表达）
{
  type: "VariableDeclaration",
  declarations: [{
    type: "VariableDeclarator",
    id: { type: "Identifier", name: "sum" },
    init: {
      type: "ArrowFunctionExpression",
      params: [...],
      body: { type: "BinaryExpression", ... }
    }
  }]
}

// ↓ Transform（插件修改 AST，如转为 ES5）

// ↓ Generate → 输出代码
"var sum = function(a, b) { return a + b; };"
```

### Q30: 如何写一个简单的 Babel Plugin？

**目标**：写一个插件，移除生产环境下所有的 `console.log` 调用。

```js
// babel-plugin-remove-console.js
module.exports = function () {
  return {
    name: 'remove-console',
    visitor: {
      CallExpression(path) {
        const callee = path.get('callee');
        // 匹配 console.log / console.warn / console.error 等
        if (
          callee.isMemberExpression() &&
          callee.get('object').isIdentifier({ name: 'console' })
        ) {
          path.remove();
        }
      },
    },
  };
};
```

**在使用时配置：**

```json
{
  "plugins": [
    ["babel-plugin-remove-console", { "exclude": ["error", "warn"] }]
  ]
}
```

**关键 API：**

| API | 作用 |
|-----|------|
| `path.node` | 当前 AST 节点 |
| `path.get()` | 获取子节点路径 |
| `path.replaceWith()` | 替换当前节点 |
| `path.remove()` | 删除当前节点 |
| `@babel/types`（`t`） | 辅助创建和检查节点类型 |
| `path.traverse()` | 在当前节点子树中继续遍历 |

## 十、CI/CD 与部署

### Q31: 你们项目的 CI/CD 流程是怎样的？

**示例流程：**

```
Push → GitHub Actions / GitLab CI
  ├── Lint（ESLint + Prettier + TypeScript）
  ├── Test（单元测试 + 集成测试）
  ├── Build（构建生产包）
  ├── Docker 镜像构建 & 推送
  └── Deploy（部署到目标环境）
```

**分支策略：**

```
feature/xxx → dev（自动部署 dev 环境）
                  │
                  ↓
              staging（预发布验证）
                  │
                  ↓
              main（生产环境）
```

**CI 流水线关键环节：**

| 环节 | 说明 |
|------|------|
| Lint | 早发现问题，避免浪费构建资源 |
| 单元测试 | 快速反馈，`jest --changedSince` 只测变更 |
| E2E 测试 | 关键路径回归，可并行或用 Testing Farm |
| 构建 | 产物缓存优化，避免重复安装依赖 |
| 部署 | 支持回滚，保留最近 N 个版本 |

### Q32: 怎么实现前端项目的自动部署和灰度发布？

**蓝绿部署：**

```
当前版本（Blue）→ 保持运行
新版本（Green）→ 部署到新环境并验证
验证通过 → 切换流量到 Green
出现问题 → 立即切回 Blue
```

**灰度发布（金丝雀发布）：**

```
新版本先只对 5% 的流量生效
监控无问题 → 逐步放大到 20% → 50% → 100%
出现异常 → 立即回滚
```

**前端灰度实现方案：**

| 方案 | 说明 |
|------|------|
| Nginx 按 Cookie 分流 | 特定 cookie 值指向新版本 |
| Nginx 按 Header 分流 | 按 User-Agent / IP 灰度 |
| Feature Flag | 通过开关控制功能上线（LaunchDarkly / 自研） |
| 版本号 + CDN | 新版本资源部署到 CDN，通过构建时版本号控制 |

**项目实践的注意事项：**
- 先部署静态资源，再切换入口 HTML（避免 HTML 引用到不存在的资源）
- 保留回滚能力：`deploy --rollback v1.2.3`
- 灰度期间做好监控：错误率、页面性能、用户反馈

### Q33: Monorepo 方案对比（Lerna / Nx / Turborepo / pnpm workspace）？

| 维度 | Lerna | Nx | Turborepo | pnpm workspace |

| **增量构建** | 不支持 | 基于计算哈希 | 基于文件变更 | 不支持 |

- **已有 Lerna 项目**：可以与 pnpm/yarn workspace 配合使用

### Q34: Monorepo 中的依赖管理问题怎么解决？

**常见问题及解决方案：**

**1. 幽灵依赖（Phantom Dependencies）**

- **问题**：某个包的依赖被其他包意外访问（如 A 依赖 B，A 的 `node_modules/b` 被 C 使用）
- **解决方案**：使用 **pnpm workspace** 的严格模式，用硬链接 + 符号链接管理依赖，`node_modules` 结构不扁平

**2. 版本一致性**

- **问题**：多个包依赖同一个库的不同版本，导致打包体积增大或运行时冲突
- **解决方案**：
  ```bash
  # 使用 syncpack 检查版本一致性
  npx syncpack list-mismatches

  # 统一版本
  npx syncpack fix-mismatches
  ```

**3. 循环依赖**

- **问题**：A 依赖 B，B 依赖 A，导致构建失败或运行时死循环
- **解决方案**：
  - 使用 `dpdm` 或 `madge` 检测循环依赖
  - 重构代码，抽离公共部分为独立包
  - 设置 ESLint 规则 `import/no-cycle` 阻止新增循环依赖

**4. 依赖重复安装**

- **问题**：不同包的相同依赖被重复安装多次
- **解决方案**：
  - pnpm workspace 自动提升公共依赖到根 `node_modules`
  - 使用 `pnpm.overrides` / `resolutions` 锁定统一版本
