## 一、整体架构

```
用户浏览器
    │
    ▼
┌──────────────────────────────────────────────────┐
│  Nginx :80（主应用）                               │
│  ├─ /                → 主应用首页                   │
│  ├─ /login           → 主应用登录页                 │
│  ├─ /user/*          → qiankun 加载 React 子应用    │
│  └─ /content/*       → qiankun 加载 Vue 子应用      │
└──────────────────────────────────────────────────┘
         │                    │
    fetch + eval         fetch + eval
         │                    │
         ▼                    ▼
   React 子应用 :8001    Vue 子应用 :8002
```

**核心概念**：qiankun 基于 **single-spa**，通过 `fetch` 拉取子应用的 HTML/JS/CSS，在主应用的容器 DOM 内执行和渲染。URL 变化时，qiankun 根据 `activeRule` 自动激活/卸载子应用。

## 二、搭建全流程（8 步）

### 第 1 步：主应用初始化

- Webpack 5 + Babel + React 18 脚手架
- CSS Modules 做样式隔离（`xx.module.css` 自动哈希类名）
- 关键 webpack 配置：
  - `publicPath: '/'` — 与 qiankun 的路由匹配一致
  - `historyApiFallback: true` — SPA 路由支持
  - `headers: { 'Access-Control-Allow-Origin': '*' }` — 允许子应用跨域 fetch
- `config.js` 集中管理子应用注册信息（name, entry, container, activeRule）

### 第 2 步：注册子应用

在主应用 `src/index.js` 中：

```js
import { registerMicroApps, start } from 'qiankun';

registerMicroApps(apps, lifecycles);
start({
  sandbox: { experimentalStyleIsolation: true },
  prefetch: 'all',
});
```

**registerMicroApps**：注册子应用列表，每个子应用指明 name、entry（子应用地址）、container（挂载容器选择器）、activeRule（路由激活规则）。

**start**：启动 qiankun 运行时。两个关键配置：
- `experimentalStyleIsolation: true` — 给子应用所有 CSS 规则加容器选择器前缀
- `prefetch: 'all'` — 空闲时预加载所有子应用资源

### 第 3 步：子应用接入（最关键的一步）

子应用必须做 4 件事才能被 qiankun 加载：

#### 3.1 UMD 输出

```js
// webpack.config.js
output: {
  library: 'sub-app-name',       // 唯一模块名
  libraryTarget: 'umd',          // UMD 格式
  chunkLoadingGlobal: 'webpackJsonp_sub_react', // webpack 5，全局变量名唯一
}
```

**为什么必须 UMD？** qiankun 通过 `eval` 执行子应用 JS 代码，它需要从模块中提取 `bootstrap`/`mount`/`unmount` 三个生命周期函数。UMD 格式让 webpack 把导出挂到一个全局变量上，qiankun 通过这个变量拿到函数。

**为什么 `chunkLoadingGlobal` 必须唯一？** 这是面试高频考点。

qiankun 的 Proxy 沙箱只能拦截 `eval()` 里的代码。但 webpack 的**动态 `import()` 产生的异步 chunk**是通过 `document.createElement('script')` 加载的——这些 `<script>` 标签直接注入真实 DOM，**绕过了 Proxy 沙箱**。chunk 文件里的 `self["webpackJsonp"]` 会直接污染真实的全局 `window`。

如果两个子应用的 `chunkLoadingGlobal` 同名，后加载的会覆盖先加载的，导致 ChunkLoadError。所以每个应用必须配唯一的名字。

| 机制 | 保护范围 | 盲区 |
|------|---------|------|
| qiankun Proxy 沙箱 | `eval()` 执行的入口 bundle | 动态 `<script>` 标签加载的 chunk |
| `chunkLoadingGlobal` | 所有 webpack chunk（含动态） | — |

两者是互补的，不是替代关系。

#### 3.2 `window.__POWERED_BY_QIANKUN__` 判断

```js
// 子应用入口
if (!window.__POWERED_BY_QIANKUN__) {
  // 独立运行模式：直接 mount 到 #root
  ReactDOM.createRoot(document.getElementById('root')).render(<App />);
}
// qiankun 模式：不自动渲染，等 mount(props) 被调用
```

这个分支避免"抢挂载"——如果没有这个判断，独立运行时正常，但 qiankun 环境下两个渲染入口同时执行会冲突。

#### 3.3 导出三个生命周期函数

```js
export async function bootstrap() {
  // 首次加载时调用一次，初始化全局资源（不渲染 UI）
}

export async function mount(props) {
  // 每次激活时调用，负责渲染
  const { container } = props;
  // container 是真实的 DOM 元素（不是字符串！）
  const root = createRoot(container.querySelector('#root'));
  root.render(<App />);
}

export async function unmount(props) {
  // 每次被切换走时调用，负责销毁 UI + 清理副作用
  const { container } = props;
  const root = createRoot(container.querySelector('#root'));
  root.unmount();
}
```

**关键细节**：`props.container` 是 DOM 元素不是字符串。必须用 `container.querySelector('#root')` 而不是 `document.getElementById('root')`——因为页面上有两个 `#root`（主应用一个，子应用一个），`getElementById` 是全局查找，会拿错。`container.querySelector` 把查找限定在当前子应用容器内。

#### 3.4 React vs Vue 生命周期差异

| | React 18 | Vue 3 |
|--|----------|-------|
| 应用实例 | 每次 mount 创建新 `createRoot` | 模块级 `let app = null`，mount 时创建 |
| 销毁 | `root.unmount()` | `app.unmount()` + `app = null` |
| 状态管理 | 不需要额外清理 | 需要 `app = null` 重置 |

Vue 3 必须用模块级变量，因为 `createApp` 返回的实例需要用变量引用才能在 unmount 时销毁。

### 第 4 步：登录与路由守卫

#### 路由结构

```jsx
<Routes>
  {/* 公开路由 — 必须放在通配路由前面 */}
  <Route path="/login" element={<Login />} />

  {/* 保护路由 — /* 捕所有非 /login 的路径 */}
  <Route path="/*" element={
    <ProtectedRoute>
      <DashboardLayout>
        <Routes>
          <Route path="/" element={<Home />} />
          <Route path="/user/*" element={<MicroAppContainer />} />
          <Route path="/content/*" element={<MicroAppContainer />} />
        </Routes>
      </DashboardLayout>
    </ProtectedRoute>
  } />
</Routes>
```

#### ProtectedRoute 守卫

```jsx
function ProtectedRoute({ children }) {
  if (!isAuthenticated()) {
    return <Navigate to="/login" replace />;
  }
  return children;
}
```

#### 关键设计决策

- **Cookie 而非 JWT**（Demo 限定）：简单直白，`isAuthenticated()` 只需检查 cookie 存在性，职责隔离
- **`<Navigate>` 而非 `useNavigate`**：路由守卫是声明式场景，`<Navigate>` 是声明式 API
- **`/login` 必须在 `/*` 外层**：如果放在 `/*` 内部，`ProtectedRoute` 包裹会导致死循环（访问 `/login` → 未登录 → 重定向 `/login` → ...）

#### qiankun 如何激活子应用

1. React Router 匹配 `/user/*` → 渲染 `<MicroAppContainer />`（空的 `<div id="micro-app-container" />`）
2. URL 变化 → qiankun 检测 `location.pathname` 与 `activeRule: '/user'` 匹配
3. qiankun **独立于 React Router** 激活子应用——它基于浏览器 URL 监听，不依赖 React Router 事件

### 第 5 步：Dashboard 布局

#### Layout Route 模式

```jsx
<Route element={<ProtectedRoute><DashboardLayout /></ProtectedRoute>}>
  <Route path="/" element={<Home />} />
  <Route path="/user/*" element={<MicroAppContainer />} />
  <Route path="/content/*" element={<MicroAppContainer />} />
</Route>
```

DashboardLayout 使用 `<Outlet />` 渲染子路由内容，结构：

```
DashboardLayout
├── Sidebar（220px 固定宽度）
└── .main
    ├── Header（56px 固定高度）
    └── .content
        └── <Outlet />  ← 子路由 + 微前端容器在这里
```

#### Sidebar activeKey 前缀匹配

```js
// 首页精确匹配（避免 /user 也满足 startsWith('/')）
// 其他菜单前缀匹配（/user/list 时"用户管理"也高亮）
function getActiveKey() {
  if (location.pathname === '/') return '/';
  return MENUS.find(m => m.key !== '/' && pathname.startsWith(m.key))?.key || '/';
}
```

#### 登出为什么 `window.location.reload()`

登出后需要完全刷新页面：
- 清除 qiankun 子应用残留状态（样式、事件监听、定时器）
- 在微前端架构下，完全刷新是最可靠的"重置"方式

### 第 6 步：应用间通信

qiankun 提供两套通信机制：

#### Props 透传（主 → 子，一次性的）

```js
// 主应用注册时传入
registerMicroApps([{
  name: 'sub-react',
  props: { namespace: 'user-mgmt', token: 'xxx' },
}]);

// 子应用接收
export async function mount(props) {
  console.log(props.token);       // 业务 props ✅
  console.log(props.onGlobalStateChange); // qiankun 自动注入 ✅
}
```

**qiankun 自动注入的 props**：`onGlobalStateChange`、`setGlobalState`（不包含 `getGlobalState`——qiankun 没有提供 getter）。

#### GlobalState（双向、运行时、响应式）

```js
// 主应用初始化
import { initGlobalState } from 'qiankun';
const actions = initGlobalState({ token: null, user: null });

// 登录时广播
actions.setGlobalState({ token: 'xxx', user: { name: 'admin' } });

// 子应用监听
props.onGlobalStateChange((state, prev) => {
  // state 变化时自动回调
}, true); // fireImmediately: true → 立即拿到当前快照
```

**封装层**：因为 qiankun 不提供 getter，所以封装 `globalState.js` 自己维护 `currentState` 副本：

```js
let currentState = {};
export function getGlobalState() { return currentState; }
export function updateGlobalState(s) { actions.setGlobalState({...currentState, ...s}); }
```

#### Redux 方案（进阶）

把主应用的 Redux store 通过 props 传给子应用：
- **React 子应用**：包一层 `<Provider store={store}>`，组件直接用 `useSelector`/`useDispatch`
- **Vue 子应用**：用 `app.provide('store', store)`，组件通过 `inject('store')` 获取

| | qiankun GlobalState | Redux |
|--|---------------------|-------|
| 读状态 | ❌ 无 getter | `store.getState()` ✅ |
| 调试 | ❌ 无 DevTools | ✅ Redux DevTools |
| 中间件 | ❌ 不支持 | ✅ 可堆叠 |
| 适用 | demo/简单场景 | 企业级复杂场景 |

### 第 7 步：CSS 隔离与性能优化

#### 7.1 CSS 样式隔离

**experimentalStyleIsolation**（推荐）：
- qiankun 为每个子应用容器生成唯一 `data-qiankun-xxx` 属性
- 给子应用所有 CSS 规则加前缀：`.title` → `[data-qiankun-app1] .title`
- 样式仅在子应用自己的容器内生效
- 兼容性好，性能开销低

**strictStyleIsolation**：Shadow DOM 强隔离，兼容性差，部分 UI 库弹窗（Portal）可能逃逸

#### 7.2 prefetch 预加载

```js
start({ prefetch: 'all' }); // 空闲时预加载所有子应用
```

通过 `requestIdleCallback` 在浏览器空闲时加载子应用静态资源，切换时直接使用缓存，减少白屏时间。

#### 7.3 webpack externals + CDN（公共依赖抽取）

**问题**：主应用、React 子应用、Vue 子应用各自打包 React/Vue → 同一份代码重复 3 次（720KB）

**解决**：

```js
// webpack.config.js
externals: {
  react: 'React',           // import React → window.React
  'react-dom': 'ReactDOM',  // import ReactDOM → window.ReactDOM
  vue: 'Vue',               // import Vue → window.Vue
}
```

```html
<!-- 主应用 index.html -->
<script src="https://unpkg.com/react@18/umd/react.production.min.js"></script>
<script src="https://unpkg.com/vue@3/dist/vue.global.prod.js"></script>
```

**关键设计**：公共依赖映射表 `loadables.js` 由主应用统一维护，三个应用共享。开发环境不启用 externals（方便热更新），生产环境启用（通过 CDN）。

**版本不一致怎么办？**
- 主应用版本 ≥ 子应用版本 → externals 安全（React 向后兼容）
- 子应用版本 > 主应用版本 → 取消子应用的 externals，随 bundle 打包自己的 React
- 企业最佳实践：组织层面统一 React 版本

### 第 8 步：部署与 CI/CD

#### Docker 化

**多阶段构建**：

```dockerfile
# 阶段 1：构建（node 环境）
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# 阶段 2：运行（nginx 环境）
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**为什么要多阶段？** Docker 的层只能加不能删。`rm -rf node_modules` 只是在新层标记"不要了"，旧层数据仍在，最终镜像 ~450MB。第二个 `FROM` 彻底切断，只带 dist/ 走，镜像 ~5.5MB。

**子应用 nginx 必须加 CORS 头**：

```nginx
add_header Access-Control-Allow-Origin *;
```

生产环境没有 webpack-dev-server，qiankun fetch 子应用资源时需要 CORS 头。

#### CI 工作流

```
git push → CI 自动触发 → npm ci → lint → test → build
```

- 按失败成本从低到高排列：lint（秒级）→ test → build（最慢）
- 每个子应用在各自仓库有独立 CI

#### CD 工作流

```
手动点 "Run workflow" → 检查 CI 是否通过（gh api）→ docker build → docker push → SSH 部署
```

**关键设计**：
- **手动触发 + CI 守卫**：不是 CI 绿了自动发布，而是手动点。点之后第一步调 GitHub API 查 CI 状态，不绿就报错退出
- **`concurrency`**：同一分支新 CD 自动取消旧跑，防止重复构建
- **`permissions`**：必须显式声明 `actions: read`，否则 `gh api` 查 CI 状态会 403
- **双重 tag**：`latest`（始终最新）+ commit SHA（精确回滚用）
- **三个项目独立部署**：主应用 + React 子应用 + Vue 子应用，各自独立镜像、独立 CD

## 三、面试高频问题

### Q1：qiankun 的原理是什么？

qiankun 基于 single-spa，核心流程：
1. `registerMicroApps` 注册子应用（name、entry、activeRule、container）
2. 监听 URL 变化（popstate/hashchange）
3. URL 匹配 `activeRule` → `fetch(entry)` 拉取子应用 HTML
4. 解析 HTML：抽出 `<script>` → eval 执行（Proxy 沙箱），抽出 DOM → 插入 container
5. 调用子应用导出的 `mount(props)`，传入 container DOM 引用
6. 切换时调用 `unmount()`，销毁子应用实例

### Q2：qiankun 的 JS 沙箱是怎么实现的？

**Proxy 沙箱**：qiankun 在 `eval()` 子应用代码前，创建一个 Proxy 对象代理 `window`。子应用代码里所有 `window.xxx` 的读写都被拦截，保存在 qiankun 内部的沙箱对象中，不会污染真实全局。卸载时恢复原始值。

**快照沙箱（SnapShotSandbox）**：兼容不支持 Proxy 的浏览器（IE 11）。activate 时遍历 window 保存快照，deactivate 时恢复。

**盲区**：动态 `<script>` 标签注入的代码绕过 Proxy，直接运行在真实全局。所以 `chunkLoadingGlobal` 必须唯一——它解决的是沙箱管不到的问题。

### Q3：qiankun 的 CSS 隔离怎么做的？

- **experimentalStyleIsolation**：劫持子应用的 `<style>` 和 `<link>`，给所有 CSS 规则加容器选择器前缀（`[data-qiankun-app1] .title`），样式只在子应用容器内生效
- **strictStyleIsolation**：把子应用包裹在 Shadow DOM 里，完全的样式隔离，但兼容性差

### Q4：子应用为什么必须 UMD 输出？

qiankun 通过 `eval` 执行子应用 JS，需要从中提取 `bootstrap`/`mount`/`unmount` 三个函数。普通 IIFE 打包后外部无法访问导出。UMD 格式把这三个函数挂到一个全局变量上（如 `window['sub-react']`），qiankun 通过这个变量拿到生命周期函数。

### Q5：`chunkLoadingGlobal` 是什么？为什么要配成唯一的？

webpack 5 的配置项（webpack 4 叫 `jsonpFunction`）。webpack 用 JSONP 模式加载动态 import 产生的异步 chunk——chunk 文件执行 `self["webpackJsonp"].push([chunkId, modules])`。所有应用在同一个页面，如果都用默认的 `webpackJsonp`，后加载的覆盖前面的，导致 ChunkLoadError。每个应用配唯一名字解决。

### Q6：`mount(props)` 里的 `container` 是什么？为什么不能用 `document.getElementById`？

`props.container` 是 DOM 元素，是主应用 `registerMicroApps` 配置中 `container: '#micro-app-container'` 对应的真实 DOM 节点。

**不能用 `document.getElementById`** 的原因：页面上有两个 `#root`（主应用一个、子应用一个），`getElementById` 是全局查找，可能拿错。必须用 `container.querySelector('#root')` 把查找限定在当前子应用容器内。

### Q7：应用间通信怎么做的？

两套机制：
1. **Props 透传**（主 → 子，一次性）：`registerMicroApps` 的 `props` 字段，子应用 `mount(props)` 接收
2. **GlobalState**（双向，运行时）：qiankun 的 `initGlobalState`/`onGlobalStateChange`/`setGlobalState`，响应式广播
3. **进阶**：Redux store 通过 props 传递，React 子应用包 Provider，Vue 子应用用 provide/inject

### Q8：微前端的性能优化怎么做？

| 优化 | 手段 |
|------|------|
| 公共依赖去重 | webpack externals + CDN，React/Vue 只加载一次 |
| 子应用预加载 | `prefetch: 'all'`，空闲时加载，切换零等待 |
| 样式隔离 | `experimentalStyleIsolation`，避免样式冲突导致的渲染异常 |
| 组件懒加载 | 子应用内部 `React.lazy` / `defineAsyncComponent` |
| 独立部署 | 改子应用不重启主应用，qiankun fetch 自动拿到新版本 |

### Q9：子应用独立部署后，主应用怎么知道子应用更新了？

**不需要知道**。qiankun 每次激活子应用时，重新 `fetch` 子应用的 HTML 入口，浏览器拿到的是子应用服务器上的最新版本。这是微前端独立部署的核心价值——子应用更新不需要通知主应用，不需要整体重启。

### Q10：遇到过什么坑？

1. **`chunkLoadingGlobal` 冲突**：两个子应用默认都叫 `webpackJsonp`，异步 chunk 加载失败。解决：每个应用配唯一名字
2. **`container.querySelector` vs `document.getElementById`**：页面上多个 `#root`，全局查找拿错。解决：限定在 container 内查找
3. **子应用 CORS 报错**：生产环境 Nginx 没配 `Access-Control-Allow-Origin`，qiankun fetch 失败。解决：子应用 Nginx 加 CORS 头
4. **`gh api` 403**：CD 里查 CI 状态时报权限错误。解决：workflow 顶层加 `permissions.actions: read`
5. **CSS 样式污染**：子应用的全局样式互相影响。解决：`experimentalStyleIsolation` + 主应用 CSS Modules

### Q11：为什么选 qiankun 而不是 Module Federation？

| | qiankun | Module Federation |
|--|---------|-------------------|
| 框架无关 | ✅ 支持 React/Vue/Angular 混合 | ✅ 但打包工具须统一 webpack 5 |
| 技术栈迁移 | 渐进式，老项目改造成本低 | 需要 webpack 5 |
| 隔离程度 | JS 沙箱 + CSS 隔离 | 共享作用域，需自行处理 |
| 学习曲线 | 中等 | 较高 |
| 适合场景 | 存量系统整合、异构技术栈 | 从零构建的微前端、同构技术栈 |

### Q12：微前端的优缺点？

**优点**：
- 独立开发、独立部署、独立技术栈
- 增量升级（老系统逐步迁移）
- 团队自治，不同团队负责不同子应用

**缺点**：
- 整体复杂度上升（沙箱、通信、路由）
- 性能开销（加载多个运行时）
- 样式隔离不完美（弹窗 Portal 可能逃逸）
- 调试体验不如单体应用
