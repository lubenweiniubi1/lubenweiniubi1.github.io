# 组件封装方法论 —— 架构面试高频题

> 典型面试题：「20 个页面都用 antd table，5 个有独特逻辑，怎么封装？」  
> 这不是一个代码题，是一道**架构决策题**。面试官想看你从识别问题 → 分析边界 → 设计接口 → 工程化 → 测试 → 部署 → 维护的完整思维链。

---

## 一、确认动机与范围（最关键的一步）

写代码之前先回答：**你到底在解决什么问题？**

### 三个层次，越往上投入越大

```
Level 3: 跨项目/跨团队的通用组件库
        （独立包、独立仓库、独立文档站、独立发版节奏）
           ▲
Level 2: 项目级复用组件
        （本项目内 monorepo package，服务于本项目的多个页面）
           ▲
Level 1: 局部封装（降低单个页面的复杂度）
        （一个文件里的 <MyTable>，把重复的 columns 和配置抽出来）
```

### 如何选择？

| 如果 | 选 |
|------|-----|
| 只是为了这个页面少写几行代码 | Level 1 — 局部封装 |
| 项目内 5+ 页面有相同的列配置/操作逻辑 | Level 2 — 项目级组件 |
| 多个项目、多个团队都要用，且需要统一的视觉和行为 | Level 3 — 独立组件库 |

### antd table 场景分析

20 个页面用 antd table，其中 5 个有独特逻辑：

- 15 个页面：分页 + 搜索 + 列配置 + 操作列，几乎一样 → **抽通用层**
- 5 个页面：有各自特殊的筛选逻辑、自定义渲染、多选树形等 → **保留扩展点**

**核心决策**：Level 2（项目级复用组件），因为跨 20 个页面但还在同一个项目内。如果未来其他项目也要用，再升级到 Level 3。

**错误示范**：一上来就开 monorepo + 独立文档站 + npm 发布 → 过度设计，浪费 80% 的精力在没人用的功能上。

## 二、分析边界（组件做什么、不做什么）

**铁律：越通用的组件确定性越低，边界越窄。**

### 边界分析框架

```
┌─────────────────────────────────────────┐
│            组件应该做的事                  │
│                                         │
│  ✅ 渲染逻辑（columns、数据映射）          │
│  ✅ 通用交互（排序、分页、行选择）          │
│  ✅ 统一的 UI 行为（loading、empty、error）│
│  ✅ 性能优化（虚拟滚动、按需渲染）           │
│  ✅ 可访问性（a11y）                     │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│            组件不应该做的事                 │
│                                         │
│  ❌ 业务数据获取（API 调用留给页面）        │
│  ❌ 业务状态管理（页面级的 context/state）  │
│  ❌ 页面路由逻辑                          │
│  ❌ 不经过接口的硬编码业务规则              │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│         灰色地带（通过扩展点解决）            │
│                                         │
│  ⚡ 5 个特殊页面的独特逻辑                  │
│  ⚡ 自定义筛选器、行操作按钮               │
│  ⚡ 列的自定义渲染                        │
│  → 不写死在组件里，用插槽/渲染函数暴露出去    │
└─────────────────────────────────────────┘
```

### antd table 场景的边界清单

| 分类 | 组件做 | 页面做 |
|------|--------|--------|
| 列定义 | 提供默认列配置，允许页面覆盖/追加 | 传递自定义 columns |
| 数据请求 | 不碰 | 页面自己 fetch，把 dataSource 传进来 |
| 分页 | 内置 pagination 逻辑，暴露 onChange | 页面决定 total、current 的值 |
| 行选择 | 内置 rowSelection 逻辑 | 页面决定选中后的业务操作 |
| 操作列 | 提供默认操作（查看/编辑/删除），允许插槽替换 | 5 个特殊页面传入自定义操作 |
| 搜索栏 | 提供通用搜索配置 | 特殊页面可传自定义 searchForm |

## 三、设计接口（Props、Slots、Events）

接口设计的黄金法则：**让简单的事情保持简单，让复杂的事情成为可能。**

### Props 设计

```typescript
interface ProTableProps<T = any> {
  // ── 数据（页面传入） ──
  dataSource: T[];
  loading?: boolean;

  // ── 列定义（有默认值、可覆盖） ──
  columns?: ColumnType<T>[];       // 页面完全自定义
  columnPreset?: 'default' | 'compact' | 'full'; // 预设列方案
  extraColumns?: ColumnType<T>[];  // 在默认列基础上追加

  // ── 分页（内置逻辑、页面控数据） ──
  pagination?: PaginationConfig;   // 页面传入 total/current
  onPageChange?: (page, pageSize) => void;

  // ── 搜索（内置搜索栏、可被替换） ──
  searchConfig?: SearchField[];    // 简单搜索：[{ name: 'keyword', label: '关键词' }]
  onSearch?: (values) => void;

  // ── 行操作（有默认、可被替换） ──
  rowActions?: ActionConfig[];     // [{ type: 'view', onClick: ... }]
  onRowAction?: (action, record) => void;

  // ── 工具条 ──
  toolbar?: ReactNode;             // 页面完全自定义的工具条区域
  onCreate?: () => void;           // 新建按钮

  // ── 扩展出口 ──
  renderBefore?: () => ReactNode;  // table 前面的区域
  renderAfter?: () => ReactNode;   // table 后面的区域
}
```

**设计要点**：
- `dataSource` / `loading` —— 数据由页面控制，组件不越界
- `columnPreset` + `extraColumns` —— 简单页面一行配置搞定，复杂页面可以完全自定义
- `onPageChange` / `onSearch` —— 事件回调，不直接改状态
- `renderBefore` / `renderAfter` —— 兜底出口，覆盖不了的场景还能自己塞东西

### Slots（插槽 / Render Props）

```typescript
// 关键：让 5 个特殊页面能注入独特逻辑，但不强迫 15 个通用页面写多余代码

interface ColumnRenderSlots<T> {
  // 按列名覆盖渲染
  [columnKey: string]: (text: any, record: T, index: number) => ReactNode;
}

// 使用
<ProTable
  columns={baseColumns}
  columnSlots={{
    status: (text, record) => <StatusBadge status={record.status} />,  // 特殊页面的自定义渲染
    // 其他列用默认渲染
  }}
  rowActionSlots={{
    extra: (record) => <SpecialButton record={record} />,  // 特殊操作
  }}
/>
```

### Events（回调）

```typescript
// 事件命名规范：on + 名词 + 动词
onRowClick?: (record: T, index: number) => void;
onSelectionChange?: (selectedKeys: Key[], selectedRows: T[]) => void;
onSortChange?: (sorter: SorterResult<T>) => void;
onFilterChange?: (filters: Record<string, any>) => void;

// 原则：组件不消费事件，只转发给页面
// ❌ 组件内：onRowClick → 弹详情弹窗 → 越界了
// ✅ 组件内：onRowClick → 调用 props.onRowClick → 页面决定弹不弹
```

## 四、代码实现与工程化

### 目录结构（按 Level 决定）

```
Level 1（局部封装）：
src/pages/xxx/components/MyTable.tsx          ← 一个文件

Level 2（项目级）：
packages/pro-table/
├── src/
│   ├── ProTable.tsx          ← 主组件
│   ├── hooks/
│   │   ├── usePagination.ts
│   │   └── useSelection.ts
│   ├── components/
│   │   ├── SearchBar.tsx
│   │   └── Toolbar.tsx
│   ├── types.ts
│   └── index.ts
├── package.json
└── README.md

Level 3（跨项目组件库）：
packages/pro-table/            ← monorepo 中的一个包
├── src/                       ← 同上
├── docs/                      ← 文档站
├── __tests__/                 ← 测试
├── .storybook/                ← Storybook
├── CHANGELOG.md               ← 版本变更
└── package.json
```

### 版本管理

| 范围 | 版本策略 | 原因 |
|------|---------|------|
| Level 1 | 无独立版本，随项目走 | 不对外暴露 |
| Level 2 | monorepo 统一版本 or 独立 0.x | 用 changesets 管理 |
| Level 3 | **SemVer 严格遵守** | 外部消费者依赖你 |

**SemVer 快速回顾**：
- `1.0.0 → 1.0.1`：Bug fix（patch）
- `1.0.0 → 1.1.0`：新功能，向后兼容（minor）
- `1.0.0 → 2.0.0`：Breaking change（major）

## 六、部署与分发

### 按层级选择分发方式

| 层级 | 分发方式 | 消费者 |
|------|---------|--------|
| Level 1 | 不部署，源码在项目内 | 当前页面 |
| Level 2 | monorepo 内其他包直接 `import` | 项目内其他页面 |
| Level 3 | npm 发布（私服 or 公服） | 其他项目/团队 |

### Level 3 部署流水线

```
Git Tag v1.0.0
    │
    ▼
CI 触发
    ├── lint + test + build
    ├── 生成 CHANGELOG（从 conventional commits）
    └── npm publish（private registry）
         │
         └── 可选：同时推到 CDN（UMD bundle）
```

- **私服**（Verdaccio / GitLab npm registry）：公司内部用的首选，安全、快
- **公服**（npmjs.com）：开源或跨公司使用
- **CDN**：如果组件需要给非 Node 环境用（老旧系统、静态页面），打一个 UMD 包推到 CDN

**关键决策**：Level 2 不需要发布，monorepo 下源码引用就够了。只有确定跨项目复用时才走 npm 发布。

## 七、维护与升级

### Breaking Change 处理策略

```
发现需要改 Props 接口
    │
    ▼
能向后兼容吗？
    ├── 能 → 旧 prop 保留 + 标 deprecated → 新 prop 生效
    │         等 2 个大版本后再删旧 prop
    │
    └── 不能 → 发 major 版本
               CHANGELOG 写清楚迁移步骤
               提供 codemod（如果影响面大）
```

### 兼容性示例

```typescript
// v1：pagination 是 boolean
<ProTable pagination={true} />

// v2：pagination 改为对象，但保留 boolean 兼容
interface PaginationConfig {
  pagination?: boolean | PaginationOptions;  // 两种都接受
}
// 内部处理：
function normalizePagination(pagination) {
  if (typeof pagination === 'boolean') {
    console.warn('[ProTable] pagination as boolean is deprecated, use object instead');
    return pagination ? { defaultPageSize: 20 } : false;
  }
  return pagination;
}
```

### 维护清单

| 维度 | 频率 | 做什么 |
|------|------|--------|
| antd 升级 | 每个 antd major 版本 | 检查 Table API 是否 break，跑全量测试 |
| 依赖审计 | 每季度 | `npm audit`，升级安全补丁 |
| 新需求收口 | 持续 | 页面提的新功能先判断"是只有你一个页面需要还是 3+ 页面需要"，只有后者才加到组件里 |
| 性能回归 | 每次发版前 | 跑一次大数据量（1000+ 行）的渲染性能测试 |
| 文档更新 | 每次 API 变更 | 同步更新 README 和 Storybook |

### 新功能收口机制（防膨胀）

```
页面提需求："ProTable 能不能支持树形数据？"
    │
    ▼
第一步：这个功能几个页面需要？
    ├── 1 个 → ❌ 不加。页面自己用插槽/扩展点实现
    ├── 3+ 个 → ✅ 加
    └── 不确定 → 先让页面自己实现，等第 3 个页面需要时再收口
```

**核心原则**：组件的功能只应该从**被多个消费者验证过的需求**中长出来，而不是从想象中。

## 面试回答模板（antd table 场景）

面试官：「20 个页面都用 antd table，5 个有自己特殊的逻辑，你怎么封装？」

```
我不会一上来就写代码。我会分七步思考：

第一步，确认动机和范围。20 个页面、同一个项目，这是项目级复用（Level 2），
不需要独立 npm 包和文档站。但需要比"每个页面复制粘贴"好一点的抽象。

第二步，分析边界。组件管渲染和通用交互（分页、排序、行选择），
组件不管数据获取和业务状态。那 5 个特殊页面不能写死在组件里，
要通过插槽和渲染函数暴露扩展点。

第三步，设计接口。Props 分四类：数据（dataSource/loading）、配置（columns/pagination）、
回调（onPageChange/onSearch）、扩展出口（columnSlots/rowActionSlots 给 5 个特殊页面用）。

第四步，实现。项目内 monorepo 的 packages/pro-table，带 usePagination 等 hooks。

第五步，测试。hooks 优先单测（usePagination/useSelection），关键流程做集成测试
（搜索→分页→选择→操作），CI 设 80% 覆盖率门禁。

第六步，部署。项目内直接源码引用就行，不发布 npm。如果未来其他项目要用了再升级到 Level 3。

第七步，维护。新功能收口机制——单个页面需要的功能不加，3 个以上页面需要才加。
antd 升级时跑全量测试。Breaking change 先 deprecate 后删除，给消费者缓冲期。
```

## 一句话总结

> **封装组件的本质不是写代码，是做减法——决定什么不封装、什么留给页面、什么留给时间验证。**
