## 一、概念理解类

### 1. 什么是 Vibe Coding？前端开发中如何应用？

**Vibe Coding** 是 Andrej Karpathy 提出的概念，核心思想是：开发者通过自然语言描述需求意图（"vibe"），让 AI 生成代码，开发者更多地扮演**审查者**和**架构师**角色，而非逐行手写代码。

前端中的应用场景：
- **快速搭建 UI 组件**：描述布局、样式、交互，AI 生成可运行的 JSX/TSX
- **样板代码生成**：表单校验、API 请求封装、路由配置等重复性工作
- **样式生成**：用自然语言描述视觉效果，AI 输出 CSS/Tailwind
- **数据转换逻辑**：描述数据格式转换需求，AI 生成处理函数

**核心公式**：`清晰的需求描述 + AI 生成 + 人工审查优化 = 高效交付`

---

### 2. Vibe Coding 与传统前端开发的本质区别

| 维度 | 传统开发 | Vibe Coding |
|------|---------|-------------|
| 工作方式 | 手动编写每一行代码 | 描述意图，AI 生成 + 人工审查 |
| 技能重心 | 语法记忆、API 熟练度 | Prompt 设计、架构判断、代码审查 |
| 效率瓶颈 | 编码速度 | 需求表达清晰度 |
| 质量保障 | 开发者全权负责 | 开发者审查 + AI 辅助检测 |
| 学习曲线 | 需要记忆大量 API | 需要培养"描述能力"和判断力 |

本质区别：**角色从"执行者"转变为"指挥者"**。开发者不再花时间在"怎么写"上，而是聚焦在"做什么"和"为什么这样做"。

## 二、前端实战类

### 3. 如何用 AI 快速生成响应式布局组件？

**回答要点**：

```
Prompt 示例：
"用 React + CSS Grid 实现一个三栏布局：
- 桌面端：左 200px 固定 | 中间自适应 | 右 200px 固定
- 平板端（<1024px）：左侧隐藏，中间和右侧 2:1 比例
- 手机端（<768px）：单栏堆叠
- 使用 CSS Grid 实现，避免用 position absolute"
```

关键技巧：
1. **明确断点值**：给出具体像素值，不要只说"响应式"
2. **指定技术方案**：Grid 还是 Flex，AI 不会猜你的偏好
3. **描述降级策略**：每个断点的具体行为
4. **迭代优化**：第一版生成后，针对性地提修改（"左边栏加个阴影"、"中间区域加最大宽度 1200px 并居中"）

个人实践：DeepSeek Coder 在 CSS Grid 布局生成上表现很好，我会先给布局骨架，再逐步细化样式细节。

### 4. 如何让 AI 生成符合设计规范的 UI 组件？

**核心原则：Prompt 中提供完整的设计约束。**

```
Prompt 示例：
"用 React + TypeScript 生成一个按钮组件 Button：
1. 遵循 Ant Design 规范
2. 支持 variant: 'primary' | 'default' | 'danger' | 'text'
3. 支持 size: 'small' | 'middle' | 'large'
4. primary 主色 #4f6ef7，danger 主色 #e5484d
5. 圆角 6px，hover 时加深 10%
6. 支持 loading 状态，显示 spinner 并禁用点击
7. 支持 disabled 状态，opacity 0.5
8. 使用 forwardRef 转发 ref
9. 完整的 TypeScript 类型定义
10. 导出为 named export"
```

关键技巧：
- 给出具体的色值、尺寸、圆角数值
- 描述交互状态（hover / active / disabled / loading）
- 指定组件 API 设计（props 接口）
- 要求完整的 TypeScript 类型

### 5. 如何处理 AI 生成的前端代码性能问题？

**回答策略**：展示你会审查 AI 代码 + 有性能优化知识。

常见 AI 代码性能坑点及处理：

1. **不必要的重渲染**
   - AI 常生成内联函数/对象，导致子组件重复渲染
   - 处理：用 `useCallback` / `useMemo` 包裹，或用 React.memo

2. **低效的列表渲染**
   - AI 可能生成 `.map()` 但忘记 `key`，或 key 用 index
   - 处理：补充稳定的唯一 key

3. **过度使用 useEffect**
   - AI 倾向于把逻辑都放进 useEffect
   - 处理：审查是否真的需要 effect，能否用事件处理器替代

4. **大体积依赖**
   - AI 可能引入 lodash 等整个库
   - 处理：改为按需引入 `import debounce from 'lodash/debounce'`

**面试时可以说**："每次 AI 生成代码后，我会过一遍性能 checklist：重渲染、列表 key、useEffect 合理性、bundle 体积，这是我的标准审查流程。"

## 三、框架专项类

### 6. 如何用 AI 辅助 React/Vue 组件开发？

**React 示例**：

```
Prompt：
"用 React + TypeScript 实现一个 useDebounce hook：
1. 接收 value 和 delay（默认 300ms）
2. delay 内 value 变化不更新，等稳定后才更新 debouncedValue
3. 组件卸载时清除 timer，避免内存泄漏
4. 完整的 TypeScript 泛型支持"
```

AI 辅助 React 开发的最佳实践：
- **先描述接口，再描述实现**：先定义 props/return type，再描述行为
- **明确边界条件**：空状态、loading、error、极端值
- **要求 hooks 规范**：依赖数组完整、清理函数、SSR 兼容

我的工作流：先用 AI 生成组件骨架 → 审查 hooks 依赖和类型 → 补充测试用例 → 让 AI 再优化。

---

### 7. 如何让 AI 生成 TypeScript 类型的前端代码？

```
Prompt 示例：
"生成一个通用的 API 请求函数 request<T>：
1. 泛型 T 为返回数据类型
2. 参数：url: string, options?: { method, body, headers }
3. 返回 Promise<{ data: T; code: number; message: string }>
4. 自动处理 JSON 解析
5. 非 2xx 状态码抛出带类型的错误
6. 完整的错误类型定义 ApiError { code, message, status }"
```

TypeScript 特定技巧：
- **先定义 interface/type**，再要求实现
- 要求 AI 使用 `unknown` 而非 `any`
- 指定严格模式偏好（`strict: true`）
- 要求导出类型供外部使用

我的经验：DeepSeek Coder 的 TypeScript 生成质量很高，尤其是类型推断和泛型约束。

---

### 8. 如何处理 AI 生成代码的跨浏览器兼容性？

AI 默认生成现代浏览器代码，需要人工补充兼容性考量：

1. **CSS 兼容性**
   - AI 可能使用 `gap` in Flexbox（Safari 旧版不支持）
   - 处理：检查 Can I Use，必要时用 `margin` 替代或用 Autoprefixer

2. **JS API 兼容性**
   - AI 可能用 `Array.at()`、`Object.hasOwn()` 等较新 API
   - 处理：确认 babel/polyfill 配置是否覆盖

3. **检测方法**
   - 让 AI 标注使用了哪些 ES20xx 特性
   - 结合项目 browserslist 配置检查

**面试时**："我的做法是让项目工程的 babel + browserslist 处理大部分兼容性，AI 生成的代码作为源码提交，构建工具自动转译。对于 CSS 兼容性，会结合 Autoprefixer 和手动检查。"

## 四、工作流优化类

### 9. 如何在团队中推广 AI Coding 工作流？

从我在 Thoughtworks 推行 ESLint 和 Code Review 的经验出发：

1. **从个人提效开始**
   - 自己先用，产出具体数据（"这个模块 AI 辅助下 3 天完成，纯手写预估 5 天"）

2. **建立团队规范**
   - AI 生成代码必须经过 Code Review（和手写代码同等标准）
   - 制定 Prompt 模板库，团队共享高质量 Prompt

3. **知识沉淀**
   - 记录哪些场景 AI 表现好（CRUD、样式、工具函数）
   - 记录哪些场景需要人工主导（复杂业务逻辑、架构决策）

4. **降低门槛**
   - 分享实际案例，而非空讲概念
   - 鼓励团队成员从低风险任务开始尝试

**展示领导力**："推动工具落地不是强推，而是让大家看到提效 + 建立安全感（代码审查不放松）。"

---

### 10. 如何控制 AI 生成前端代码的成本？

1. **Prompt 优化**
   - 一次说清楚需求，减少多轮对话
   - 用"不要..."排除不需要的方案，缩小搜索空间

2. **缓存复用**
   - 常用组件模板沉淀为团队 Prompt 库
   - 同类型问题直接复用已验证的 Prompt

3. **选择性使用**
   - **适合 AI**：样板代码、工具函数、样式生成、单元测试
   - **不适合 AI**：核心业务逻辑、架构设计、安全敏感代码

4. **批量处理**
   - 攒一批同类任务（如 5 个组件的单元测试）一起生成

## 五、质量保证类

### 11. 如何验证 AI 生成的前端代码质量？

我的**三层验证体系**：

```
Layer 1: 静态检查
├── ESLint 检查（语法规范）
├── TypeScript 编译（类型安全）
└── Prettier 格式化（风格一致）

Layer 2: 人工审查
├── 逻辑正确性（边界条件、错误处理）
├── 性能审查（重渲染、内存泄漏）
├── 可读性（命名、拆分粒度）
└── 安全性（XSS、注入）

Layer 3: 自动化测试
├── 单元测试（Jest + React Testing Library）
├── 集成测试（关键用户流程）
└── E2E 测试（核心业务场景）
```

面试话术："我不会因为代码是 AI 生成的就降低标准。TS 类型检查 + ESLint + 人工 Review + 测试用例，四道闸门一道不少。"

---

### 12. 如何处理 AI 生成代码的安全问题？

前端常见安全隐患及审查要点：

1. **XSS 防护**
   - AI 可能生成 `dangerouslySetInnerHTML` 
   - 审查：确认输入经过 DOMPurify 等库消毒

2. **敏感信息泄露**
   - AI 可能在代码中硬编码 API key / token
   - 审查：敏感信息必须走环境变量

3. **输入验证**
   - AI 生成的表单可能缺少前端校验
   - 补充：必填、格式、长度限制

4. **CSRF / CORS**
   - AI 不会主动考虑跨域安全
   - 审查：确认请求携带正确的 credentials 配置

我会在 Review Checklist 里专门加一条"安全检查"。

## 六、前端特有问题

### 13. 如何用 AI 优化前端性能？

```
Prompt 示例：
"分析以下 React 组件的性能问题并给出优化方案：
- 提供组件代码
- 要求分析：不必要的重渲染、大计算量、bundle 体积
- 给出具体的优化代码，标注改了什么、为什么改"
```

AI 在性能优化中的角色：
- **发现者**：快速扫描代码找常见模式（缺少 key、未 memo）
- **建议者**：提供多种优化方案（懒加载 vs 预加载、虚拟滚动 vs 分页）
- **不是决策者**：选哪个方案需要开发者根据业务判断

实际案例：我曾让 DeepSeek Coder 分析一个列表页的首屏性能，它指出了缺少 `React.memo` 和图片懒加载，优化后 FCP 降低了 40%。

### 14. 如何让 AI 生成无障碍访问（ARIA）友好的代码？

```
Prompt 示例：
"生成一个模态框组件 Modal，要求：
1. 打开时焦点自动移到模态框内第一个可聚焦元素
2. Esc 键关闭
3. 关闭后焦点回到触发按钮
4. role='dialog', aria-modal='true', aria-labelledby
5. 点击遮罩层关闭
6. 完整的键盘导航支持（Tab 循环）"
```

审查要点：
- 语义化标签（`<button>` 而非 `<div onclick>`）
- ARIA 属性完整且正确
- 键盘可操作（Tab、Enter、Esc）
- 屏幕阅读器友好（alt 文本、label 关联）

---

### 15. 如何处理 AI 生成代码的样式冲突？

AI 生成样式时容易忽略项目已有的 CSS 架构，解决方案：

1. **在 Prompt 中明确样式方案**
   - "使用 CSS Modules" / "使用 Tailwind CSS" / "使用 styled-components"
   - 这样 AI 生成的代码自然带作用域隔离

2. **审查要点**
   - 全局样式是否污染（避免标签选择器）
   - z-index 是否合理（避免随意设 9999）
   - 是否复用了已有的 design tokens / CSS 变量

3. **团队约定**
   - 统一 CSS 架构（BEM / CSS Modules / CSS-in-JS）
   - 在 Prompt 模板中固化样式规范

**最佳实践**："最好的样式冲突处理是在 Prompt 阶段就指定样式方案，比事后修复高效得多。"

## 面试准备建议

1. **准备实际案例**：展示你如何用AI解决过的真实前端问题
2. **掌握Prompt技巧**：学会编写清晰、具体的前端需求描述
3. **理解前端最佳实践**：能够判断AI生成代码的质量
4. **关注AI工具链**：了解Copilot、Cursor等前端AI工具
5. **平衡人机协作**：展示何时用AI、何时手动优化的判断力

**核心考察点**：面试官想看到的不仅是你会用AI，更是你作为前端工程师的专业判断力和系统思维能力。
