# React 事件系统

> 合成事件 + 事件委托 + 虚拟 DOM 冒泡模拟，React 事件系统的三个核心支柱。

---

## 一、合成事件（SyntheticEvent）

### 为什么需要？

浏览器事件 API 有兼容差异（IE vs Chrome vs Firefox）。React 封装了一层 **SyntheticEvent**，对所有浏览器暴露统一接口。

```jsx
function handleClick(e) {
  // e 是 SyntheticEvent，不是原生 Event
  // 但 API 和原生一致
  e.preventDefault();
  e.stopPropagation();
  console.log(e.target);        // 事件源（触发事件的真实 DOM）
  console.log(e.currentTarget); // 当前处理事件的元素（绑定 handler 的那个）
  console.log(e.nativeEvent);   // 底层原生事件对象
}
```

### target vs currentTarget

| 属性 | 含义 | 类比 |
|------|------|------|
| `e.target` | 事件最初发生的那个具体元素 | 谁触发 |
| `e.currentTarget` | 当前正在处理事件的那个元素 | 谁处理 |

## 二、事件委托（Event Delegation）

### React 不给每个元素单独绑定事件

```jsx
// ❌ 原生方式（每个按钮独立绑定，内存开销大）
button1.addEventListener('click', handler1);
button2.addEventListener('click', handler2);

// ✅ React 方式（所有事件委托到根节点）
// React 16：document.addEventListener('click', dispatchEvent)
// React 17+：rootContainer.addEventListener('click', dispatchEvent)
```

**收益**：减少内存占用 / 动态元素自动生效 / 统一事件调度。

### 事件委托位置的变化

| 版本 | 委托位置 | 为什么 |
|------|---------|--------|
| React 16 | `document` | 历史设计 |
| React 17+ | 根容器节点（`#root`） | 微前端多版本共存、避免全局事件冲突 |

```
React 16：                    React 17+：
document                      #root
  └─ #root                      └─ <App/>
       └─ <App/>
            └─ <button>              └─ <button>
↑ 事件委托到这里              ↑ 事件委托到这里
```

## 三、事件冒泡模拟

React 的事件冒泡**不是真正的 DOM 冒泡**——因为所有事件只绑定在根节点上。React 通过**遍历 Fiber 树**来模拟冒泡。

### 核心流程（简化）

```
1. 原生事件在 <button> 上触发
2. 冒泡到根节点，dispatchEvent 被调用
3. dispatchEvent 拿到 e.target → 找到对应 Fiber 节点
4. 从目标 Fiber 向上遍历 return 链，逐个收集事件处理器
5. 按冒泡顺序（目标→根）依次执行收集到的 handler
6. 如果某个 handler 调了 e.stopPropagation()，跳过后续 handler
```

### collectListeners 核心逻辑

```js
function collectListeners(startFiber, eventType) {
  const listeners = [];
  let current = startFiber;

  while (current !== null) {
    const props = current.memoizedProps;
    const propName = reactEventToProp(eventType); // 'click' → 'onClick'
    if (props[propName]) {
      listeners.push({
        instance: current.stateNode, // 真实 DOM
        listener: props[propName],   // 事件处理器
      });
    }
    current = current.return; // 向父 Fiber 遍历
  }

  return listeners; // 顺序：目标 → 根（冒泡顺序）
}
```

### 关键：Fiber 和 DOM 的关联

React 创建 DOM 时会在 DOM 节点上存 Fiber 引用：

```js
domNode._reactInternalFiber = fiber;           // 内部属性，含随机后缀
// dispatchEvent 时：
const fiber = e.target._reactInternalFiber;    // 从真实 DOM 找到 Fiber
```

## 四、捕获阶段（onClickCapture）

React 的捕获阶段同样是在根节点监听**原生冒泡**来实现的，不是真正的 DOM 捕获。

**实现方式**：收集 listeners 后，捕获阶段的手动反转顺序（根→目标）：

```jsx
<div onClickCapture={() => console.log('捕获')}>  {/* 先执行 */}
  <button onClick={() => console.log('冒泡')}>    {/* 后执行 */}
    Click
  </button>
</div>

// 输出：捕获 → 冒泡
```

## 五、事件池（Event Pool）—— React 16 已废弃

React 16 复用事件对象以节省内存，回调执行后清空属性：

```jsx
// React 16：必须调用 e.persist() 才能异步访问
function handleClick(e) {
  e.persist();  // 必须！否则异步访问全空
  setTimeout(() => console.log(e.target), 1000);
}

// React 17+：直接访问，事件池已移除
function handleClick(e) {
  setTimeout(() => console.log(e.target), 1000); // 正常工作
}
```

## 六、合成事件 vs 原生事件

| 特性 | 合成事件 | 原生事件 |
|------|---------|---------|
| 命名 | 驼峰 `onClick` | 全小写 `onclick` |
| 绑定位置 | 委托到根节点 | 直接绑到元素 |
| 事件对象 | SyntheticEvent | 原生 Event |
| 阻止冒泡 | `e.stopPropagation()` 只影响 React 内部 | `e.stopPropagation()` 阻止原生冒泡 |
| 阻止原生冒泡 | `e.nativeEvent.stopImmediatePropagation()` | — |

### 注意：合成事件的 stopPropagation 不阻止原生冒泡

```jsx
function handleClick(e) {
  e.stopPropagation(); // 只阻止 React 树的冒泡模拟，不阻止原生事件冒泡
}

// 如果需要阻止原生冒泡（影响 root 外的 listener）：
function handleClick(e) {
  e.nativeEvent.stopImmediatePropagation();
}
```

## 七、Portal 中的事件冒泡

Portal 把 DOM 渲染到父组件之外，但**事件冒泡仍然按 React 组件树**，不是 DOM 树。

```jsx
<div onClick={() => console.log('父组件收到冒泡')}>
  {ReactDOM.createPortal(
    <button onClick={() => console.log('Portal 内按钮')}>Click</button>,
    document.body  // DOM 上在 <body> 下，但事件冒泡仍到父 <div>
  )}
</div>

// 点击按钮 → 输出：
// Portal 内按钮
// 父组件收到冒泡 ← 虽然 DOM 不在 <div> 下，事件仍冒泡过来了
```

**原因**：React 的事件冒泡沿 Fiber 的 `return` 链，不是 DOM 的 `parentNode` 链。Portal 的 Fiber 仍在父组件的子树中。

## 八、React 18 的事件优先级

React 18 根据事件类型分配优先级：

| 事件类型 | 示例 | 优先级 |
|---------|------|--------|
| 离散事件 | click、input、keydown | 高（同步处理） |
| 用户阻塞事件 | drag、scroll（拖动时） | 中 |
| 连续事件 | scroll（惯性）、mousemove | 低（可被中断） |

配合 `useTransition`：
```jsx
function handleInput(e) {
  setInputValue(e.target.value);  // 高优先，立即响应
  startTransition(() => {
    setSearchResult(query);       // 低优先，可被新输入中断
  });
}
```

## 九、最佳实践

1. 默认用 React 合成事件，需要原生行为时通过 `e.nativeEvent` 访问
2. `stopPropagation` 只影响 React 树，需要影响原生树时用 `nativeEvent.stopImmediatePropagation`
3. React 17+ 无需 `e.persist()`
4. 注意 React 16 → 17 事件委托位置变化对微前端的影响
5. Portal 的事件按 React 树冒泡，不要按 DOM 树去推理
